## 一、SKILL 的目的与边界

### 1.1 解决什么问题

对某只股票做**当日或近几日**的股价快速诊断，基于实时数据给出当前状态判断。

**核心定位**：短期诊断（1 天 ~ 1 周），不是长期分析。

### 1.2 触发条件

**触发**：
- "今日 XX 股价怎么样"
- "最近几天 XX 走势怎么看"
- "今天 XX 为什么涨/跌"
- "XX 现在什么情况"
- "分析一下 XX 今日走势"

**不触发**（应使用 `stock-deep-analysis`）：
- "分析 XX 半年走势"
- "XX 的估值是多少"
- "XX 的护城河分析"

### 1.3 与 `stock-weekly-forecast` 的关系

**关键区分**：用户问"下周怎么走"→ 用 `stock-weekly-forecast`（必须预测）；用户问"现在什么状态"→ 用本skill（不预测）。

| 用户问法 | 应用的skill |
|---|---|
| "XX 下周怎么走" / "预测 XX 未来一周" | `stock-weekly-forecast` |
| "今日 XX 股价怎么样" / "今天 XX 为什么涨/跌" | `stock-daily-diagnosis`（本skill） |
| "分析 XX 半年走势" / "XX 的估值" | `stock-deep-analysis` |

**⚠️ 常见错误**：用户要预测时只给诊断 → 用户会说"你这个有意义么"。必须区分"诊断当前状态"和"预测未来走势"。

### 1.4 与 `stock-deep-analysis` 的关系

| 维度 | stock-daily-diagnosis | stock-deep-analysis |
|---|---|---|
| 时间窗口 | 1 天 ~ 1 周 | 3-12 个月 |
| 数据来源 | 实时 API | 年报/季报/行业数据 |
| 核心问题 | "现在什么状态" | "值多少钱" |
| 输出 | 状态诊断 + 关键价位 + 结构分析 | 完整估值报告 |
| 速度 | 5-10 分钟 | 30-60 分钟 |

### 1.4 核心原则

1. **不预测**——只诊断当前状态，不说明天涨还是跌
2. **必须给等待条件**——什么数据出现时判断会改变
3. **必须做深度结构分析**——不能只看表面信号
4. **必须用 API 拉数据**——不能用搜索引擎估计

## 二、核心方法论：7 维度诊断

```
输入：股票代码 + "今日/最近"
    ↓
【维度 1】实时数据采集（API 拉取）
    ↓
【维度 2】近期走势上下文（多日 K 线）

    ↓
【维度 4】近期事件扫描（公告/评级/中标）
    ↓
【维度 5】多空博弈判断（表面信号列表）
    ↓
【维度 6】关键价位识别（支撑/压力/状态）
    ↓
【维度 7】深度结构分析（核心创新）
    ├─ 7.1 信号拆解

    ├─ 7.4 消化压力评估
    └─ 7.5 历史模式匹配
    ↓
【输出】一句话总结 + 关键价位表 + 等待条件
```

## 三、维度 1：实时数据采集

### 3.1 数据源优先级（v2.2 更新）

**腾讯 API > 东方财富 API**（2026-06-30 实测：东方财富返回过时缓存数据，6/30收盘误报为99.46元，实际为107.23元）

| 优先级 | 数据源 | 用途 | 可靠性 |
|---|---|---|---|
| **1（首选）** | 腾讯 `qt.gtimg.cn` | 实时行情 | ✅ 实时准确 |
| **1（首选）** | 腾讯 `web.ifzq.gtimg.cn` | K线数据 | ✅ 实时准确 |
| 2 | 东方财富 `push2.eastmoney.com` | 实时行情 | ⚠️ 可能返回缓存 |
| 2 | 东方财富 `push2his.eastmoney.com` | K线数据 | ⚠️ 可能返回缓存 |
| 3 | 新浪 `hq.sinajs.cn` | 备用验证 | ⚠️ 偶尔返回 Forbidden |

### 3.2 必拉数据

**必须用 API 拉取，不能用搜索引擎**（搜索引擎返回的是历史快照，不是实时数据）。

| 数据 | 首选 API | 备用 API | 说明 |
|---|---|---|---|
| 实时行情 | `qt.gtimg.cn/q=shXXXXXX` | `push2.eastmoney.com` | 收盘/开盘/最高/最低/成交量/换手率/PE |
| K 线（日线） | `web.ifzq.gtimg.cn/appstock/app/fqkline/get` | `push2his.eastmoney.com` | 近 N 日的 OHLCV |
| 资金流向 | `push2.eastmoney.com/fflow` | 外盘/内盘比例 | 主力/超大单/大单/中单/小单 |
| 大盘指数 | `qt.gtimg.cn/q=sh000001` | `push2.eastmoney.com` | 上证/深证/沪深 300 |

### 3.3 API 端点速查

```bash
# 【首选】腾讯实时行情（GBK 编码，字段用 ~ 分隔）
curl -s "http://qt.gtimg.cn/q=sh603019" | iconv -f GBK -t UTF-8
# 字段：[3]=当前价 [4]=昨收 [5]=今开 [6]=成交量(手) [35]=最新价/量/额 [38]=换手率

# 【首选】腾讯K线（前复权日线）
curl -s -L "https://web.ifzq.gtimg.cn/appstock/app/fqkline/get?_var=kline_dayqfq&param=sh603019,day,,,20,qfq"
# 返回 JSON，data.sh603019.qfqday 数组，每条 [日期,开盘,收盘,最高,最低,成交量]

# 【备用】东方财富实时行情（沪市用 1. 前缀，深市用 0. 前缀）
curl -s -A "Mozilla/5.0" \
  "https://push2.eastmoney.com/api/qt/stock/get?secid=1.603019&fields=f43,f44,f45,f46,f47,f48,f60,f168,f169,f170"

# 【备用】东方财富K线（需要 ut 参数）
curl -s -A "Mozilla/5.0" \
  "https://push2his.eastmoney.com/api/qt/stock/kline/get?secid=1.603019&fields1=f1,f2,f3,f4,f5,f6&fields2=f51,f52,f53,f54,f55,f56,f57,f58,f59,f60,f61&klt=101&fqt=0&end=20500101&lmt=10&ut=fa5fd1943c7b386f172d6893dbbd1"

# 资金流向（东方财富，可能被限流）
curl -s -A "Mozilla/5.0" \
  "http://push2.eastmoney.com/api/qt/stock/fflow/kline/get?lmt=0&klt=101&fields1=f1,f2,f3&fields2=f51,f52,f53,f4,f55,f56,f57,f58&secid=1.603019&num=10"

# 备用验证：新浪财经
curl -s "http://hq.sinajs.cn/list=sh603019" | iconv -f GBK -t UTF-8
```

### 3.4 强制规则

1. **必须用 API，不能用搜索引擎**
2. **必须标注来源**——"腾讯 API"或"东方财富 API"
3. **API 失败时必须明说**——不能偷偷估计数据
4. **【v2.2 新增】必须多源交叉验证**——用 ≥2 个数据源验证收盘价，不一致时以腾讯为准
5. **【v2.2 新增】禁止展示中间过程**——不给用户看 curl 命令、Python 代码、JSON 原始数据，只给最终分析结果
6. **【v2.2 新增】东方财富被限流时**——资金流向用外盘/内盘比例替代（外盘>内盘=买入主导），K线用腾讯 API 替代

### 3.5 多源交叉验证流程（v2.2 新增）

```
步骤 1：用腾讯 API 拉实时行情（首选）
步骤 2：用东方财富 API 拉实时行情（备用）
步骤 3：对比两个数据源的收盘价
    → 差异 < 0.01 元 → 数据可信，继续分析
    → 差异 ≥ 0.01 元 → 以腾讯为准，标注"东方财富数据异常"
步骤 4：K线数据同样多源验证
    → 腾讯 K线 API（web.ifzq.gtimg.cn）为首选
    → 东方财富 K线 API 为备用
```

**2026-06-30 实测案例**：
- 东方财富返回 6/30 收盘 99.46 元（过时缓存）
- 腾讯返回 6/30 收盘 107.23 元（实时准确）
- 差异 7.77 元，必须以腾讯为准

### 3.6 资金流向 API 限流 fallback（v2.2 新增）

当东方财富资金流向 API 返回空数据（限流）时：

| 替代方案 | 数据源 | 含义 |
|---|---|---|
| 外盘/内盘比例 | 腾讯 API 字段 [7]/[8] | 外盘>内盘=买入主导，反之=卖出主导 |
| 资金流向估算 | 成交额 × 外盘占比 | 估算主动买入金额 |

**输出时标注**："资金流向数据暂缺（API限流），以下基于外盘/内盘比例推断"

### 3.4 输出模板

```
| 指标 | 数值 | 来源 |
|---|---|---|
| 收盘价 | XX 元 | 东方财富 API |
| 涨跌幅 | +X.XX% | 东方财富 API |
| 最高/最低 | XX / XX 元 | 东方财富 API |
| 成交量 | XX 万手 | 东方财富 API |
| 换手率 | X.XX% | 东方财富 API |
| PE(TTM) | XX 倍 | 东方财富 API |
```

## 四、维度 2：近期走势上下文

### 4.1 为什么要看近期走势

单日涨跌没有意义——必须放在**近期走势的上下文中**才能判断"今天意味着什么"。

### 4.2 输出模板

```
| 日期 | 收盘 | 涨跌幅 | 成交额 | 性质 |
|---|---|---|---|---|
| 6/22 | 27.42 | +5.30% | 9.63 亿 | 放量反弹 |
| 6/23 | 26.58 | -3.06% | 4.63 亿 | 回吐涨幅 |
| 6/24 | 26.65 | +0.26% | 6.43 亿 | 横盘整理 |
| 6/25 | 26.57 | -0.30% | 5.74 亿 | 横盘，盘中破支撑 |
```

### 4.3 "性质"的标注规范

| 性质 | 判断标准 |
|---|---|
| 放量反弹 | 涨幅 > 2% 且成交额 > 前日 1.5 倍 |
| 回吐涨幅 | 前日大涨后今日下跌，跌幅 > 前日涨幅的 50% |
| 横盘整理 | 涨跌幅 < ±1% |
| 恐慌暴跌 | 跌幅 > 5% 且成交额显著放大 |
| 缩量触底 | 下跌但成交额萎缩至近期最低 |
| 放量突破 | 突破关键价位且成交额放大 |
| 放量涨停 | 涨停且成交额显著放大 |

### 4.4 关键判断：反弹是否有效

**反弹有效**的条件：
- 反弹后 3 个交易日内不跌破反弹起点
- 反弹期间成交额维持或放大
- 主力资金持续流入

**反弹无效（"一日游"）**的条件：
- 反弹后 3 个交易日内跌回原点
- 反弹期间成交额萎缩
- 主力资金在反弹日卖出

## 五、维度 3：资金流向分析

### 5.1 数据来源

- **主力/散户资金流向**：东方财富资金流向 API
- **融资融券数据**：同花顺/证券之星新闻

### 5.2 分析框架：4 种资金模式

| 模式 | 主力 | 散户 | 含义 |
|---|---|---|---|
| **主力吸筹** | 买入 | 卖出 | 底部信号（聪明钱在买） |
| **散户接盘** | 卖出 | 买入 | 顶部信号（散户追高） |
| **全面做多** | 买入 | 买入 | 趋势确认 |
| **全面做空** | 卖出 | 卖出 | 恐慌信号 |

### 5.2b 【必做】主力资金 API 字段定义 + 内部一致性校验

**目的**：避免直接用 f51 字段误读主力方向。本节是整个维度 3 最容易踩坑的环节。

**API 端点**：
```
push2.eastmoney.com/api/qt/stock/fflow/daykline/get?secid={secid}&fields1=f1,f2,f3&fields2=f51,f52,f53,f54,f55,f56,f57,f58
```

**字段定义**（f51-f55，对应资金流向）：
| 字段 | 含义 | 单笔成交额 |
|---|---|---|
| f51 | 主力净流入 | = 超大单 + 大单（API 直接给的"主力"） |
| f52 | 超大单净流入 | ≥ 100 万元 |
| f53 | 大单净流入 | 20-100 万元 |
| f54 | 中单净流入 | 4-20 万元 |
| f55 | 小单净流入 | < 4 万元 |

**东方财富"主力"标准定义**：
> 主力 = 超大单 + 大单 = **f52 + f53**

**【必做】3 项内部一致性校验**（输出诊断结论前必须做）：

| # | 校验项 | 通过条件 | 不通过时怎么办 |
|---|---|---|---|
| 1 | **5 字段之和 ≈ 0** | \|f51+f52+f53+f54+f55\| < 0.5 亿 | 说明字段解码错误，必须重读字段文档 |
| 2 | **f51 vs (f52+f53) 方向一致** | 符号相同，量级相当 | 符号相反时，**以 (f52+f53) 为准**（标准定义优先） |
| 3 | **主力 vs 散户合理性** | 主力/散户不应"同方向 + 同量级"（如主力+散户各买 20+ 亿） | 检查解码错误或字段理解错误 |

**正确诊断模式**（中科曙光 2026-06-30 案例）：
```
原始数据: f51=+17.18, f52=-5.47, f53=-11.71, f54=-2.90, f55=+20.07
校验 1: 5 字段之和 = +17.17（异常，应≈0）
校验 2: f51=+17.18 vs f52+f53=-17.18（符号相反）
校验 3: 主力应卖 -17.18 vs 散户买 +20.07（方向相反合理）

正确诊断：主力净卖出 17.18 亿（机构在卖），散户净买入 20.07 亿（接盘）
错误诊断：直接用 f51 = "主力买入 17 亿"（完全相反！）
```

**常见错误**（必须避免）：
- ❌ 直接用 f51 当主力净流入，不做内部一致性校验
- ❌ 看到 f51 = +17 亿就解读为"主力买入 17 亿"
- ❌ 散户/主力"同买 37 亿"这种异常结构没发现

**正确做法**：
- ✅ 始终按"主力 = f52 + f53"重算
- ✅ 输出主力方向前做 3 项校验
- ✅ 校验不通过时，明确标注"主力资金数据待核实"

### 5.3 资金面拐点识别

**拐点 = 资金模式切换**：
- 从"散户接盘"→"主力吸筹" = **底部拐点**
- 从"主力吸筹"→"散户接盘" = **顶部拐点**

### 5.4 融资融券的辅助判断

- **融资净买入增加** → 杠杆资金看好（偏多）
- **融资净偿还增加** → 杠杆资金撤退（偏空）
- **融资 + 主力同时买入** → 强烈看多信号
- **融资偿还 + 主力买入** → 矛盾信号（机构买，杠杆撤）

### 5.5 输出模板

```
| 日期 | 主力净流入 | 游资净流入 | 散户净流出 | 模式 |
|---|---|---|---|---|
| 6/23 | +1887 万 | +2889 万 | -4776 万 | 主力+游资买入 |
| 6/24 | +3599 万 | +4586 万 | -8185 万 | 主力+游资持续买入 |

融资数据：
- 6/23 融资净偿还 1881 万
- 6/24 融资净偿还 1086 万

资金面矛盾信号：主力+游资在买，杠杆资金在撤
```

## 七、维度 5：多空博弈判断

### 7.1 核心方法：矛盾信号分析

短期股价由**多空力量博弈**决定。当多空信号同时存在时，必须：
1. 列出所有多头信号
2. 列出所有空头信号
3. **给出倾向**（偏多/偏空/中性）
4. **给出等待条件**（什么数据出现时判断会改变）

### 7.2 输出模板

```
| 信号 | 方向 | 强度 | 解读 |
|---|---|---|---|
| 6/22 反弹被回吐 | 空 | 强 | 反弹无力 |
| 盘中破 26.00 支撑 | 空 | 强 | 支撑被击穿 |
| 主力+游资持续买入 | 多 | 中 | 聪明钱在底部吸筹 |
| 券商维持买入评级 | 多 | 中 | 机构认为估值已低 |
| PE(TTM) 10.78 倍 | 多 | 中 | 历史低位 |
| 大盘 +0.15% | 中性 | 弱 | 无系统性风险 |

倾向：偏空（空头信号强度 > 多头信号强度）
等待条件：6 月销量数据（7 月初）——如果同比正增长，转为偏多
```

### 7.3 倾向判断的规则

| 情况 | 倾向 |
|---|---|
| 空头信号数量 > 多头，且空头强度更强 | **偏空** |
| 多头信号数量 > 空头，且多头强度更强 | **偏多** |
| 多空数量相当，强度相当 | **中性** |
| 矛盾信号并存，无法判断 | **中性偏 X**（给出倾向 + 等待条件） |

## 八、维度 6：关键价位识别

### 8.1 价位类型

| 类型 | 定义 | 识别方法 |
|---|---|---|
| **支撑位** | 股价下跌到此价位后反弹 | 近期多次触及的低点 |
| **压力位** | 股价上涨到此价位后回落 | 近期多次触及的高点 |
| **整数关口** | 心理价位（如 20、25、30 元） | 无需计算 |
| **均线位** | MA5/MA10/MA20/MA60 | 均线计算 |

### 8.2 支撑/压力的状态标注

| 状态 | 含义 |
|---|---|
| ✅ 有效 | 股价在支撑位上方，或在压力位下方 |
| ⚠️ 已被触及 | 股价触及但未突破 |
| ❌ 已被击穿 | 股价突破支撑/压力 |
| 🔄 已转化为反向 | 支撑被击穿后变为压力，或反之 |

### 8.3 输出模板

```
| 价位 | 含义 | 状态 |
|---|---|---|
| 25.81 元 | 今日盘中最低 | ⚠️ 已被触及 |
| 26.00 元 | 双底支撑 | ⚠️ 已被击穿一次 |
| 26.57 元 | 当前收盘 | — |
| 27.42 元 | 6/22 反弹高点（压力） | ✅ 有效（远离） |
| 22.00 元 | 52 周最低 / 极端支撑 | ✅ 有效（远离） |
```

## 九、维度 7：深度结构分析（核心创新）

### 9.1 为什么需要这个维度

维度 1-6 给出的是**表面信号**——"主力在买""涨停了""事件是正面的"。

但表面信号可能被误读。维度 7 的目的是**深入市场结构，看表面信号背后真正发生了什么**。

**核心问题**：市场结构在告诉我们什么？

### 9.2 子维度 7.1：信号拆解

**做什么**：把聚合信号拆成子信号，检查方向是否一致。

**为什么重要**：聚合信号可能掩盖内部分歧。

**操作步骤**：

```
步骤 1：找出聚合信号
    → 如"主力净流入 23 亿"

步骤 2：拆成子信号
    → 超大单（>100 万）：+25.81 亿
    → 大单（20-100 万）：-2.81 亿

步骤 3：检查方向一致性
    → 超大单买，大单卖 → 方向不一致

步骤 4：得出结论
    → 不是"一致看多"，是"单一力量推动"
```

**输出模板**：

```
| 聚合信号 | 子信号 | 方向 | 一致性 |
|---|---|---|---|
| 主力净流入 23 亿 | 超大单 +25.81 亿 | 买 | ❌ 不一致 |
| | 大单 -2.81 亿 | 卖 | |
| 散户卖出 23 亿 | 中单 -15.68 亿 | 卖 | ✅ 一致 |
| | 小单 -7.32 亿 | 卖 | |
```

**关键发现示例**：主力内部有分歧——超大单（可能是单一大机构）在买，大单（可能是其他机构）在卖。这不是"一致看多"，是"单一力量推动"。

### 9.3 子维度 7.2：市场结构分析

**做什么**：分析涨停/跌停 + 换手率的组合含义。

**为什么重要**：同样的涨停，换手率不同含义完全不同。

**关键组合**：

| 组合 | 含义 | 后续概率 |
|---|---|---|
| 涨停 + 换手率 < 3% | 一致看多，无人卖出 | ~70% 次日继续涨 |
| 涨停 + 换手率 3-7% | 多空有一定分歧 | ~50% 次日继续涨 |
| **涨停 + 换手率 > 7%** | **多空激烈交战** | **~35% 次日继续涨** |
| **二板涨停（首板后回调再涨停）** | **趋势确认，信号更强** | **~55% 继续涨** |
| 跌停 + 换手率 < 3% | 一致看空，无人买入 | ~70% 次日继续跌 |
| 跌停 + 换手率 > 7% | 有资金在跌停板接盘 | ~40% 次日继续跌 |

**二板涨停定义**：首板涨停后，经过 1-3 天回调/震荡，再次涨停。比首板涨停信号更强，因为市场经过消化后再次确认上涨趋势。
- 2026-06-30 中科曙光案例：6/25 首板涨停→6/26-6/29 回调→6/30 二板涨停

**操作步骤**：

```
步骤 1：确认今日涨跌状态（涨停/跌停/普通涨跌）
步骤 2：查看换手率
步骤 3：匹配组合
步骤 4：给出后续概率（标注"历史统计，非预测"）
```

**输出模板**：

```
市场结构：
- 今日状态：涨停 +10.00%
- 换手率：9.00%
- 组合：涨停 + 高换手率（>7%）
- 含义：多空激烈交战，非一致看多
- 历史统计：~35% 次日继续涨（非预测，仅供参考）
```

### 9.4 子维度 7.3：催化剂新鲜度

**做什么**：评估今日涨跌的催化剂是否已被市场消化。

**为什么重要**：已被消化的催化剂不会持续推动股价。

**操作步骤**：

```
步骤 1：列出所有可能的催化剂
步骤 2：确认每个催化剂的公布时间
步骤 3：查看催化剂公布后的市场反应
步骤 4：判断催化剂是否已被消化
步骤 5：如果已消化，追问"今日涨跌的真正原因是什么？"
```

**催化剂新鲜度判断**：

| 状态 | 判断标准 |
|---|---|
| **新鲜** | 今日公布，市场尚未反应 |
| **半新鲜** | 1-2 天前公布，市场部分反应 |
| **已消化** | 3+ 天前公布，市场已充分反应 |
| **过期** | 1 周+ 前公布，不再是对今日的催化 |

**输出模板**：

```
| 催化剂 | 公布时间 | 市场反应 | 新鲜度 |
|---|---|---|---|
| 减持终止 | 6/22 晚间 | 6/23 -2.20%，6/24 +2.15% | ❌ 已消化（3 天） |
| 中信建投研报 | 6/25 | 今日涨停 | ⚠️ 一家券商研报不足以导致涨停 |

追问：今日涨停的真正催化剂是什么？
→ 可能是未公开信息（如大额订单）→ 需要后续验证
→ 可能是纯粹资金行为（如某大基金建仓）→ 持续性存疑
```

### 9.5 子维度 7.4：消化压力评估

**做什么**：评估今日成交量中买入方的成本集中度，以及后续可能的卖压。

**为什么重要**：大量资金在某个价位买入后，如果股价不继续涨，这些资金会变成卖压。

**操作步骤**：

```
步骤 1：确认今日成交额
步骤 2：确认今日买入成本集中区（通常 = 收盘价附近）
步骤 3：计算"消化压力"
    → 如果明日股价 > 今日收盘 → 买入方浮盈，卖压小
    → 如果明日股价 < 今日收盘 → 买入方浮亏，卖压大
步骤 4：给出关键指标
    → 明日成交额 > X 亿 → 资金持续流入，涨停有效
    → 明日成交额 < Y 亿 → 资金枯竭，涨停可能无效
```

**输出模板**：

```
消化压力评估：
- 今日成交额：125.37 亿
- 买入成本集中区：97.82 元（涨停价）
- 消化压力：如果明日不涨，125 亿资金浮亏 → 变成卖压
- 关键指标：
  → 明日成交额 > 80 亿 → 资金持续，涨停有效
  → 明日成交额 < 50 亿 → 资金枯竭，涨停可能无效
```

### 9.6 子维度 7.5：历史模式匹配

**做什么**：用历史数据匹配当前的"涨跌+换手率+成交额"组合，给出先验概率。

**为什么重要**：历史不能预测未来，但可以给出"基准概率"。

**操作步骤**：

```
步骤 1：确认当前模式
    → 涨停 + 高换手率 + 高成交额
步骤 2：查找历史同类模式
    → 过去 1 年，A 股涨停 + 换手率 > 7% 的案例
步骤 3：统计后续走势
    → 次日继续涨的概率
    → 次日回调的概率
步骤 4：给出先验概率（标注"历史统计，非预测"）
```

**输出模板**：

```
历史模式匹配：
- 当前模式：涨停 + 换手率 9% + 成交额 125 亿
- 历史统计（A 股近 1 年）：
  → 涨停 + 换手率 > 7%：~35% 次日继续涨，~65% 次日回调或横盘
  → 涨停 + 换手率 < 3%：~70% 次日继续涨
- 结论：当前模式属于"高不确定性"类型
- ⚠️ 历史统计仅供参考，不构成预测
```

## 十二、常见错误防范

| 错误 | 正确做法 |
|---|---|
| 用搜索引擎搜"今日股价" | 用 curl 调 API |
| 搜不到数据时估计 | 明说"搜不到" + 列方案 |
| 只说"涨了"或"跌了" | 给出近期走势上下文 |
| 不看资金流向 | 必须分析主力/散户/融资 |
| 只看单日数据 | 必须看多日趋势 |
| 给出"明天会涨/跌" | 不预测，只诊断当前状态 |
| 不给等待条件 | 必须给出"什么数据出现时判断改变" |
| **涨停就全看多** | **必须做深度结构分析（维度 7）** |
| **跌停就全看空** | **必须做深度结构分析（维度 7）** |
| **不拆解聚合信号** | **必须拆成子信号检查方向一致性** |
| **不评估催化剂新鲜度** | **必须判断事件是否已被消化** |
| **不评估消化压力** | **必须计算买入成本集中度和后续卖压** |
| **直接用 f51 当主力资金，不做校验** | **必须按 f52+f53 重新计算，并做 3 项内部一致性校验（见 5.2b）** |
| **只用东方财富一个数据源** | **必须用腾讯+东方财富双源验证，不一致时以腾讯为准（见 3.5）** |
| **东方财富返回过时数据时继续用** | **收盘价差异≥0.01元时必须切换到腾讯数据源** |
| **展示 curl/Python 中间过程给用户** | **只给最终分析结果，中间过程隐藏** |
| **资金流向API限流时放弃分析** | **用外盘/内盘比例替代（见 3.6）** |
