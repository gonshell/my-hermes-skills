# 三角评审法详细操作指南

## 触发时机

以下任一条件满足时，触发三角评审：

1. 文档是**企业级需求文档/决策参考**，且有多个利益相关方（业务、技术、管理）
2. 用户说"帮我review一下"、"评审"、"从不同角度看看"
3. 结构设计阶段已完成，需要验证结构是否覆盖关键决策维度
4. 文档修订完成，需要最终质量验证

## 角色配置模板

### 角色1：资深产品经理

```
背景设定：
- 10年+企业软件产品经验
- 主导过多个从0到1的企业级产品

输出格式：

- P0问题列表（必须修复）
- P1问题列表（应该修复）
- P2问题列表（考虑修复）
- 修改建议
```

## 评审触发指令模板

```
请以【角色名】视角，对以下文档进行结构评审：

文档标题：[标题]
文档路径：[路径]

评审要求：
1. 假设你是[角色背景]，最关注什么？
2. 当前文档是否满足你的关注点？
3. 哪些P0问题会导致你无法用它做决策？

请输出：
- 评分（1-5，5=完美）
- P0/P1/P2问题列表
- 具体修改建议
```

## P0/P1/P2 判定规则

### P0（阻塞级）

以下任一条件满足即为P0：

- 缺少做决策所必需的**核心章节**（如供应商锁定风险之于CIO）
- 关键数据**完全缺失**且无可信来源
- 逻辑链**根本断裂**，无法从前提推到结论
- ROI模型**完全不透明**（只有结论，无假设和场景）

### P1（重要级）

以下任一条件满足即为P1：

- 章节存在但**深度不足**（只有概述，缺细节）
- 数据来源**可信度低**（个人博客、非权威来源）
- 缺少重要的**对比维度**（如国产vs进口、不同供应商对比）
- 实施路径**不够具体**（只有方向，缺工期和资源）

### P2（优化级）

以下任一条件满足即为P2：

- 连续纯文字超过3段（可读性问题）
- 缺少架构图/流程图/对比表
- 过渡段落缺失或生硬
- 术语表/执行摘要不完善

## 评审结果合成方法

### 原则

1. **P0问题不过夜**：任一视角提出P0，必须修复，否则文档不能作为正式决策材料
2. **跨视角P1优先**：两个及以上视角同时提出的P1，比单一视角的P1更紧急
3. **保留争议点**：如果三个视角对某个问题有分歧，记录争议，供用户决策

### 合成操作

1. 读取三份评审报告
2. 提取所有P0 → 合并去重 → 按严重程度排序
3. 提取所有P1 → 按提出次数排序（2次以上优先）
4. 提取所有P2 → 按提出次数排序
5. 生成综合结论

### 合成报告模板

```markdown
# 三角评审综合结论

## 评审概览
- 产品经理评分：X/5
- CIO评分：X/5
- 市场分析师评分：X/5
- 综合评级：[A/B/C]

## 必须修复的P0问题（共N个）
1. [问题1] — [提出者]
2. [问题2] — [提出者]

## 应修复的P1问题（共N个，按优先级）
1. [问题1] — [提出者x2]
2. [问题2] — [提出者]

## 争议点记录
| 争议点 | 视角A观点 | 视角B观点 | 建议 |
|--------|---------|---------|------|

## 下一步行动
- [ ] 修复P0问题
- [ ] 确认P1问题处理方案
- [ ] 用户决策争议点
```
