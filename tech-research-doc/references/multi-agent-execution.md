# 多智能体执行模式参考

## 适用场景

企业级文档需要深度修订，且满足以下条件：

1. 文档结构需要重新设计（非局部修改）
2. 数据可信度需要核查（非简单文字校对）
3. 技术可行性需要验证（非描述性分析）
4. 用户明确授权执行（非仅讨论方案）

## 执行角色配置

### 内容主笔（Content Lead）

```
职责权重：60-70%
参与方式：全职（在数据研究员和技术架构评审确认后执行）
核心任务：
1. 结构调整：根据评审意见重新设计章节结构
2. 内容重写：按新结构撰写完整内容
3. 过渡模板：在章节之间添加过渡段落，确保逻辑连贯
4. 执行摘要：重构执行摘要，确保每个结论有数据支撑和置信度标注

输入：
- 评审报告（P0/P1/P2问题列表）
- 数据研究员结论（已确认的数据）
- 技术架构评审结论（已确认的技术路径）

输出：
- 完整修订版文档

关键原则：
- 等数据研究员和技术架构评审确认后，再开始执行
- 每一章必须明确：输入是什么？输出是什么？
- 先确认策略（目录+行文逻辑），再开始写
```

### 数据研究员（Data Researcher）

```
职责权重：20-25%
参与方式：兼职（先于内容主笔执行，可并行）
核心任务：
1. 数据可信度核查：评估现有数据的来源和可信度
2. 补充权威数据：用官方/学术来源替换估算值
3. 一手数据调研：如需要，进行用户访谈或市场调研
4. 数据缺口报告：识别无法核实的数据点，标注[E]

核查优先级（P0先于P1）：
- P0：直接支撑核心结论的数据（如ROI数字、市占率）
- P1：支撑次要结论的数据（如实施周期估算）

输入：
- 原文档
- 评审报告（数据可信度相关问题）

输出：
- 数据核查报告（每条数据：来源/可信度/建议）

数据可信度分级：
[N] 官方发布（可直接引用）
[T] 行业经验（多方印证，非官方）
[E] 估算（基于有限信息推断，需注明推断依据）
```

### 技术架构评审（Technical Architect）

```
职责权重：10-15%
参与方式：兼职（与数据研究员并行执行）
核心任务：
1. 技术可行性验证：评估技术方案是否真正可行
2. 技术风险识别：识别技术债务、集成风险、安全风险
3. 工期估算修正：基于实际情况修正实施周期
4. 供应商锁定风险评估：评估依赖单一供应商的风险

验证清单：
□ Multi-Agent权限隔离是否可行？工期多长？
□ Policy Engine选型是否适合监管行业？
□ 国产LLM适配难度如何？
□ 等保/信创认证周期是否纳入估算？
□ ERP集成复杂度是否被低估？

输出：
- 技术可行性验证报告
- 技术风险清单
- 工期修正建议
```

## 工作计划模板

### 内容主笔工作计划

```markdown
# 内容主笔-工作计划

## 策略确认清单（执行前必须确认）

### Q1: 目录结构
- 新目录是否已与用户确认？
- 每章的输入/输出是否明确？
- P0问题是否已全部解决？

### Q2: 整合方案
- 数据研究员的结论如何整合进正文？
- 技术架构评审的结论如何整合进正文？
- 过渡模板是否准备好？

### Q3: 写作优先级
- 执行摘要优先级最高（最后写，但最先规划）
- P0问题对应的章节次高
- P1问题对应的章节第三

### Q4: 过渡模板
- 章节之间的过渡段落模板是什么？
- 如何确保逻辑连贯？

## 执行步骤

1. [ ] 读取数据研究员结论
2. [ ] 读取技术架构评审结论
3. [ ] 确认目录结构（向用户汇报）
4. [ ] 开始写作
5. [ ] 完成后向用户汇报
```

### 数据研究员工作计划

```markdown
# 数据研究员-工作计划

## 核查优先级

| 优先级 | 数据点 | 当前位置 | 建议 |
|--------|-------|---------|------|
| P0 | [核心结论数据] | 第X章 | 需官方来源替换 |
| P1 | [次要结论数据] | 第X章 | 行业经验可接受 |

## 核查方法

1. 官方文档优先（NVIDIA Datasheet、Gartner、McKinsey）
2. 学术论文次之（SOSP、MLSys、ICML）
3. 开源benchmark第三（MLCommons）
4. 估算最后（标注[E]，说明推断依据）

## 数据缺口处理

无法核实的数据点：
- 优先：寻找替代数据源
- 次优：降级为[T]或[E]
- 最差：删除该数据点，在文档中注明"数据缺口"

## 交付物

数据核查报告，包含：
- 每条被质疑数据的核查结果
- 建议的替换数据或降级方案
- 无法核实的数据点清单
```

## 数据可信度核查清单

```markdown
# 数据可信度核查清单

## 必查项

### 市场规模数据
- [ ] Gartner/IDC/Forrester报告？（优先查最新年份）
- [ ] 中国市场数据？（注意国内外口径差异）
- [ ] 样本量是否足够？（Gartner通常>1000家企业）
- [ ] 是否跨年数据重叠？（可能导致数字虚高10-30%）

### 性能数据
- [ ] 官方Datasheet？（优先PDF，非营销页）
- [ ] 独立Benchmark？（MLCommons等）
- [ ] 实测数据？（注明测试环境）

### ROI数据
- [ ] 假设条件是否清晰？
- [ ] 是否包含场景测试（乐观/基准/悲观）？
- [ ] 中国市场额外成本是否纳入？（等保、信创、ERP集成）

### 技术工期数据
- [ ] 是否针对中国市场调整？
- [ ] 国产LLM适配周期是否单独列出？
- [ ] 等保2.0认证周期（3-6个月）是否纳入？
```

## 技术可行性验证清单

```markdown
# 技术可行性验证清单

## Multi-Agent系统

### 权限隔离
- [ ] 自研Permission Broker工期：8-12周
- [ ] 商业方案（如Aporia）是否适用？
- [ ] 开源方案（如Open Policy Agent）是否满足需求？

### 选型验证
- [ ] Policy Engine类型：规则引擎 vs ML引擎
- [ ] 监管行业必须用规则引擎（OPA/IBM ODM）
- [ ] 性能要求是否明确？（延迟<100ms等）

## 中国市场特殊性

### 合规认证
- [ ] 等保2.0：周期3-6个月，费用30-80万
- [ ] 信创认证：周期3-6个月
- [ ] 数据安全法/个人信息保护法合规

### 集成复杂度
- [ ] ERP碎片化程度（SAP/用友/金蝶/自研混合）
- [ ] 国产LLM适配难度
- [ ] 私有化部署需求

## 工期修正规则

原始估算 × 1.3-1.5 = 中国市场实际工期

原因：
1. 等保/信创认证：+3-6个月
2. ERP集成复杂度：+20-30%
3. 国产LLM适配：+15-25%
4. 沟通协调成本：+10-15%
```
