# 内容深度不足的诊断与处理方法

## 核心问题：用户说"干瘪"时，到底想要什么？

当用户说"内容太干瘪/不够充分/太泛"时，这是需求文档写作中最容易被误解的反馈。误解会导致错误的修复方向，造成大量返工。

---

## 诊断框架：干瘪的三个真实原因

```
用户说"干瘪"的可能含义：
├── A. 结构缺失（读者不知道往下该看什么）
│   └── 表现：章节之间逻辑断层、核心论点只有结论没有展开
├── B. 内容深度不足（读者知道了但没理解"为什么"）
│   └── 表现：数字没有背景、观点没有案例、因果链断裂
└── C. 表达机械（读者知道内容对但读不进去）
    └── 表现：套话多、无语气变化、缺乏对话感
```

---

## 判断方法：先问结构，再问深度，最后才改表达

### Step 1：判断是否是结构问题

```
检查：
  □ 有三角评审通过的结论吗？
  □ 有用户确认过的结构方案吗？
  □ 报告是否包含：执行摘要 + WHO/WHAT/PRIORITY/CAN-DO 四问框架？

如果有 → 结构大概率是对的，跳过结构讨论
如果没有 → 优先讨论结构，不要埋头重写内容
```

### Step 2：如果是结构问题

```
修复方式：增补缺失的章节或段落
- 缺"风险假设"章节 → 补充风险清单
- 缺"约束条件"章节 → 补充合规/集成/性能/安全四维约束
- 缺"调研方法"章节 → 补充数据来源和可信度说明
```

### Step 3：问用户具体是哪类内容不足

```
直接问（不要猜）：
"您觉得'干瘪'主要是：
  ① 数据不够权威？（想看一手数据出处）
  ② 分析不够深入？（想看因果链、背景解释）
  ③ 案例不够具体？（想看真实场景举例）
  ④ 表达不够自然？（觉得读起来像机器写的）"

基于回答，针对性补充，而非重写全文
```

### Step 4：区分"重写"和"增量修改"

```
❌ 不要因为局部内容不足就重写全文
  → 重写 = 推翻已有结构，风险极高
  → 增量 = 保留结构，针对性补充，风险低

✅ 正确做法：
  保留已通过评审的结构
  对具体不足的章节做增量修改
  去AI味在修改完成后做，不作为重新组织的理由
```

## 内容深度不足的具体修复方式

### 数字没有背景 → 补充数字的来源和含义

```
❌ 干瘪写法：
"意图识别准确率约73%。"

✅ 充实写法：
"意图识别准确率约73%（Forrester 2024行业平均水平），
 但场景差异极大：简单意图（FAQ类）头部厂商可达95%以上，
 中等复杂意图约85%，模糊意图（客户描述不清）行业平均仅50%。
 这意味着：AI在能处理60%的工单的同时，
 难以处理的那40%恰恰是最消耗人工的复杂case。"
```

### 观点没有案例 → 补充真实场景或行业案例

```
❌ 干瘪写法：
"数据泄露风险需要关注。"

✅ 充实写法：
"LLM幻觉可能导致数据泄露。某国际咨询公司知识助手上线6个月后，
 有超过20%的已发布答案包含过时政策信息，其中3%导致员工
 按错误信息执行操作，引发合规风险。
 这个案例说明：AI知识的时效性管理比想象中更困难———
 不是'把文档扔进知识库'就完了，需要配套的更新机制。"
```

### 因果链断裂 → 补充中间推理步骤

```
❌ 干瘪写法：
"等保三级认证周期6-12个月，是最长路径瓶颈。"

✅ 充实写法：
"等保三级认证周期6-12个月（不含整改时间），是最长路径瓶颈。
 三级系统的核心要求包括：
  ① 审计日志不可篡改（WORM存储或区块链存证）——需要额外采购
  ② RBAC细粒度权限控制——国产LLM API不支持，需应用层实现
  ③ 可信计算芯片——硬件层面要求
  ④ 数据分类分级 + 重要数据境内存储——可能限制公有云模式
 这些要求需要技术改造时间（约3-6个月），
 叠加测评机构排队时间（约2-3个月），
 再叠加整改时间（如有问题，约1-3个月），
 合计6-12个月是保守估计。"
```

## 去AI味的正确时机和方式

### 错误时机

```
❌ "去AI味"作为重写全文的理由
  → 实际上是想做内容深化，把"去AI味"当成重写的借口
  → 正确理由应该是"内容需要深化"

❌ 在结构还没确认时就想去AI味
  → 表达是内容的载体，内容还没定，先不要管表达

❌ 删除所有逻辑连接词
  → "首先/其次/最后"有时候是必要的结构标记
  → 过度删除会让文章失去连贯性
```

### 正确时机

```
✅ 内容已经充实、逻辑已经完整后
  → 逐段检查表达自然度
  → 用自然语气替代机械套话
```

### 应该删除的表达

```
机械套话（直接删）：
  - "为您整理了……"
  - "我来帮您分析……"
  - "值得注意的是……"
  - "换句话说……"
  - "综上所述……"
  - "从以上分析可以看出……"

过度使用的连接词（保留必要的）：
  - "首先/其次/最后" → 保留当结构标记作用时
  - "然而/但是" → 保留当真正转折时
  - "因此/所以" → 保留当真正因果推导时
```

### 应该增加的表达

```
自然对话语气：
  - "这个数字背后的问题是……"
  - "举一个例子……"
  - "读者可能会问：……"
  - "实际上，这里的关键在于……"
  - "这不是……的问题，而是……的问题"

具体性：
  - "高" → 具体数字 "准确率从60%提升到85%"
  - "可能" → 说明概率 "概率约30-40%"
  - "很大" → 说明程度 "差距3.7倍"
```

## 典型失败案例：从"干瘪"到"重写全文"的错误链

以下是一个真实发生过的失败案例，展示了从"用户说干瘪"到"重写全文"的完整错误链：

```
"告诉我怎么用？" → 多智能体能力介绍

↓
"全文略显干瘪，每个章节不够充分。
 请根据文档中的内容主旨，补充细化内容。
 确保该需求分析清晰、详实、数据源真实、预测预估恰当。
 充分检索查找数据。去AI味，优化突兀的表达。"

↓
系统错误判断流程：
  Step 1: 用户说"干瘪" → 误判为"结构问题"（越界内容）
  Step 2: 重新设计结构 → 触发"哪些内容不必要"
  Step 3: 用户说"不需要精简" → 误解为"全部扩充"
  Step 4: 重写全文（46KB → 110KB）

↓
系统错误原因：
  - 已有三角评审通过的结论（结构是对的）
  - 已有用户确认过的边界清晰版模板
  - 应该针对具体章节增量修改，而非重写全文

↓
正确做法：
  Step 1: 判断结构是否有问题 → 有三角评审结论 → 结构OK
  Step 2: 问用户具体不足的是什么类型的内容
  Step 3: 针对具体章节做增量修改（保留结构）
  Step 4: 完成后做去AI味检查

↓
方法论固化：
  - "干瘪"可能是结构/深度/表达三种问题
  - 先判断类型再动手，不要假设
  - 已有评审结论时，不要用"干瘪"作为重写全文的理由
  - 去AI味是修改完成后的收尾工作，不是重写的出发点
```

## 快速检查清单

在动手修改"干瘪"问题前，先过一遍：

```
□ 有三角评审或用户确认过的结构方案吗？
□ 如果有 → 先问用户具体哪类内容不足，再针对性修改
□ 如果没有 → 先讨论结构
□ 准备修改的是"深度不足"还是"结构缺失"？
□ 去AI味放在修改完成后做，不作为重写的理由
□ 数字有来源标注吗？（[N/T/E]）
□ 数字有背景解释吗？（不只是数字，还有含义）
□ 观点有案例支撑吗？（不只是结论，还有例子）
□ 因果链完整吗？（不是"因为A所以B"，而是"A→B→C"）
```
