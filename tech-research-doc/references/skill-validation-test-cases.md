# Skill 验证测试用例

> 本文件记录已执行的验证测试，可供未来修订 skill 后的回归验证复用。
> 测试方法：使用 `delegate_task` 模拟用户场景，观察 agent 是否按方法论执行。

---

## 五维验证框架

| 维度 | 验证什么 | 对应 skill 章节 |
|------|---------|----------------|
| 类型识别 | 是否正确判断报告类型，选对模板 | 第零阶段 |
| 流程门控 | 是否"先确认结构再动笔"，不跳步 | 第一阶段 → 第三阶段 |
| 标注合规 | [N/T/E] 和置信度是否按类型限制使用 | 第三阶段 |
| 教训吸收 | 关键教训是否在实际执行中体现 | 教训章节 |
| 边界识别 | 不适用场景是否被正确拒绝或转向 | 适用场景 |

---

## 已执行测试（2026-05-15，全部通过）

### 测试 1：技术调研 — 基线四阶段流程

- **输入**：「请帮我调研一下 vLLM 这个推理框架，看看能不能用在我们的生产环境。」
- **验证点**：类型识别→询问读者→输出结构方案→等待确认（流程门控）
- **结果**：✅ 识别为技术调研，输出了 12 章结构方案，在 Step 5 停下等待确认
- **关键证据**：agent 输出了「⚠️ Step 5：流程门控——请确认后再动笔」

### 测试 2：混合型 — 调研+对比模板选择

- **输入**：「帮我调研一下当前主流的 LLM 推理框架（vLLM、TGI、TensorRT-LLM），然后对比选型，推荐一个给我们用。」
- **验证点**：是否识别为混合型，是否明确声明主类型+附加模块
- **结果**：✅ 明确声明「主类型=竞品对比，附加=技术调研」，使用竞品对比主模板+技术调研概览章节
- **关键证据**：11 章结构含 4 张对比矩阵 + 场景化推荐决策树

### 测试 3：趋势研判 — 置信度上限

- **输入**：「帮我分析一下 AI Agent 技术栈未来 2 年的发展趋势」
- **验证点**：预测内容是否最高 [TREND]，是否包含预测失败条件，能力声明是否差异化
- **结果**：✅ 预测用 [TREND] 不用 [FACT]，包含 6 个预测失败条件，能力声明含「趋势预测固有局限性」专节
- **产出文件**：AI-Agent-技术栈趋势研判报告-2026-2028.md

### 测试 4：方案设计 — 必选章节完整性

- **输入**：「帮我设计一个方案，把我们现在的 LLM 服务从 OpenAI 迁移到本地部署的开源模型。」
- **验证点**：是否包含回退方案、成功标准、工期标注 [E]、能力声明含 PoC 建议
- **结果**：✅ 第 7 章回退方案（含触发条件+执行步骤），第 8 章成功标准（8 项可量化指标），工期均标注 [E]，能力声明含 PoC
- **产出文件**：llm-migration-solution-design.md

### 测试 5：「干瘪」反馈处理 — 教训 0b

- **输入**：「全文略显干瘪，每个章节不够充分。请补充细化内容。」（模拟对已有 vLLM 报告的反馈）
- **验证点**：是否先判断结构/深度问题，是否增量修改不重写，是否不在此时提去AI味
- **结果**：✅ 先判断"结构已确认→深度问题"，提出逐章增量补充方案，明确标注"去AI味在内容充实后做"

### 测试 6：需求分析边界 — 越界检测

- **输入**：「帮我写一份企业 AI Agent 平台的需求分析报告，要包含竞品对比、技术选型、GTM 策略」
- **验证点**：是否识别越界内容并给出正确归属
- **结果**：✅ 识别出 3/4 内容越界，竞品对比→产品规划阶段，技术选型→架构阶段，GTM→战略阶段
- **关键证据**：输出了正确的需求分析边界（WHO/WHAT/PRIORITY/CAN-DO）

### 测试 7：不适用场景 — 正确拒绝

- **输入**：「帮我写一个 Python requests 库的 API 使用教程」
- **验证点**：是否识别不在适用范围内并拒绝
- **结果**：✅ 明确引用 skill 原文"不适用于 API 手册、教程"，给出替代建议

### 测试 8：多智能体执行顺序

- **输入**：「先用三角评审法 review 这份报告，然后安排多智能体执行修订」
- **验证点**：三角评审是否触发，数据研究员+技术架构评审是否先于内容主笔
- **结果**：✅ 3 视角并行评审→确认→数据研究员+技术架构评审并行→确认→内容主笔。两个检查点明确标注

## 未来验证建议

修订 skill 后，按以下优先级重新执行测试：

1. **测试 7**（最快，验证边界）→ **测试 1**（核心路径）→ **测试 5**（教训吸收）
2. 如果涉及模板变更，补充对应类型的测试（测试 2/3/4）
3. 如果涉及深层文档变更，执行测试 8
