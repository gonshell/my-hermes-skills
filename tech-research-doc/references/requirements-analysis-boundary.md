# 需求分析报告的边界：业务需求 vs 产品需求 vs 技术需求

## 权威方法论来源

| 标准/框架 | 核心贡献 |
|-----------|---------|
| **IEEE 830-1998** | SRS应避免设计痕迹，聚焦"做什么"而非"怎么做" |
| **IIBA BABOK v3.0** | 需求分层：业务需求 → 用户需求 → 解决方案需求 |
| **ISO/IEC/IEEE 29148** | 明确需求工程全生命周期，规定需求跟踪矩阵 |
| **Karl Wiegers《Software Requirements》** | 需求分类：业务/用户/功能/非功能/设计约束，明确边界 |

## 三层需求的定义与边界

```
┌─────────────────────────────────────────────────┐
│               业务需求 (Business) WHY              │
│          "为什么要做？带来什么业务价值？"            │
│          战略目标、ROI、KPI、业务规则               │
└─────────────────────────────────────────────────┘
                        ↓ 分解/转化
┌─────────────────────────────────────────────────┐
│               产品需求 (Product) WHAT              │
│          "产品应该具备什么功能？"                   │
│          用户故事、功能规格、验收标准、用例          │
└─────────────────────────────────────────────────┘
                        ↓ 细化/设计
┌─────────────────────────────────────────────────┐
│               技术需求 (Technical) HOW             │
│          "我们如何实现这些功能？"                   │
│      技术架构、接口规范、性能指标、技术选型          │
│      ← 需求分析阶段不涉及！                       │
└─────────────────────────────────────────────────┘
```

## 判断标准：这是需求还是设计？

| 问句 | 答案类别 |
|------|---------|
| "系统**做什么**？" | ✅ 需求 |
| "系统**如何做**？" | ❌ 设计/技术方案 |
| "系统**用什么技术**做？" | ❌ 实现决策 |

## 需求分析报告的正确边界

### 应该回答的4个问题

| # | 问题 | 需求分析产出 |
|---|------|------------|
| 1 | **WHO（谁来用）** | 干系人分析 + 用户画像 |
| 2 | **WHAT（需要什么）** | 功能需求清单（ID + 描述 + 类型） |
| 3 | **PRIORITY（先做哪个）** | P0/P1/P2优先级矩阵 |
| 4 | **CAN-DO（能做吗）** | 约束清单（合规/集成/性能/安全） |

### 不应该回答的问题（应标注"待后续阶段决策"）

| 问题 | 正确归属阶段 |
|------|------------|
| 用哪个LLM底座？ | 技术架构阶段 |
| 用哪个Policy Engine？ | 技术架构阶段 |
| 选哪个供应商？ | 采购/战略阶段 |
| 怎么进入市场？ | GTM战略阶段 |
| 竞品功能对比？ | 产品规划阶段 |
| 供应商锁定风险评估？ | 战略决策阶段 |
| 精确ROI计算？ | 商业可行性阶段 |

## 常见越界内容及处理方式

| 越界内容 | 处理方式 |
|---------|---------|
| "Policy Engine选型：OPA vs 商用方案" | 改为"细粒度权限管理能力"（需求描述），标注"实现方案待架构阶段决策" |
| "竞品对比：阿里 vs 百度 vs 华为" | 完全删除，不出现在需求分析报告中 |
| "市场进入建议：先切入政务/金融" | 完全删除，属于GTM战略 |
| "供应商锁定风险：阿里依赖度高" | 完全删除，属于战略决策阶段 |
| "ROI = 3.2倍"（无假设条件） | 改为"估算2.0-2.5倍 ⚠️（基于假设X/Y/Z）" |
| "技术架构：RAG + Multi-Agent" | 改为"需要知识检索和协作能力"（需求属性） |

## 需求文档的正确结构

```
需求分析报告
├── 1. 文档概述（背景/目的/范围/术语）
├── 2. 调研方法说明（一手/二手/推断分类，⚠️可信度标注）
├── 3. 干系人分析与用户画像
├── 4. 功能需求清单（ID + 描述 + 提出者 + 可信度）
├── 5. 需求分类与优先级（P0/P1/P2矩阵）
├── 6. 约束条件识别（合规/集成/性能/安全）
├── 7. 风险与假设
├── 8. ROI估算（粗粒度，含⚠️假设条件）
└── 附录（术语表/数据来源/争议汇总）
```

## 需求分析报告 vs 其他文档类型

| 文档类型 | 回答 | 不回答 |
|---------|------|-------|
| **需求分析报告** | WHO/WHAT/PRIORITY/CAN-DO | 技术选型/供应商/GTM |
| **技术架构文档** | HOW（技术实现方案） | 市场策略/采购决策 |
| **GTM战略** | 如何进入市场/竞争策略 | 需求定义/技术选型 |
| **商业可行性报告** | 精确ROI/投资回报周期 | 具体功能规格 |

## 教训：为什么混界是致命问题

需求分析混入技术选型会导致：
1. **决策者困惑**：看完不知道该做什么（需求）vs该选什么（技术）
2. **架构师绑架**：早期锁定了技术方案，限制了后续设计空间
3. **返工风险**：技术选型决策后，如果方向调整，需求分析需重写
4. **边界模糊**：评审时无法判断问题该在哪个阶段解决

**正确顺序：需求分析 → 技术架构 → 采购/供应商选择 → GTM战略**
