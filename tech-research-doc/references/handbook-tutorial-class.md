# 技术实战手册 / 入门教程类文档

技术研究报告（6 种类型）和**实战手册/入门教程**是两类不同的文档——读者目的、目标体验、章节组织方式都不同。本 reference 描述实战手册的独立方法论，并附本 session 学到的关键教训。

---

## 与研究报告的核心区别

| 维度 | 研究报告 | 实战手册 / 入门教程 |
|------|---------|-------------------|
| 读者目的 | 形成判断、做出决策 | 学会操作、解决具体问题 |
| 读后状态 | 能说出"应该选 A" | 能照着命令跑通 |
| 章节组织 | 递进式论证（现象→根因→方案） | 按"读者会卡住的顺序"组织 |
| 数据来源 | 大量 Datasheet / 论文 / Benchmark | **官方文档、FAQ、命令参考** |
| 篇幅倾向 | 越精炼越好 | 越覆盖"踩坑点"越好 |
| 核心资产 | 推理链、置信度标注 | 命令、配置、坑清单 |

**判断标准**：用户说"做一份实战手册 / 快速实践 / 入门教程 / 快速上手 / quickstart" → 走本 reference；说"调研 / 选型 / 趋势 / 评估" → 走 6 种研究报告类型。

---

## 实战手册的章节组织原则

### 按"读者会卡住的顺序"组织，不按"功能模块的目录顺序"

**反例**（按官方文档目录顺序抄）：
- 第一章：CLI Reference
- 第二章：Configuration
- 第三章：Messaging Gateway
- 第四章：MCP
- 第五章：Cron
- 第六章：Memory
→ 读者读完 CLI Reference 仍然不知道"装完第一步干什么"

**正例**（按读者实际操作顺序）：
1. 心智模型（30 秒搞清楚 4 个东西）
2. 装完必做的 N 件事（按使用频率排）
3. **三件套**——如果该系统有多个相似概念要分层讲（如 SOUL.md / AGENTS.md / MEMORY.md）
4. 最高频的 2-3 个扩展功能
5. 排障速查
6. 按目标选深入路径

**关键**：手册是"导游路线图"，不是"目录索引"。读者要的是"先走哪、再走哪"。

### 必须包含的 5 个部分

| 部分 | 目的 | 篇幅占比 |
|------|------|---------|
| 30 秒心智模型 | 4 个核心文件 / 4 个核心概念 | 5% |
| 装完必做的事 | 5-7 个最高频操作 | 25% |
| 关键概念分层 | 容易混淆的相似概念（如身份/项目/记忆） | 15% |
| 3 个最常用扩展功能 | 选 2-3 个最常用的深入讲 | 30% |
| 排障速查表 + 大家常问的 N 个问题 | 症状→命令 / FAQ | 25% |

### 主动取舍（不贪多）

实战手册的本质是**减法**——把 200 页官方文档砍到 30 分钟能读完的精华。

**取舍原则**：
- ✅ 保留：日常 80% 场景会用的
- ❌ 砍掉：6+ 种配置变体、9+ 种扩展来源、全量 Provider 列表、CLI 全量命令
- 🎯 立场：速查级，不复述官方
- 📌 每节末尾标"想深入看哪个官方页面"——把读者导回官方

**反过度设计**（用户多次纠正的偏好）：
- 写之前先问"最常用的 5-6 件事"是什么，不是"功能目录里有什么"
- 5 个最高频 + 2-3 个最常用扩展 > 全覆盖但每节都浅

## ⚠️ 关键教训：研究/写作前的资料核查

### 本 session 失败案例（2026-06-08）

**场景**：用户要求"参考官方网站做一份 hermes-agent 快速实践手册"。**我做了一个 10 节目录大纲提交给用户。**

**失败做法**：
- 用 `delegate_task` 派发子 agent，让它去抓官方 6 个页面的代码块
- 收到子 agent 报告后，没亲自读任何一页
- 凭子 agent 的二手摘要直接编目录
- 第一版目录漏掉了 **SOUL.md / AGENTS.md / .cursorrules 三件套**，漏掉了 **官方 FAQ 的高频问题**

**用户原话**：
> "不是，你浏览了网站么？看了他的手册么？请核对手册材料... 请深入思考，好好整理问题，规划目录结构、内容组织，再开始写。"

**用户原话（第二轮）**：
> "SOUL.md、BOOT.md 等各种设定呢？"

→ 第二轮才意识到 SOUL.md / AGENTS.md 是官方分层的核心概念，**第一版完全漏掉**。

### 失败原因剖析

`delegate_task` 用于资料收集时，存在两个系统性偏差：

| 偏差 | 表现 | 后果 |
|------|------|------|
| **目录错觉** | 子 agent 报告里"列了所有 H2/H3 标题" | 主 agent 误以为"我已读完全部内容"，实际只读了标题 |
| **代码块幻觉** | 子 agent 报告里"列了 verbatim YAML/命令" | 主 agent 误以为"命令是对的"，但可能漏了关键上下文（坑、注意、版本） |
| **判断力外包** | 把"哪节该写、不该写"的判断也外包出去 | 漏掉用户常识但子 agent 不认为是重点的内容（如 SOUL.md 三件套） |

### 正确做法（强制流程）

```markdown
写实战手册 / 教程类文档前的强制流程：

1. ⏸️ 不要直接 delegate_task 就完事
2. 👀 至少亲手读 3-5 个核心页面的完整正文
3. 📋 列出"读者会卡住的具体问题"清单（不要只列章节标题）
4. 🎯 明确"最常用的 5-6 件事"是什么
5. 🗣️ 把目录+定位先发给用户确认，再动笔
6. ✍️ 写完后自检：5 个最容易踩的坑是否覆盖？

强制门槛：必须能在不看子 agent 报告的情况下，
凭自己读的内容回答"三件套是什么"、"FAQ 第 3 个问题是什么"。
```

### ⏸️ 哪些情况下 delegate_task 仍然 OK

- ✅ 数据收集**且**只需汇总（多个 Benchmark 数字、多个版本号）
- ✅ 任务明确且能自我验证（"找到所有带 X 的 URL"）
- ❌ **不适用于**：写手册 / 教程 / 文档——这些需要主 agent 自己的判断力

**判断标准**：
> 如果任务的核心是"理解内容并做出取舍"，主 agent 必须亲自读。
> 如果任务的核心是"机械汇总并归一化"，子 agent 可以代理。

### 子 agent 返回结果的二次核查

即使子 agent 报告很详尽，仍要做**最少 3 项核查**：

1. **术语核查**：用户提到的核心概念（如 SOUL.md / AGENTS.md）子 agent 是否覆盖？
2. **数字核查**：官方文档里关键的 2-3 个数据点（如命令名、版本号）子 agent 报告里是否准确？
3. **遗漏核查**：翻官方页面侧边栏 / TOC，**比对子 agent 给的章节列表**——差太多的就是漏了

## 实战手册的常见错误模式

### 错误 1：照抄官方目录

| 症状 | 表现 | 应做 |
|------|------|------|
| 章节顺序跟官方 docs 一样 | "第一章 CLI 第二章 Config 第三章..." | 按读者实际操作顺序重排 |

### 错误 2：只给命令不给场景

```markdown
❌ 错：
hermes cron create "30m" "..."

✅ 对：
**场景**：每 30 分钟提醒一次
hermes cron create "30m" "Remind me to check the build"
```

### 错误 3：堆砌代码不解释

```markdown
❌ 错：
mcp_servers:
  github:
    command: "npx"
    args: [...]
    env: {...}

✅ 对：
**Stdio 模式**（本地跑的进程）：

```yaml
mcp_servers:
  github:
    command: "npx"           # 启动 npx
    args: [...]              # 跑这个 npm 包
    env: {...}               # 注入的 API key
```
```

### 错误 4：把"全部 20+ Provider 列表"塞进手册

实战手册里加 1-2 个最常用的例子，**末尾标"完整列表见官方"**就行。

### 错误 5：不区分"实例级"vs"会话级"vs"项目级"

很多系统有"看起来像、但作用域不同"的概念（SOUL.md / AGENTS.md / MEMORY.md）。实战手册**必须**用一节专门讲清"哪一层管什么"，否则用户永远搞不清"我改完一个文件为啥没生效"。

## 自检清单（实战手册定稿前）

- [ ] 心智模型在第 0 节，30 秒能讲清核心文件 / 核心概念
- [ ] 章节顺序按"读者会卡住的顺序"，不是按官方目录
- [ ] 覆盖了 5-7 个"装完必做的事"
- [ ] 关键相似概念有专门的分层讲解（如身份 / 项目 / 记忆）
- [ ] 排障速查表存在（症状 → 命令）
- [ ] 至少 3 个大家常问的 FAQ（来自官方 FAQ 页面）
- [ ] 主动取舍——砍掉了哪些、为什么砍掉，写在引言里
- [ ] 每节末尾标"想深入看哪个官方页面"——不抄官方内容
- [ ] 2-3 个端到端实战场景（不是单个命令的孤立例子）
- [ ] 关键文件位置速查附录
- [ ] 写作前主 agent 亲自读至少 3-5 个核心页面（不是只读子 agent 报告）
