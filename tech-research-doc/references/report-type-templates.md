# 6种报告类型模板

本文档为每种报告类型提供：标准结构模板、必选/可选章节标记、最小可行结构、质量检查清单。

---

## 1. 技术调研（Technology Survey）

**核心问题**："是什么？能不能用？"

### 标准结构

```
1. 执行摘要（必选）
   - 一段话说清楚：这是什么技术、成熟度如何、能不能用
2. 背景与目标（必选）
   - 为什么要调研、调研范围和边界
3. 术语表（可选，跨团队时必选）
4. 技术概览（必选）
   - 核心概念、架构图、关键组件
5. 核心特性分析（必选）
   - 每个特性用「现象→根因→方案」三层结构
6. 生态与社区（必选）
   - GitHub stars/贡献者、商业支持、文档质量
7. 适用场景分析（必选）
   - 适合什么、不适合什么
8. 落地路径（必选）
   - 上手难度、工期估算、人员要求
9. 已知风险与局限（必选）
10. 参考文献（必选）
11. AI 能力边界声明（必选）
```

### 最小可行结构（时间有限时）

```
执行摘要 → 技术概览 → 核心特性（3-5个）→ 适用场景 → 风险 → 参考文献
```

### 质量检查清单

- [ ] 执行摘要是否能让决策者5分钟内做出"继续/停止"判断
- [ ] 每个特性是否包含 trade-off（优点+缺点）
- [ ] 是否有"不适合"的场景（而非只说好话）
- [ ] 生态数据是否有时间标注（"截至 2026-05"）
- [ ] 所有数据标注了 [N/T/E]
- [ ] 置信度只使用了 [FACT] 和 [HCF]
- [ ] AI 能力边界声明已包含

## 2. 深度分析（Deep Analysis）

**核心问题**："为什么？底层原理是什么？"

### 标准结构

```
1. 执行摘要（必选）
   - 核心发现：为什么是这样、根因是什么
2. 问题定义（必选）
   - 现象描述、为什么值得深入分析
3. 背景知识（必选）
   - 读者理解分析所需的预备知识
4. 推理模型（必选）⚠️ 核心章节
   - 构建分析框架/因果模型
   - 明确假设和约束
   - 每步推理链可见
5. 逐层分析（必选）
   - 按推理模型逐层展开
   - 每层结论标注置信度
6. 验证与反证（必选）
   - 有没有反例？反例如何解释？
   - 是否存在替代解释？
7. 结论与推论（必选）
   - 分析结论 + 可推出的进一步结论
8. 未解问题（必选）
   - 本次分析未能回答的问题
9. 参考文献（必选）
10. AI 能力边界声明（必选）
```

### 最小可行结构

```
问题定义 → 推理模型 → 逐层分析（3-5层）→ 验证与反证 → 结论
```

### 质量检查清单

- [ ] 推理模型是否画了因果图或逻辑链
- [ ] 每步推理是否有前提条件声明
- [ ] 是否考虑了替代解释（不只一个答案）
- [ ] 是否有反例或边界情况讨论
- [ ] 置信度可使用到 [REASON]，但必须写明推理链
- [ ] [JUDGMENT] 不作为核心结论支撑
- [ ] AI 能力边界声明已包含

## 3. 趋势研判（Trend Forecast）

**核心问题**："往哪走？何时入场？"

### 标准结构

```
1. 执行摘要（必选）
   - 核心判断：技术走向 + 建议行动时机
2. 技术现状（必选）
   - 当前阶段（萌芽/成长/成熟/衰退）
   - 关键参与者和生态格局
3. 信号捕捉（必选）⚠️ 核心章节
   - 技术信号：论文、开源项目、专利趋势
   - 市场信号：融资、采用率、人才流动
   - 政策信号：标准制定、监管动态
4. 时间线构建（必选）
   - 过去：关键技术节点和转折点
   - 现在：当前处于什么位置
   - 未来：预测关键节点 + 置信度
5. 竞争格局演进（必选）
   - 各参与者的战略动态
   - 预测格局变化
6. 入场时机建议（必选）
   - 什么时候投入？
   - 早期/中期/晚期的收益-风险比
7. 风险因素（必选）
   - 预测可能失败的原因
   - 黑天鹅事件识别
8. 参考文献（必选）
9. AI 能力边界声明（必选）⚠️ 必须强调预测的局限性
```

### 最小可行结构

```
技术现状 → 信号捕捉（3类信号）→ 时间线 → 入场建议 → 风险
```

### 质量检查清单

- [ ] 信号是否有来源标注（论文/融资数据/政策文件）
- [ ] 时间线是否标注了置信度（[FACT]/[TREND]/[REASON]）
- [ ] 是否有明确的预测失败条件
- [ ] 入场建议是否区分了不同组织规模/阶段
- [ ] 没有把 [TREND] 当 [FACT] 使用
- [ ] AI 能力边界声明特别强调了预测局限性
- [ ] 参考文献≥5个（趋势类对信息量要求高）

## 4. 竞品对比（Competitive Analysis）

**核心问题**："选哪个？A vs B？"

### 标准结构

```
1. 执行摘要（必选）
   - 结论：推荐选哪个、为什么
2. 评估范围（必选）
   - 参评对象列表、排除对象及原因
   - 评估维度说明
3. 对比矩阵（必选）⚠️ 核心章节
   - 功能对比表
   - 性能对比表（有量化数据）
   - 生态对比表
4. 逐维度分析（必选）
   - 每个维度：评分 → 理由 → 数据来源
   - 不只列数据，要解释差异的根因
5. 场景化推荐（必选）
   - 不同场景选不同方案
   - 决策树形式
6. 迁移成本（必选）
   - 从 A 切到 B 的代价
   - 锁定风险评估
7. 综合评分与推荐（必选）
   - 加权评分（权重需声明假设）
8. 参考文献（必选）
9. AI 能力边界声明（必选）
```

### 最小可行结构

```
评估范围 → 对比矩阵（功能+性能）→ 场景化推荐 → 综合推荐
```

### 质量检查清单

- [ ] 对比维度是否有明确的选取理由
- [ ] 每个对比项有数据支撑（非仅主观判断）
- [ ] 是否包含"不推荐"的场景
- [ ] 迁移成本是否考虑了隐性成本（学习曲线、社区支持）
- [ ] 评分权重是否声明了假设
- [ ] 置信度只使用了 [FACT] 和 [HCF]
- [ ] 没有把任意一方作为"默认推荐"

## 5. 方案设计（Solution Design）

**核心问题**："怎么做？分几步？风险在哪？"

### 标准结构

```
1. 执行摘要（必选）
   - 方案概述：目标、路径、工期、风险等级
2. 目标与约束（必选）
   - 要解决什么问题
   - 硬约束（预算/时间/技术栈/合规）
3. 方案总览（必选）⚠️ 核心章节
   - 架构图/流程图
   - 关键决策点
4. 实施路径（必选）
   - 分阶段计划（含里程碑）
   - 每阶段输入/输出
   - 工期估算 + 置信度
5. 技术方案详解（必选）
   - 核心组件设计
   - 接口/数据流
   - 关键配置
6. 风险与缓解（必选）
   - 技术风险、人员风险、时间风险
   - 每个风险的缓解措施
7. 回退方案（必选）
   - 如果方案失败，怎么退回
8. 成功标准（必选）
   - 如何判断方案成功
   - 可量化的验收指标
9. 参考文献（必选）
10. AI 能力边界声明（必选）
    ⚠️ 必须声明：方案未经实际验证，建议先做 PoC
```

### 最小可行结构

```
目标与约束 → 方案总览（架构图）→ 实施路径（分阶段）→ 风险 → 成功标准
```

### 质量检查清单

- [ ] 硬约束是否逐条列出
- [ ] 架构图/流程图是否包含
- [ ] 实施路径是否分阶段（不是一锅粥）
- [ ] 每个阶段有明确的完成标准
- [ ] 风险是否有缓解措施（不只是列出风险）
- [ ] 回退方案是否存在
- [ ] 成功标准可量化
- [ ] 工期估算标注了 [E] 或 [T]

## 6. 状态评估（Status Assessment）

**核心问题**："到什么水平？差距多大？"

### 标准结构

```
1. 执行摘要（必选）
   - 当前水平概述、关键差距、优先改善方向
2. 评估框架（必选）
   - 评估维度和权重
   - 评分标准说明
   - 基准线（对标对象）
3. 现状盘点（必选）⚠️ 核心章节
   - 逐维度评分 + 证据
   - 与基准线的差距
4. 差距分析（必选）
   - 每个差距的根因
   - 差距严重程度排序
5. 优势盘点（必选）
   - 不仅是找差距，也要盘点优势
6. 提升路径（必选）
   - 按优先级排序的提升建议
   - 预期投入和收益
7. 附录：评估数据（必选）
   - 原始数据、评分依据
8. 参考文献（必选）
9. AI 能力边界声明（必选）
   ⚠️ 必须声明：评估基于公开信息和逻辑推理，非实地审计
```

### 最小可行结构

```
评估框架 → 现状盘点（评分+证据）→ 差距分析 → 提升路径
```

### 质量检查清单

- [ ] 评估框架是否在评估前与用户确认
- [ ] 每个维度评分是否有数据或事实支撑
- [ ] 基准线是否明确（对标谁）
- [ ] 差距是否区分了严重程度
- [ ] 是否包含优势（而非只找问题）
- [ ] 提升建议是否按优先级排序
- [ ] 置信度只使用了 [FACT] 和 [HCF]

## 混合型组合规则

| 混合类型 | 处理方式 | 主模板 | 叠加模块 |
|---------|---------|--------|---------|
| 调研+对比 | 最常见，先搞清是什么再选型 | 技术调研 | 对比矩阵+场景化推荐 |
| 分析+方案 | 先诊断再开药 | 深度分析 | 实施路径+风险缓解 |
| 调研+趋势 | 先摸底再判断方向 | 技术调研 | 信号捕捉+时间线 |
| 对比+方案 | 选型后给落地步骤 | 竞品对比 | 实施路径+回退方案 |
| 评估+方案 | 审计后给改善计划 | 状态评估 | 提升路径扩展为完整方案 |

**原则**：主类型决定四阶段主流程，附加模块从对应类型中抽取相关章节插入。
