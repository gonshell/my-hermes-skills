# 洞见模板与调研建议模板

> 从实战中固化的两个可复用模板，适用于趋势研判、深度分析、情报简报等分析类文档。

## 一、洞见模板（v2）

### 旧模板的问题（已废弃）

```
现象：[数据罗列]
可能的"人"解释（2-3 个）：
- 技术：[解释]
- 产品：[解释]
- 商业：[解释]
未验证：[需 XX 数据]
```

四个结构性缺陷：
1. **现象段是"数据归纳"不是"反常发现"** — 用数字证明话题值得关注，但读者已看标题，不需要被说服
2. **三维框架（技术/产品/商业）是预设的** — 导致硬凑；有些洞见在某个维度上无话可说
3. **解释之间无优先级** — 并列 = 放弃判断
4. **未验证段只说"缺数据"** — 不给替代假设，无法证伪

### 改进后模板

```
现象：[1-2 句反常描述 + 判断钩子]
可能的解释（维度名从现象涌现，不预设）：
- [维度名即问题]：[解释]
- [维度名即问题]：[解释]
- [维度名即问题]：[解释]
倾向：[最相信哪个 + 一句理由]
未验证 / 替代假设：[如果洞见是错的，最可能的替代解释 + 怎么区分]
```

### 四个改进点的规则

**改进 A：现象段 = 反常断言，不是数据罗列**

规则：用 1-2 句话描述一个具体的反常，不用聚合数字。如果一定要用数字，只用对比性数字（一个 vs 另一个）。最后一句必须是"判断钩子"——把标题的抽象概念落地的断言。

❌ "B 站 22 条总播放 1.07 亿，是晚间档 899 万的 11.9 倍。11 条教程总时长 100+ 小时。"
✅ "B 站排名第一的不是 AI 新闻，是搬运 MIT 课程的账号。高播放 ≠ 热点，高播放 = 补课。"

**改进 B：维度名从现象涌现，不强制三维**

规则：每个洞见的解释维度不预设数量和标签。维度命名是具体的判断句（"为什么是搬运+中文字幕而不是原创"），不是通用分类（"技术/产品/商业"）。

好处：读者扫一眼维度名就知道你在回答什么问题。

**改进 C：加"倾向"行**

规则：解释段之后标明你最相信哪个解释 + 一句理由。这不是结论，是立场标记——"在现有证据下我暂时站这里，但有理由可能让我移动"。

**改进 D：未验证段升级为"替代假设"**

规则：至少包含一个替代假设——"如果这个洞见是错的，最可能的另一种解释是什么"。加上"怎么区分两种解释"的验证方法。

❌ "需 UP 主历史数据 + B 站推荐机制透明度"
✅ "替代解释是'算法在推课不是用户在补课'。区分方法：看这些教程的完播率（主动补课 → 高完播；算法推流 → 低完播）。"

### 完整示例（洞见 2：B 站再教育基础设施）

```
现象：B 站排名第一的不是 AI 新闻，是搬运 MIT 课程的账号（AlfredTaylorHD 1046 万）。
22 条里 11 条是教学，但播放量最高的那条跟 AI 无关。高播放 ≠ 热点，高播放 = 补课。

可能的解释：
- 为什么是"搬运 + 中文字幕"而不是原创：中文技术圈对英文教学资源的依赖是结构性的
  ——不是知识缺口，是语言缺口。头部 UP 主做的是本地化分发，不是知识生产。
- 为什么集中在这个时间点：2026 H1 是 LLM 能力跨过"可用"阈值，补课需求集中爆发。
- B 站的生态选择：把"难学"的长视频教学当护城河，对抗抖音/小红书的泛知识侵蚀。

倾向：更相信"结构性依赖"——如果只是补课需求，不会集中出现"搬运 + 重组"这种特定形态。

未验证 / 替代假设：如果"再教育基础设施"是错的，替代解释是"算法对长视频教学内容的
系统性倾斜"。区分方法：看这些教程的完播率（主动补课 → 高完播；算法推流 → 低完播）。
```

## 二、调研建议模板（v2）

### 旧模板的问题（已废弃）

```
1. **话题标题** → 对应洞见 N（§x.x）
```

三个缺陷：
1. 是"话题清单"不是"可执行调研指令"——读完不知道第一步做什么
2. 不区分调研类型（技术/市场/商业/政策）——无法判断该用什么方法
3. 交叉引用方向反了——指向"已分析完的洞见"，而不是"洞见留下的未验证假设的下一步"

### 改进后模板

```
### 建议 N：[一句核心问题，不是话题标题]

**类型：** 技术 / 市场 / 商业 / 政策 / 战略分析
**为什么现在做：** [时间窗口 + 不做会错过什么]
**切入路径：**
  1. [第一步：具体动作 + 数据源]
  2. [第二步]
  3. [第三步]
**预期产出：** [查完手里有什么——一个判断？一个数据集？一个框架？]
**来源锚点：** 洞见 N（§x.x）的未验证假设 [一句话复述]
```

### 字段设计理由

| 字段 | 为什么必须有 |
|---|---|
| 核心问题（不是标题） | 把"话题"变成"问题"。问题有答案，话题没有。 |
| 类型 | 决定方法（读论文 vs 爬数据 vs 读财报 vs 读法规） |
| 为什么现在做 | 替代"评估口径"。标时间窗口和紧迫性，安排优先级 |
| 切入路径 | 把"去调研"变成"第一步做 X"。解决"不知道从哪开始" |
| 预期产出 | 开始前就知道"做完算完成"。防止调研发散 |
| 来源锚点 | 不是"对应洞见 N"，而是"洞见 N 的哪个未验证假设驱动了这个建议" |

### 调研建议的分类

| 类型 | 方法 | 典型产出 |
|---|---|---|
| 技术 | 读论文 / 跑模型 / 对比 benchmark | 技术笔记 + 技术判断 |
| 市场 | 爬数据 / 趋势分析 / 竞品对比 | 判断 + 数据集 |
| 商业 | 读财报 / 价格追踪 / 竞争分析 | 追踪基线 + 走势判断 |
| 政策 | 读法规 / 找处罚案例 / 跨国对比 | 合规清单 + 政策方向判断 |
| 战略分析 | 框架分析 / 多源推理 | 可行性框架 + 对比矩阵 |

### 完整示例（建议 2：B 站教学流量）

```
### 建议 2：B 站 AI 教学流量是"用户补课"还是"算法推流"？

类型：市场
为什么现在做：11 条教学视频总播放 1.07 亿，头部集中度极高。如果这是结构性蓝海，
6-12 个月内生态位固化。现在进入窗口正在关闭。
切入路径：
  1. 爬取 11 条教程的完播率/弹幕密度/投币比（B 站 API），判断主动学习 vs 被动推流
  2. 追踪头部 UP 主过去 6 个月发布频率和播放趋势
  3. 对比抖音/小红书同类教学量级，判断 B 站是否独占这个生态位
预期产出：一个判断（结构性蓝海 vs 算法假象）+ 一个数据集
来源锚点：洞见 2 的替代假设——完播率区分"算法推课 vs 用户补课"
```

### 调研建议来源不限于洞见

调研建议可以来自两类：
1. **洞见延伸**（主）：来自洞见的未验证假设/替代假设 → 设计验证路径
2. **事实层直接学习需求**（补充）：来自事实综述中的技术/政策事件，不经过洞见环节

示例（非洞见驱动的纯技术学习）：
```
### 建议 5：1M token 上下文 + 代码模型专项化的技术实现路径
类型：技术学习
来源锚点：§2.1 技术事实——非洞见驱动，来自技术事实层的直接学习需求
```

## 三、文档架构模式：分层密度

适用于情报简报、行业动态、分析日报等"多信息密度混合"文档。

### 核心问题

文档里通常有 4 层信息密度，但被硬塞进同一个章节层级：
- L1 元信息（数据来源、评估口径）
- L2 事实综述（事件罗列）
- L3 反方观点（反面论证）
- L4 洞见（深度推理）

读者扫目录无法预知每章的信息密度——这是"读不快"的根本原因。

### 解决方案：每章内部强制分层

```
文档头：元信息 block（callout，不占编号）
  ↓
§N 主题章节
  §N.1 事实（L2）   ← H3
  §N.2 反方（L3）   ← H3，如有
  §N.3 洞见 N（L4） ← H3，全局编号，如有
```

### 关键设计规则

| 规则 | 做法 |
|---|---|
| 洞见全局编号 | `洞见 1`~`洞见 N`，跨章节连续，可被引用 |
| 反方全局编号 | `反方 1`~`反方 N`，跨章节连续 |
| 事件跨章不重复 | 同一事件只在一个"主场"展开，其他章节用一句话引用 |
| 无洞见章节如实标注 | `callout：本章为事实层，暂无洞见`，不强造 |
| 元信息降级 | 数据来源、评估口径 → 顶部 callout，不占 H2 编号 |
| 输出物后置 | 调研建议、行动建议 → 文末（在所有分析之后），不前置 |
| 附录不占主体编号 | 视频/数据索引 → 附录 A |
