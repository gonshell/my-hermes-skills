# Session 整合报告生成模式

> 新增于 2026-06-21 session。源案例：宇通客车研究 session 整合为一份完整报告。

## 适用场景

用户要求把**一个长 session 内的所有产出整合成一份最终报告**：

- "请将这个对话 session 中每次对话的最终版本的内容整合一个完整的报告版本"
- "整合整个 session 的内容"
- "把所有讨论的成果汇总成一份文档"
- "请生成最终交付版"

**关键信号**：
- session 涉及多轮讨论（通常 5+ 轮）
- 每次讨论产出了独立文件（*.md, *.py 等）
- 中间过程包含了试错、追问、反思
- 用户明确要求"整合"而非"重写"

## ⚠️ 核心原则

1. **去掉所有过程**：讨论、追问、试错、反思过程全部删除
2. **保留所有最终结论**：以"权威报告"形式呈现
3. **统一数据时点**：所有数据统一到一个截止日期（通常是报告生成日）
4. **逻辑递进**：从现状到原因到护城河到估值
5. **章节独立**：每章有独立价值，读者可跳读

## 标准动作链

### 步骤 1：先列出 session 内的所有产出

```
反例：直接开始写整合报告
正例：先列出 session 内所有产出文件/章节
```

**清单模板**：

| # | 阶段 | 产出物 | 性质 |
|---|---|---|---|
| 1 | 阶段 A | xxx_xxx_xxx.md | 时点数据 |
| 2 | 阶段 B | xxx_xxx_xxx.md | 因果分析 |
| 3 | 阶段 C | xxx_xxx_xxx.md | 战略评估 |
| 4 | 反思 | 自批评记录 | 元分析 |
| 5 | 重写 | xxx_诚实版.md | 修订版 |

**判断每个产出的"价值"**：
- ✅ 保留：最终结论 + 数据 + 决策建议
- ⚠️ 部分保留：方法论 + 关键发现
- ❌ 删除：试错过程 + 追问记录 + 自我反思

### 步骤 2：决定整合策略

**策略 A：完全重写**（适用于碎片化产出）
- 优点：结构统一
- 缺点：可能丢失原始叙述

**策略 B：抽取 + 重排 + 改写**（推荐）
- 优点：保留原文措辞，控制改动
- 缺点：需要较多时间

**策略 C：保留原始文件 + 写摘要**（适用于多版本场景）
- 优点：版本可追溯
- 缺点：用户需要看多份文件

**实战推荐**：B 策略（抽取 + 重排 + 改写）。原因是：
- 用户的"整合"意思是"我要一份干净的最终版"
- 但要避免"为了整合而大改"导致内容失真
- 抽取关键内容 → 按新目录重排 → 改写为独立连贯的章节

### 步骤 3：设计新目录（基于内容逻辑，非原文件结构）

**目录设计原则**：
- 从"是什么"到"为什么"到"怎么办"
- 每章 500-2000 字
- 章节间逻辑递进

**典型整合报告结构**（8 章 + 3 附录）：

```
一、执行摘要（核心结论 1 页可读）
二、公司/对象概况（基础事实）
三、当前状态（时点数据）
四、动因分析（因果）
五、护城河/竞争优势（战略）
六、风险与替代威胁
七、估值/预测参考（区间）
八、研究方法与数据声明

附录 A：核心数据汇总
附录 B：参考来源
附录 C：报告价值与使用建议
```

### 步骤 4：抽取关键内容（按主题，非按文件）

```
❌ 错误做法：按文件抽取，整块搬过来

✅ 正确做法：按主题抽取，散落在多文件的内容合并
```

**抽取表模板**：

| 主题 | 来源文件 | 关键内容 |
|---|---|---|
| 半年股价走势 | A.md §1, B.md §2 | 数据 + 走势 |
| 业绩拐点分析 | B.md §3, C.md §1 | 因果链 |
| 护城河 | D.md §5, E.md §2 | 横向对比 |
| 估值 | F.md §1, G.md §3 | 区间 |

### 步骤 5：统一数据时点

**所有数据统一到"报告生成日"为截止**。例如报告日期 2026-06-21，则：
- 财务数据：截至 2025 年报
- 月度数据：截至 2026-05 月产销快报
- 行情数据：截至 2026-06-18 收盘
- 政策数据：截至 2026-06

**不要在整合报告里写"截至撰写时""最新数据"等模糊措辞**——必须明确日期。

### 步骤 6：删除过程性内容

**必须删除的内容**：
- 用户与 AI 的对话
- 试错记录
- 追问的"为什么"和"再想想"
- 反思元分析（除非作为方法论说明）
- 多个版本对比（"v1 这样做，v2 那样做"）

**必须保留的内容**：
- 结论（带条件、带置信度）
- 数据（带来源）
- 方法论（说明用了什么方法）
- 局限性（明确说明不适用情况）

### 步骤 7：保留原始文件

**整合版 ≠ 替换原文件**。原始文件应该保留：
- 作为整合版的"原料"
- 便于追溯
- 便于版本对比

```
整合版：宇通客车_深度分析报告_整合版.md       ← 新文件（推荐用这个）
原文件 1：宇通客车_半年分析.md               ← 保留
原文件 2：宇通客车_护城河分析.md            ← 保留
原文件 3：宇通客车_估值分析_诚实版.md        ← 保留（重要参考）
```

## 典型陷阱

### 陷阱 1：把整合报告做成"目录罗列"

```
❌ 错误：
第一章：来自 A 报告的结论
第二章：来自 B 报告的结论
...

✅ 正确：
第一章：核心结论（综合 A + B + C 的关键判断）
第二章：现状分析（综合 A + B 的数据）
...
```

### 陷阱 2：数据时点不统一

```
❌ 错误：
"2024 年营收 372 亿"（2024 年报披露）
"2026 年 Q1 营收 59 亿"（2026Q1 季报）
"近期销量 3,480 辆"（2026-05 快报）
"上月销量 1,728 辆"（2026-01 快报）

读者无法判断"近期"和"上月"的具体月份。

✅ 正确：
所有数据明确标注月份/季度/年份，且截止到报告日。
```

### 陷阱 3：保留"反思"作为正文内容

```
❌ 错误：
"在反思环节，我承认上版报告存在以下问题..."

✅ 正确：
把反思结论作为"方法论说明"放在第 8 章（独立章节），
不放在正文中。
```

### 陷阱 4：试图把所有内容都包含

**整合报告的目标是"用户 1 小时读完能形成完整认知"**。不是"把 session 内的所有内容都搬过来"。

**实战建议**：
- 整合报告 ≤ 30 KB
- 章节数 ≤ 10
- 关键内容 = 结论 + 数据 + 方法 + 局限性

### 陷阱 5：忽略跨章节的数据冲突

**多个原文件可能有数据冲突**（如同一指标的不同口径）。整合时必须：
1. 识别冲突
2. 选择最新/最权威的数据
3. 在脚注或附录说明差异

## 质量检查清单

整合报告完成前，逐项检查：

- [ ] 数据时点统一（明确日期）
- [ ] 删除所有过程性内容（对话、追问、试错）
- [ ] 删除所有反思作为正文（仅保留作为方法论说明）
- [ ] 章节逻辑递进（从事实到分析到结论）
- [ ] 每章独立可读（不是其他章节的引用堆叠）
- [ ] 数据有来源标注
- [ ] 局限性有显式声明
- [ ] 附录包含核心数据汇总
- [ ] 原始文件保留（不要删除）
- [ ] 文件大小合理（≤ 30 KB）

## 实战案例（宇通客车研究 session）

**Session 包含的产出**：
1. 宇通客车_600066_半年分析_2026-06.md
2. 宇通客车_竞品护城河替代威胁分析.md
3. 宇通客车_自身护城河分析.md（v1）
4. 宇通客车_自身护城河分析_v2.md
5. 宇通客车_估值分析_诚实版.md

**整合策略**：
- 保留 v2 护城河（v1 被 v2 替代）
- 保留诚实版估值（反思版的最终结论）
- 删除所有反思、追问、试错
- 抽取每个产出的关键内容，按主题重排

**整合后结构**：
```
一、执行摘要
二、公司概况
三、半年股价走势
四、股价涨跌动力分析
五、护城河与竞争格局
六、风险与替代威胁
七、估值参考（诚实版）
八、研究方法与数据声明
附录 A：核心财务数据
附录 B：券商一致预期
附录 C：报告价值与使用建议
```

**文件大小**：24 KB（符合 ≤ 30 KB 标准）

**原始文件全部保留**，整合版作为"主报告"。

## 经验教训

1. **整合不是简单拼接**：抽取 + 重排 + 改写需要时间
2. **数据时点统一是基本功**：不统一就失去专业性
3. **反思不放正文**：反思的结论可以放，但"反思过程"不放
4. **保留原文件是底线**：不要因为"整合"就删除其他产出
5. **整合版有用户价值**：用户在一份文件里看到完整研究，比翻 5 份文件更有效
