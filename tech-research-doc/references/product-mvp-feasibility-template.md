# 产品 / idea MVP 可行性调研 — 详细模板

本文件是 6 种报告类型之外的"第 7 种混合型"详细模板：当用户给出一个产品 idea 或创业设想，要求"全链路可行性"时使用。

## 适用信号

- "X 这个产品能不能做"
- "X 类产品 idea 可行性"
- "X + Y + Z 这个组合 idea 怎么样"
- "X 创业机会评估"
- "想做一个 X 类 App/硬件，可行吗"
- "调研一下这类产品的可行性"

**主类型 = 技术调研**（最宽，可承载"能不能做"的根本问题）
**叠加模块 = 方案设计（MVP 路径）+ 竞品对比（产品形态）+ 状态评估（合规/风险）**

## 何时不要套用本模板

- 用户只问"技术能不能实现" → 用纯技术调研（6 类型之一）
- 用户只问"商业模式" → 用纯方案设计
- 用户只问"选 A 还是 B" → 用纯竞品对比
- 用户已经在做、问"现状评估" → 用状态评估
- 用户要的是"做这个产品的实战手册/教程" → 走 `references/handbook-tutorial-class.md`

## 标准 8 章结构

### 1. 执行摘要（必选）— 决策者 5 分钟判断

**关键格式**：

| 维度 | 判断 | 关键证据 | 置信度 |
|------|------|---------|--------|
| 技术可行性 | ✅/⚠️/❌ | 1-2 句证据 | [FACT/HCF] |
| 市场需求 | ✅/⚠️/❌ | 1-2 句证据 | [FACT/HCF] |
| 商业可行性 | ✅/⚠️/❌ | 1-2 句证据 | [FACT/HCF] |
| 合规可行性 | ✅/⚠️/❌ | 1-2 句证据 | [FACT/HCF] |

**末尾必须给"结论一句话" + "MVP 路径推荐（轻/中/重）" + "3 条关键风险"**。

### 2. 问题域拆解（必选）— idea 内部结构的拆解

**目的**：idea 里通常包含多个子功能，先拆清楚每个子功能的真实关系，再决定哪些是主菜、哪些是引流。

**典型结构**：

| 子功能 | 用户真实需求 | 商业本质 | AI 角色 |
|--------|------------|---------|--------|
| 功能 A | "用户用这个想干什么" | 收入/引流 | 必需/可选/不需要 |
| 功能 B | ... | ... | ... |
| 功能 C | ... | ... | ... |

**核心洞察段落**：用一段话点出"哪些是主菜（高频复购）、哪些是引流入口（吸引下载）"，通常会反驳用户的直觉组合。

### 3. AI 技术现状（**必选** — 当 idea 含 AI 模块时，可独立成轴贯穿全文）

> 2024-2026 几乎所有新 idea 都含 AI 元素。本章升级为独立轴。如果用户明确说"要增加 AI 技术"，则必须独立成第三章，贯穿后续每一章。

**小节结构**：

```
3.1 三大 AI 任务拆解
    - 任务 A 的能力边界（声学/视觉/文本/推荐）
    - 任务 B 的能力边界
    - 任务 C 的能力边界
3.2 2024-2026 的真实能力边界
    - SOTA 准确率/性能
    - 营销词 vs 真实能力（如"翻译"vs"情绪分类"）
3.3 自研 vs API 接入的决策矩阵
    - 哪几个模块必须接 API
    - 哪几个值得自研
3.4 国产化方案（如果面向中国市场）
    - 国产 ASR/LLM/多模态定价对比
    - 降本路径
```

### 4. 市场与竞品（必选）

**小节结构**：

```
4.1 主要竞品（按产品形态分组）
    - 纯 App 类 / 硬件+App 类 / 新兴 AI 类
4.2 市场规模
    - 总盘子 / 子赛道 / 渗透率
4.3 付费意愿
    - 行业基准 / 同类订阅价格
4.4 竞品的 AI 技术逆向（如有）
```

### 5. 商品形态与商业模式（必选）

**小节结构**：

```
5.1 形态选择矩阵（App/小程序/硬件/SaaS）
5.2 商业模式拆解（订阅/CPS/自营/广告）
5.3 AI 成本对毛利的影响（最容易被忽略）
```

### 6. MVP 路径与成本（必选）— 6-8 周怎么搭

**小节结构**：

```
6.1 推荐 MVP 方案（含分阶段工时表）
6.2 技术架构（pipeline 图）
6.3 必接 API 清单 + 月度成本估算
6.4 团队配置（最小团队/理想团队）
```

### 7. 风险与红线（必选）— 合规是产品 idea 的高发地雷

**小节结构**：

```
7.1 内容合规（生成式 AI 备案/算法备案/广告法）
7.2 历史处罚案例（避坑指南 — 真实案例比法规条文更有说服力）
7.3 平台审核红线（App Store/微信/抖音/淘宝）
7.4 动物/用户伦理（如适用）
7.5 跨境数据合规（如适用）
```

### 8. 落地推荐（必选）— 给出 2-3 个具体方案

**格式**：每个方案明确给出"投资 / 周期 / 适合人群 / 风险"四要素。最后给决策建议（"如果你有 X 资源，选 Y 方案"）。

```
方案 A：轻量 MVP（¥3-5 万 / 4 周 / 副业试水）
方案 B：中等 MVP（¥8-15 万 / 8 周 / 首次创业）⬅️ 通常推荐
方案 C：重投入（¥50-100 万 / 16 周 / 已有硬件供应链）
```

## 附录（必选）

- **附录 A：AI 能力边界声明**（来自 SKILL.md 主模板）
- **附录 B：数据缺口清单**（明确写"找不到 X"比假装数据齐全更诚实）
- **附录 C：参考文献**（按学术 / 官方 / 行业 / 风险案例分级）

## 协作模式：元问题→收敛→样板→批量（2026-06-09 session 新增）

**问题**：AI 默认行为是"列方向让用户选"，但用户**不能在没有上下文的情况下选方向**——这会导致对话在错误轨道上反复。

**正确流程（4 步）**：

```
Step 1：先回答"做这个决策需要什么输入"（元问题）
  → 不是直接给方向，而是问"我需要知道什么才能给你方向"
  → 把决策的依赖条件列出来（用户身份、约束、用途、成功定义）

Step 2：用户回答输入后，再给方向
  → 收敛到 2-3 个具体方向，每个方向说"为什么、适合谁、风险"

Step 3：做样板（不要批量铺开）
  → 选 1 个方向做 1 个完整样板（按报告类型走完 8 章）
  → 让用户对"颗粒度、判断框架、深度"校准

Step 4：用户确认样板 OK 后，再批量做剩下
  → 避免"N 个方向都做错"重写 N 次
```

**反例（本 session 的真实失败路径）**：

```
Round 1：用户问"AI 怎么听懂宠物叫声"
   → AI：直接给"两层 AI pipeline"技术答案
   → 用户：继续深入讨论

Round 2：AI 列了 5 个方向（A/B/C/D/E 场景）
   → AI："你要做哪个？"
   → 用户："做给做选择的选择依据，做选择需要什么输入？"
   → ⚠️ AI 第一次失败：列方向而不问依赖

Round 3：AI 问"角色/约束/用途/成功定义"4 层
   → 用户回答"投资+执行 / 3 个月内 / 拿到订单"
   → AI：再给 5 场景分析
   → ⚠️ AI 第二次失败：用户已经给输入了，应该给收敛方案

Round 4：用户选"5 个都没看过"
   → AI：进入"5 场景全对比"研究综述模式
   → ⚠️ AI 第三次失败：应该先做 1 个样板

Round 5：用户要 3 方案分别细化（行动清单 / PRD / 尽调）
   → 终于出活
```

**核心教训**：
- **不要列方向让用户盲选**——必须先问"做这个选择需要什么"
- **不要直接铺开 N 个**——先做 1 个样板让用户校准颗粒度
- **样板错了 = 后面全错**——1 个样板的成本远低于 N 个铺开重做

## 标准交付物：三方案并列（M1 阶段专用，2026-06-09 新增）

**触发条件**：当 M1 阶段（30 天验证期）需要落地时，**不要只交付 1 份"研究报告"**——必须并列 3 份：

| 方案 | 形态 | 颗粒度 | 适合谁 | 长度 |
|------|------|--------|--------|------|
| **A 行动清单** | M1 30 天执行节奏表 | 按天/按周/按角色 | 已有团队，今天开干 | 短 |
| **B 产品 PRD** | 微信小程序产品需求文档 | 页面级 + 接口级 | 拿给技术照着做 | 中 |
| **C 商业尽调** | 备忘录 | 财务/风险/竞争 | 拿给投资人/合伙人看 | 长 |

**为什么必须 3 份并列**：
- A 解决"团队/执行"问题——30 天谁做什么
- B 解决"产品/技术"问题——页面长什么样、接口怎么定
- C 解决"商业/决策"问题——市场多大、风险在哪

**只写 1 份 = 答案不完整**。研究报告回答"该不该做"，但 MVP 落地需要"该谁做、怎么做、用什么资源做"。

**共享一致性原则**：
- 3 份文档**共享同一套数据标度**（【已验证/待验证/未验证】 或 [FACT/HCF/REASON]）
- 共享同一套 M1 决策标准（2/4 通过等）
- 共享同一套 12 个月路线图
- 从 A→B→C 是"执行→产品→商业"的递进，**不能有矛盾**

**生成顺序**：A 先做（最快、最窄、最易校准）→ 用户确认颗粒度 → 批量做 B + C。

## "数据真实"的硬约束（2026-06-09 session 用户明确纠偏）

**用户原话**：
> "请按上面的方案框架写好方案，请深度思考，注意细节，方案可行，数据真实，可落地可实现，一定要根据现实方案作为参考。"

**用户第二次明确纠偏**（5 场景分析后）：
> "我必须先把这层矛盾讲清楚......'数据真实' ≠ 'AI 看起来真实'......否则我给你一份'看起来很详细但全是编的'的东西，是对你决策最大的伤害。"

**硬约束（必须执行）**：

1. **每份方案/报告，AI 不得编造以下数据**：
   - 用户量、市场规模、付费率（无公开数据的标"未找到公开数据"）
   - 竞品具体数字（融资额、用户数、定价——只能引用公开来源）
   - 团队配置成本（薪资、市场行情——标"待验证"）
   - 时间估算（工期、里程碑——基于类似项目标"行业基准"）

2. **数据标度三档**（强制使用）：
   - ✅ **【已验证】/ [FACT]**：公开数据/行业标准/官方文档
   - ⚠️ **【待验证】/ [HCF]**：行业经验估算
   - ❌ **【未验证】/ [REASON]**：假设/推断（**必须明示**）

3. **"最后一公里"必须明示**（不替代用户/团队做的真实动作）：
   - 找兽医合作——AI 写不出真实兽医的电话
   - 找 100 个种子用户——AI 给不出真实 KOL 的微信
   - 办 ICP 资质——AI 给不出具体办理流程
   - 谈合作——AI 替代不了面对面/电话沟通

4. **报告末尾必须有"AI 能力边界声明"**——这已在 SKILL.md 主模板里说明，**本模板重复强调**：诚实性是底线。

**反例（本 session 真实踩坑）**：
```
❌ 错误：给"5 个付费用户 = 500 元"做精确财务预测
   → 数字是 AI 编的，没有真实数据支撑

✅ 正确：5 个付费用户 × 29 元 ≈ 500 元 [待验证]
        + 30 天转化率 5% [未验证，基于 ChatGPT Plus 5-8% 类比]
        + 中国宠物主付费习惯远低于美国 [已验证]
        → 让用户自己判断能不能用作决策依据
```

## 数据收集阶段的子 Agent 任务分配

由于本类报告横跨多个垂直领域，**强烈建议**用 `delegate_task` 并行收集数据。下面是经验证的任务切分（**最多 3 个并行任务**，超出需分批 — 这是本 session 验证过的硬限制）：

| 子 Agent | 负责领域 | 工具集 | 输出字数 |
|----------|---------|--------|---------|
| Agent 1 | **AI 技术现状**（自研 vs API、模型选型、能力边界） | web | 2500-3500 |
| Agent 2 | **竞品和市场**（同类产品、定价、用户量、市场规模） | web | 3000-3500 |
| Agent 3 | **合规 + 电商**（法规、广告法、平台规则、变现路径） | web | 3000-4000 |

**每个子 Agent 必须遵守的规则**：
- 回答语言：中文
- 只做事实收集，不做可行性判断或商业建议
- 优先 `web_search`，不 `browser_navigate`
- 找不到的写"未找到公开数据"，**严禁编造**
- 每个数据点附 [来源 URL] [发布日期] [置信度：FACT/HCF/REASON]
- 末尾明确列"数据缺口清单"

**给子 Agent 的 context 模板**：

```
这是为「{idea 名称}」全链路 MVP 可行性调研做数据收集。本任务只覆盖「{垂直领域}」。

要求：
- 回答语言：中文
- 直接做事实收集，不要做可行性判断或建议
- 优先用 web_search，不要用 browser_navigate
- 如果搜不到明确数据，写"未找到公开数据"并标注，不要编造
- 重点抓 2024-2026 的最新数据
```

## 写作执行的关键决策

| 决策点 | 错误做法 | 正确做法 |
|--------|---------|---------|
| **idea 涉及"AI 翻译"类营销词** | 顺着用户措辞写 | 必须独立指出"翻译是营销词，本质是 X 类分类" |
| **idea 涉及监管敏感功能** | 含糊带过"需要注意合规" | 列出 2-3 条具体法规 + 真实处罚案例 + 应对措辞 |
| **idea 包含多个子功能** | 均匀着墨每个 | 在第 2 章明确点出"主菜 vs 引流"，后续章节按权重分配 |
| **idea 涉及商品销售** | 默认走自营电商 | 默认推荐 CPS 分成（淘宝/京东联盟），理由：零库存 |
| **idea 涉及"出海"** | 默认海外为主 | 2026 年中国市场是主战场（如宠物 3,453 亿、22% 渗透率 vs 美国 70%） |

## Review 自检清单（产品/idea MVP 专项）

- [ ] 8 章结构是否完整？执行摘要 + 问题域拆解 + AI 技术现状 + 市场 + 商业模式 + MVP 路径 + 风险 + 落地推荐
- [ ] 结论是否敢于说"不推荐"？可行性报告最常见的失败是结论模棱两可
- [ ] 合规章节是否包含真实处罚案例？仅引法规不够
- [ ] 落地推荐是否给出 2-3 个具体方案（含投入/周期/人群）？
- [ ] AI 能力边界是否明确指出"哪些是营销词、哪些是真实能力"？
- [ ] 数据缺口是否明示？（找不到的写"未找到公开数据"，不编造）
- [ ] **M1 落地是否同时提供 3 份并列交付物**（行动清单 / PRD / 尽调）？
- [ ] **是否先做 1 个样板让用户校准，再批量做剩下**？

## 反例：常见失败模式

**反例 1：把"AI 翻译"当真功能写**
> ❌ "AI 翻译宠物语言是核心功能，技术上 Whisper 即可实现"
> ✅ "AI 翻译宠物语言是营销词；真实能力是 5-13 类情绪分类 + LLM 文本润色；Whisper 对动物音频会幻觉"

**反例 2：合规只在最后一章一句话带过**
> ❌ "需要注意相关法规"
> ✅ "2025-04 已有售价 899 元的'AI 狗语翻译器'被专家拆机；2024-09 硕腾动保用'宠物医生名义'推荐兽药被罚 12 万；建议产品功能描述用'AI 情绪识别'而非'AI 翻译'"

**反例 3：MVP 路径只给一个方案**
> ❌ "建议先做 MVP，6 周上线"
> ✅ "三个方案：轻量（¥3-5 万 / 4 周 / 副业试水）；中等（¥8-15 万 / 8 周 / 首次创业）⬅️ 推荐；重投入（¥50-100 万 / 16 周 / 已有硬件供应链）"

**反例 4：技术现状章节一笔带过**
> ❌ "现在 AI 技术很成熟，可以用 GPT 等大模型"
> ✅ "Whisper 对猫狗叫声 WER 无公开数据，主流'翻译'都是 5-13 类情绪分类 + LLM 文本润色；Qwen3-Omni 是国产替代首选；国产 LLM 单次成本 ¥0.05（Qwen3-Omni）vs ¥0.005（DeepSeek V4 Flash），差 10x"

**反例 5：列方向让用户盲选（本 session 真实失败）**
> ❌ "给你 5 个方向：A 智能硬件 / B 兽医 / C 保险 / D 电商 / E 陪伴，你要做哪个？"
> ✅ "做选择需要先回答 4 个问题：你的角色（投资/执行）？约束（团队/资金/时间）？用途（投别人/自己做）？成功定义？"
> 用户答完后再给方向；用户选 1 个方向后先做样板，再批量。

**反例 6：编造数据做精确财务预测**
> ❌ "M1 末预计 5 个付费用户 = 500 元收入"
> ✅ "5 个付费用户 × 29 元 ≈ 500 元 [待验证]；中国宠物主付费习惯远低于美国 [已验证]；建议 M1 末用真实数据验证"
