# 技术研究报告设计方法

## 适用场景

用户给出一组技术概念或主题，要求产出一份科学技术类研究报告。目标读者是想建立系统认知的工程师、技术决策者或研究人员，看完后能做出判断和决策。

不适用于：API 手册（直接给命令列表）、教程（需要步骤引导）、综述论文（需要引用格式）。

---

## 报告类型识别（首要步骤）

收到需求后，**第一步不是动笔，而是识别报告类型**。不同类型走不同流程、不同结构、不同质量标准。

### 六种报告类型

| 类型 | 读者核心问题 | 典型场景 | 信号词 |
|------|------------|---------|--------|
| **技术调研** | "是什么？能不能用？" | 新技术选型、架构可行性 | "调研、了解、评估、对比一下" |
| **深度分析** | "为什么？底层原理？" | 性能瓶颈根因、技术路线分歧 | "深入分析、底层原理、根因、为什么" |
| **趋势研判** | "往哪走？何时入场？" | 技术栈演进、投资时机 | "趋势、方向、未来、预测、演进" |
| **竞品对比** | "选哪个？A vs B？" | 供应商选型、框架选型 | "对比、选型、竞品、横评、哪个好" |
| **方案设计** | "怎么做？分几步？" | 落地实施、迁移方案 | "方案、落地、实施、迁移、怎么做" |
| **状态评估** | "到什么水平？差距多大？" | 成熟度审计、技术债盘点 | "评估、审计、成熟度、差距、现状" |

### 混合型处理

现实中混合型最常见（如"调研+对比"、"分析+方案"）。处理规则：

1. **识别主类型**（读者最关心的那个问题）→ 决定主流程
2. **识别附加模块**（读者还需要什么）→ 按需叠加
3. **在结构方案中明确标注**："本文主类型=X，附加Y模块"

> 详见 `references/report-type-templates.md`：每种类型的完整结构模板、检查清单、以及常见混合组合。

## AI 能力边界声明

**每次报告必须包含能力边界声明**（放在文末或附录）。这是诚实性的底线。

```
━━━ AI 研究能力边界 ━━━

本报告由 AI 辅助生成，以下能力边界声明适用于全文：

✅ AI 能做的：
  · 搜索整合公开信息（官方文档、学术论文、开源仓库）
  · 结构化分析和逻辑推理
  · 多源数据交叉验证
  · 生成结构化报告框架

❌ AI 不能做的：
  · 实验验证（无法运行 benchmark、无法实测性能）
  · 定量建模（无法建立精确的数学模型）
  · 专家级领域判断（如芯片设计的工艺节点判断）
  · 一手调研（无法采访专家、无法获取内部数据）

📊 报告中的标注体系：
  · [FACT] 已验证事实  · [HCF] 高置信推断
  · [REASON] 合理推测  · [TREND] 趋势外推
  · [N] 官方数据  · [T] 行业经验  · [E] 估算
  详见 references/confidence-annotation-guide.md
━━━━━━━━━━━━━━━━━━━━━
```

> 详见 `references/ai-capability-boundary.md`：完整边界声明模板和按报告类型的差异化声明。

## 核心设计理念

**最好的技术文档不是告诉读者"是什么"，而是让读者能够自己说出"原来是这样"。**

设计的本质是降低认知负担：

- 把最难理解的概念放在后面（读者已有上下文）
- 把最重要的结论放在前面（满足急性子读者）
- 把最深度的对比放在中间（满足认真学习的读者）

## 四阶段写作流程

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
第零阶段：类型识别（最先执行）
  ⚠️ 识别报告类型 → 决定后续流程分支
  ⚠️ 详见上方「报告类型识别」章节

第一阶段：结构设计（不动笔）
  Step 1：明确读者是谁
  Step 2：确定核心问题
  Step 3：设计章节结构（按报告类型选择模板）
  Step 4：标注数据来源
  ⚠️ Step 5：【关键】将结构方案发给用户确认，再动笔

第二阶段：数据收集（不动笔）
  ① 官方 Datasheet → ② 学术论文 → ③ 开源框架 → ④ 行业分析
  ⑤ GitHub 官方仓库（README、AGENTS.md、architecture.md 等）
  📚 学术检索：使用 arxiv skill（而非粗略 web search）
  🤖 AI 领域专用：使用 ai-agent-research skill

第三阶段：写作执行（按类型走差异化分支）
  ┌─ 技术调研 → 「现象 → 根因 → 方案」三层结构
  ├─ 深度分析 → 「问题 → 推理链 → 结论」+ 推理模型构建
  ├─ 趋势研判 → 「现状 → 信号捕捉 → 时间线 → 置信度标注」
  ├─ 竞品对比 → 「维度矩阵 → 逐项评分 → 综合推荐」
  ├─ 方案设计 → 「目标 → 路径 → 风险 → 工期」+ 决策树
  └─ 状态评估 → 「基准线 → 当前水平 → 差距 → 提升路径」

第四阶段：Review（动笔后）
  每个结论有出处吗？
  因果链断在哪？
  读者能据此做决策吗？
  置信度标注是否合规？（不允许高级别标注用于低权限类型）
  AI 能力边界声明是否包含？
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> **⚠️ 关键教训：本阶段和第三阶段之间必须有"用户确认结构"环节。** 先将章节结构方案（目录 + 行文逻辑 + 数据来源标注）发给用户确认，获得通过后再进入写作执行。跳过此步会导致返工（用户说"内容太干瘪"/"结构不对"→ 重写，浪费一次完整迭代）。
>
> 确认示例：
> ```
> 结构方案如下：
> - 第一章：项目概览（回答"是什么"）
> - 第二章：架构设计（回答"怎么组织"）
> - 第三章：核心特性（回答"解决什么问题"）
> - 第四章：应用场景（回答"适合用在哪儿"）
> - 数据来源：GitHub 官方 README + architecture.md
> 请确认是否合适？
> ```

## 第一阶段：结构设计（不动笔）

### Step 1：明确读者是谁

| 问题 | 决定什么 |
|------|---------|
| 有没有相关背景 | 解释的深度 |
| 是判断"能不能用"还是"怎么用" | 内容侧重点 |
| 6个概念还是60个 | 结构复杂度 |

### Step 2：确定核心问题

大多数技术调研文档都在回答**三个本质问题**：

```
① 差距有多大？（性能/功能/生态对比）
② 能不能用？（约束条件、风险因素）
③ 怎么用？（落地路径、工期成本）
```

如果用户的需求没有明确这三个问题，**先问清楚再动笔**。

### Step 3：设计章节结构

先设计结构再动笔。问自己："这个文档需要哪些数据源支撑？哪些有官方出处，哪些是估算？"

**标准结构模板（可裁剪）：**

```
1. 执行摘要          ← 必须有，给决策者的结论摘要
2. 背景与目标        ← 为什么要做这件事
3. 术语表            ← 跨团队阅读时，术语统一很重要
4. 问题域总览        ← 权威 Benchmark 框架 + 芯片对比表
5. 分析正文          ← 「现象 → 根因 → 方案」三层，重复 N 段
6. 综合对比          ← 训练 vs 推理、方案对比
7. 落地路径与成本评估 ← 场景化推荐、人天估算
8. 已知风险与局限    ← 技术风险清单
9. 参考文献           ← 按可信度分级（A.1 官方/A.2 学术/A.3 开源）
```

### Step 4：标注数据来源

在动笔前先标注每个数据点的预计来源：

| 数据类型 | 预计来源 | 标注 |
|---------|---------|------|
| H100 官方参数 | NVIDIA Datasheet | [N] |
| 昇腾 910B 算力 | 昇腾社区/估算 | [T] / [E] |
| 量化精度损失 | MMLU 基准 + 行业经验 | [T] |
| 多卡扩展效率 | 无公开数据 | [E] |

## 第二阶段：数据收集（不动笔）

### 数据优先级（按可信度）

```
① 官方 Datasheet
  └── NVIDIA H100 Datasheet、MLCommons 官方结果
  └── 最可信，适合做结论的支撑

② 学术论文（SOSP、MLSys、ICML）
  └── vLLM SOSP 2023（同行评审）
  └── 适合解释"为什么"

③ 权威开源项目（GitHub 官方）
  └── benchmark_throughput.py、DeepSpeed GitHub
  └── benchmark 脚本是标准实测工具，应告知用户需自行运行

④ 行业分析（招股书、券商研报）
  └── 招股书、IDC 数据
  └── 可信度中等，适合做背景引用

⑤ 个人博客/媒体报道
  └── 仅用于补充上下文，不能做结论支撑
```

### 工具选择

```
文献调研（多个关键词、多个来源）
  → delegate_task + 多个 web search 并行
  → 汇总提取关键数字

核查单个已知 URL
  → browser_navigate（MLCommons 结果页、Benchmark Dashboard）

反复 404/超时
  → 立即降级到 [T]/[E]
  → 生成数据缺口报告，不反复重试
```

### ⚠️ delegate_task 数据收集的已知限制

```
delegate_task 子 agent 可能没有 web/browser 工具权限（取决于环境配置）。
当子 agent 工具不可用时，它们只能依赖训练数据，无法做实时搜索验证。

应对策略：
1. 先在主 agent 中完成关键 URL 的 browser_navigate 核查
2. 将已核查的事实作为 context 传入 delegate_task，让子 agent 基于已验证数据做分析
3. delegate_task 返回的结果必须标注"基于训练数据/基于实时搜索"
4. 核心数据点（[N] 级别）尽量在主 agent 中亲自核查，不依赖子 agent
```

### 硬件类调研的特殊性

当文档涉及芯片性能数据时，NVIDIA 数据**优先查 Datasheet PDF，而非产品营销页**：

| 来源 | 可信度 | 说明 |
|------|--------|------|
| NVIDIA Datasheet PDF | 最高 | SXM5 HBM3 实为 **3.35 TB/s**，非产品页的"3 TB/s" |
| MLCommons 官方结果 | 最高 | 独立于厂商，公信力最强 |
| 昇腾 910B 参数 | 中 | 无官方 Datasheet，行业广泛引用值 ~320 TFLOPS / ~900 GB/s |
| 其他国产芯片参数 | 低-E | 厂商不公开，实为估算或业内传闻 |

> **MLCommons 是国产芯片调研的首选中立数据源**：在 MLPerf 上无提交的芯片，难以做性能横向比较，这本身就是一条结论。

## 第三阶段：写作执行

### 通用技术调研：三明治结构

```markdown
1. 执行摘要          ← 必须有，给决策者的结论摘要
2. 背景与目标        ← 为什么要做这件事
3. 术语表            ← 跨团队阅读时，术语统一很重要
4. 问题域总览        ← 权威 Benchmark 框架 + 芯片对比表
5. 分析正文          ← 「现象 → 根因 → 方案」三层，重复 N 段
6. 综合对比          ← 训练 vs 推理、方案对比
7. 落地路径与成本评估 ← 场景化推荐、人天估算
8. 已知风险与局限    ← 技术风险清单
9. 参考文献           ← 按可信度分级（A.1 官方/A.2 学术/A.3 开源）
```

### 应用场景分析：五步法

当调研对象是平台/框架/系统时，场景分析是重要的章节。步骤如下：

```
Step 1：明确能力边界（What can it do?）
  → 从核心特性反推：该项目擅长什么？
  → 识别不擅长的场景（同样重要）

Step 2：构建场景分类矩阵
  → 横轴：收益程度（高/中/低）
  → 纵轴：适配难度（低/中/高）
  → 填入典型场景

Step 3：场景详析（每个核心场景用以下模板）
  ┌─────────────────────────────────────┐
  │ 【场景名称】                           │
  │ 典型流程：（ASCII 流程图或文字描述）      │
  │ 适用任务：（具体任务列表）               │
  │ 为什么适合 HiClaw：（优势分析）          │
  └─────────────────────────────────────┘

Step 4：场景匹配矩阵
  → 场景 vs 能力（表格形式）
  → 一目了然地告诉读者"这个工具适不适合我"

Step 5：选型建议
  → 决策树形式
  → 帮读者快速判断"我该不该用这个"
```

### GitHub 项目调研的特殊性

对于 GitHub 开源项目，数据收集路径：

```bash
# 1. 基本信息（stars/forks/issues）
curl -s https://api.github.com/repos/{owner}/{repo} | python3 -c "..."

# 2. 目录结构
curl -s https://api.github.com/repos/{owner}/{repo}/contents/ | python3 -c "..."

# 3. README 内容（英文 + 中文）
https://raw.githubusercontent.com/{owner}/{repo}/main/README.md
https://raw.githubusercontent.com/{owner}/{repo}/main/README.zh-CN.md

# 4. 架构文档（通常在 docs/architecture.md）
https://raw.githubusercontent.com/{owner}/{repo}/main/docs/architecture.md

# 5. 开发者导航（AGENTS.md — AI agent 代码指南）
https://raw.githubusercontent.com/{owner}/{repo}/main/AGENTS.md
```

> **教训：搜索不到时，先用 GitHub API 查仓库列表，再读 README.md。** HiClaw 项目早期无官网，搜索引擎无收录，但 GitHub 仓库已有完整信息。

### 三层结构（每个技术点都用这个节奏）

```
1. 现象/问题      ← 读者知道为什么需要这个技术
2. 根因分析        ← 简洁定义 + 原理
3. 解决方案        ← 实践细节（含代码/配置）
```

### 每个结论标注 [N/T/E]

| 标注 | 含义 | 使用场景 |
|------|------|---------|
| `[N]` | 官方发布 | 直接引用，标注来源 URL |
| `[T]` | 行业经验 | 多方印证的估算，注明"非官方" |
| `[E]` | 估算 | 仅用于填补空缺，注明推断依据 |

### 置信度分级（预测和分析类内容必须使用）

[N/T/E] 标注数据来源，置信度标注**判断的可信程度**。两者独立使用、互不替代。

| 级别 | 标注 | 含义 | 允许使用的报告类型 | 要求 |
|------|------|------|------------------|------|
| L1 事实 | `[FACT]` | 公开可验证的数据或事件 | 所有类型 | 必须附来源 URL 或引用 |
| L2 高置信推断 | `[HCF]` | 多个独立信源得出一致结论 | 所有类型 | 必须列出 ≥2 个独立信源 |
| L3 合理推测 | `[REASON]` | 基于已知事实的逻辑推理 | 深度分析、趋势研判 | 必须写明推理链 |
| L4 趋势外推 | `[TREND]` | 基于历史数据的外推 | 仅趋势研判 | 必须标注时间范围和假设 |
| L5 个人判断 | `[JUDGMENT]` | 经验性判断 | 仅在明确标注时 | 必须声明"非定量支撑" |

**使用原则**：
- 技术调研/竞品对比/状态评估：只用 `[FACT]` 和 `[HCF]`
- 深度分析：可到 `[REASON]`
- 趋势研判：可到 `[TREND]`
- 任何类型不得使用 `[JUDGMENT]` 作为核心结论的支撑
- 一段话中标注级别不一致时，**以最低级别为准**

> 详见 `references/confidence-annotation-guide.md`：每种标注的完整示例、常见错误、以及混合使用规则。

### 因果链要完整

```
❌ 断裂（只给结论）：
"昇腾推理慢，因为显存带宽低"
→ 读者不知道带宽低为什么导致慢

✅ 完整：
"昇腾 HBM2 带宽 ~900 GB/s vs H100 HBM3 ~3.35 TB/s（差距 3.7x）
  → 大矩阵乘法（GEMM）受限于数据供给
  → GPU 算力利用率低（昇腾 ~30-40% vs H100 ~80%）
  → 相同算力规格，吞吐差距扩大至 ~3x"
```

### 内容设计原则

- **用场景引入，不用定义引入**。先说"LLM 本质是个文字生成器，你需要 XXX"，而非先给术语定义
- **引入对比和权衡**：适合什么场景 + 不适合什么场景（最难主动写，但最有价值）
- **保留实战经验**：第 X 步容易被忽略、调错了怎么排查、什么时候用它而不是另一个

### 可读性设计

| 要素 | 要求 |
|------|------|
| 架构图/流程图 | 放在文档开头 |
| 核心结论 | 用摘要框 |
| 对比 | 用表格（不是为了堆砌数据） |
| 连续纯文字 | 不超过 3 段 |
| 代码示例 | 之后必须有解释，不能只放代码 |

## 第四阶段：Review（动笔后）

问自己三个问题：

| 问题 | 不过关 | 过关 |
|------|--------|------|
| **每个结论有出处吗？** | "昇腾性能差"（无来源） | "昇腾 910B 吞吐 ~600-800 tok/s [T]，H100 ~2,000-2,500 tok/s [N]" |
| **因果链断在哪？** | "因为带宽低，所以慢"（缺中间推理） | 带宽数字 → GEMM 理论 → 利用率 → 吞吐 |
| **读者能据此做决策吗？** | 只有技术数据，没有落地路径 | 有场景化推荐（客服机器人用方案A，知识库问答用方案B）|

### 自检清单

- [ ] 架构总览图是否在文档开头
- [ ] 每个章节是否有"问题引入"段落
- [ ] 是否包含 trade-off（而非只讲优点）
- [ ] 连续纯文字是否超过 3 段
- [ ] 代码示例之后是否有解释
- [ ] 每个结论标注了 [N/T/E]
- [ ] 因果链是否有断裂

## 教训总结（从实战中固化的方法论）

**教训0：需求分析报告的边界比内容更重要（本session核心教训）**

这是企业文档写作中最常见的致命错误：**在需求分析报告中混入技术选型、GTM战略、供应商选择**。

```markdown
❌ 错误做法：
需求分析报告
├── 第一章：市场分析（GTM战略 ❌）
├── 第二章：需求分析
├── 第三章：技术选型（架构阶段的事 ❌）
└── 第四章：供应商选择（采购阶段的事 ❌）

✅ 正确做法：
需求分析报告
├── WHO（谁来用）→ 干系人 + 用户画像
├── WHAT（需要什么）→ 功能需求清单
├── PRIORITY（先做哪个）→ P0/P1/P2矩阵
└── CAN-DO（能做吗）→ 约束条件识别
    ⚠️ 所有技术选型标注"待架构阶段决策"
```

**判断标准**：文档在"需求分析"阶段是否回答了"怎么做"或"用谁做"的问题？是的 → 越界。

**触发信号**：当用户说"还是没看明白需求调研分析"时，第一反应是检查文档边界，而非增加内容。

---

**教训0b："干瘪"不一定是结构问题，而是内容深度问题（本次session核心教训）**

当用户说"内容太干瘪/不够充分"时，原因可能有三种，处置方式截然不同：

| "干瘪"的真正含义 | 表现 | 应做 |
|----------------|------|------|
| **结构缺失** | 章节之间逻辑断层、核心论点未展开 | 补充结构（如缺"风险假设"章节） |
| **内容深度不足** | 数字没有背景、观点没有案例、因果链断裂 | 充实现有章节，不改结构 |
| **表达机械** | "首先/其次/最后"、套话多、无自然语气 | 去AI味，不重写内容 |

**本次session的真实路径：**

```
用户说"全文略显干瘪，每个章节不够充分"
    ↓
系统错误判断：认为结构有问题（越界内容）
    → 错误行动：重新设计结构 → 触发"哪些内容不必要"
    → 更大错误：用户说"不需要精简"后，理解为"全部扩充"
    → 最终行动：从头重写整个文档（46KB → 110KB）
    ↓
实际上：原文档结构正确（已通过三角评审验证）
用户真正想要的：在原结构基础上，充实每章的论点展开、案例、数字背景
```

**正确做法（教训）：**

```
当用户说"干瘪/不够充分"时：
  Step 1：先判断是"结构问题"还是"深度问题"
    → 有三角评审通过的结论吗？有 → 结构大概率是对的，跳过结构讨论
    → 没有 → 先讨论结构
  Step 2：问用户"具体是哪些章节觉得干瘪？希望补充什么类型的内容（数据/案例/分析）？"
  Step 3：针对具体章节做增量修改，而非重写全文
  Step 4：去AI味在修改完成后做，而不是作为重新组织的理由
```

**去AI味的正确时机和方式：**

```
❌ 错误时机：
"去AI味"作为重写全文的理由 → 实际上是想做内容深化

✅ 正确时机：
内容已经充实后，逐段检查表达自然度

❌ 错误做法：
删除所有"首先/其次/最后" → 破坏逻辑连贯性

✅ 正确做法：
保留必要的逻辑连接词（转折、因果、递进）
删除的是"为您整理"、"值得注意的是"、"换句话说"等机械套话
增加自然对话语气（"这个数字背后的问题是……"、"举一个例子……"）
```

**教训1：先确认"干瘪"的真正含义，再动手**

当用户说"内容太干瘪"，可能意味着：

| 可能的含义 | 应做的 |
|-----------|--------|
| 数据不够权威 | 补充 Datasheet/MLCommons 出处 |
| 缺少分析推理过程 | 补全因果链 |
| 缺少对比参照 | 加 Benchmark 对比表 |
| 缺少场景化落地 | 加场景化推荐 |

**更好的做法**：先问用户"你说的'干瘪'是指？"或"最想看到哪类内容（数据/分析/落地路径）"，再动手。

**教训2：browser 工具不适合系统性文献调研**

- 适合：直接访问已知 URL（MLCommons 结果页、Benchmark Dashboard）
- 不适合：多个关键词、多个来源的系统性调研

正确做法：`delegate_task` + 多个 web search 并行 → 汇总数字。

**教训3：迭代顺序搞反是最大的时间浪费**

```
❌ 直接写完 → 用户说"太干瘪" → 重写（浪费一次完整迭代）

✅ 收到需求 → 先不写 → 问用户三个问题 → 确认后再动笔
✅ 大纲确认 → 用户确认 → 再写全文
```

**教训4：用户提需求的方式直接影响文档质量**

```
❌ 不够好的提法：
"帮我写一份大模型国产化适配的文档，内容详细一点"

✅ 更好的提法（五个要素）：
"帮我写一份大模型国产化适配的决策参考文档
  - 读者：技术决策者
  - 核心问题：昇腾910B能不能用？差距多大？怎么落地？
  - 数据源：需要有 MLCommons 对比、量化精度损失、行业人天估算
  - 结构：先给结论（执行摘要），再展开分析
  - 约束：控制在 5000 字以内"
```

## 三角评审法（Three-Perspective Review）

针对企业级需求文档的标准评审流程，用三个真实角色视角进行结构验证：

```
┌─────────────────────────────────────────────────────────┐
│              三角评审法（三角凳原则）                        │
│                                                         │
│     产品经理视角              CIO视角                     │
│     （业务可行性）           （技术战略）                   │
│          ▲                 ▲                            │
│           \               /                             │
│            \     CIO     /                              │
│             \   视角     /                               │
│              ▼         /                                │
│           市场分析师视角  ←──────────┘                    │
│           （竞争格局）                                     │
└─────────────────────────────────────────────────────────┘
```

**三个视角职责：**

| 视角 | 核心问题 | 评分权重 | 典型 P0 问题 |
|------|---------|---------|------------|
| 产品经理 | 业务价值是否成立？需求是否有优先级？ | 40% | 竞争需求对比缺失、热力图空白 |
| CIO | 技术路径是否清晰？供应商锁定风险？ | 35% | 供应商锁定风险几乎空白、自研vs采购决策依据不足 |
| 市场分析师 | 数据是否可信？市场论证是否充分？ | 25% | 第一手用户调研数据缺失、ROI模型不透明 |

**评审结果合成方法：**
- P0 问题（任一视角提出）→ 必须修复，否则文档不能作为正式决策材料
- P1 问题（两个及以上视角提出）→ 应修复
- P2 问题（单个视角提出）→ 考虑修复

## 多智能体执行模式（企业文档修订）

当需要对一个已有文档进行深度修订时，使用此执行模式：

```
用户授权
    ↓
┌──────────────────────────────────────────┐
│           Orchestrator（协调者）             │
│  职责：分解任务 → 派发 → 汇总 → 判断是否完成    │
└──────────────────────────────────────────┘
    │
    ├── 内容主笔（60-70%工作量，全职）
    │   职责：结构调整、内容重写、过渡模板
    │   优先级：P0
    │
    ├── 数据研究员（20-25%工作量，兼职）
    │   职责：核实数据可信度、补充一手数据
    │   优先级：P1（先于内容主笔执行）
    │
    └── 技术架构评审（10-15%工作量，兼职）
        职责：验证技术可行性、补充技术风险
        优先级：P1（与数据研究员并行）
```

**关键原则：**
- 数据研究员和技术架构评审 **先于** 内容主笔执行
- 两者结论确认后，内容主笔再按确认的策略执行
- 每一轮执行前，必须向用户确认策略（不是等写完再确认）

**多智能体派发示例：**
```
用户说："请按修订方案执行文档重构"
→ 派发 3 个 subagent（角色=leaf）
→ 各自返回工作结果
→ 读取结果文件
→ 确认策略后，再派发内容主笔执行
```

## 教训：结构比内容更重要

企业级需求文档的结构类型：

| 类型 | 特征 | 适用场景 | 不适用场景 |
|------|------|---------|-----------|
| **递进式** | 每章有明确的输入（上一章输出）和输出（下一章输入） | 需求分析报告、决策参考 | 市场概览、术语手册 |
| **并列式** | 各章独立，无前后依赖 | 市场分析、术语表 | 需要论证推理的报告 |

**⚠️ 核心教训：企业AI Agent需求文档必须是递进式结构**

```
正确顺序：
需求调研 → 需求分类建模 → 技术可行性验证 → 需求优先级 → 实施路径

错误顺序（并列式）：
市场分析 + 技术架构 + 需求分析 + ROI + 战略建议
（各章独立，读者无法跟随论证逻辑）
```

**每章必须明确：**
- 输入：本章依赖哪一章的什么结论？
- 输出：本章给下一章提供什么结论？
- 结论支撑：数据来源是什么？[N/T/E]？
