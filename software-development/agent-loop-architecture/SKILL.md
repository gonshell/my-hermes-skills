# Agent Loop Architecture

Design, review, and implement autonomous agent loop systems — AI agents that self-correct through iterative execute→verify→decide cycles.

## When to Use

- User presents a spec or code for an "autoloop", "agent loop", "self-correcting agent", "autonomous execution engine"
- User asks to design a system where AI iterates until a goal is met
- Reviewing existing loop/agent architectures for correctness

## Core Architecture Pattern

```
┌──────────────────────────────────────────────┐
│                LoopEngine                     │
├──────────┬──────────┬──────────┬─────────────┤
│ TaskState│ Context  │ Executor │  Verifier   │
│  (FSM)   │  Manager │ (LLM+   │  (hard +    │
│          │  (L0/L1/ │  tools)  │   semantic) │
│          │   L2)    │          │             │
├──────────┴──────────┼──────────┼─────────────┤
│     LLMClient       │  Tools   │ Controller  │
│  (function calling)  │  (sandbox)│ (decisions) │
└──────────────────────┴──────────┴─────────────┘
```

## 6 Critical Design Decisions

### 1. Multi Tool-Call Per Turn (NOT Single Action Per Round)

**Anti-pattern**: One JSON action per round → read file costs a full round + verification overhead.

**Correct design**: Use native function calling. Let the LLM call multiple tools in one turn (read→write→test). Only verify after the full turn completes.

```python
# Turn = one LLM interaction (may have N tool calls)
# Round = one Turn + Verify + Decide
# The LLM decides when to stop calling tools (no tool_calls in response = done)
```

**Why**: Real coding needs sequential operations. A single "round" should be one complete thought (read → understand → fix → verify), not one atomic action.

### 2. Verification Hierarchy: Hard Every Round, Semantic Only at End

**Anti-pattern**: LLM semantic check every round → expensive, slow, unreliable as a gate.

**Correct design**:
- **Hard check (every round, zero LLM cost)**: pytest, syntax check, file existence. Deterministic, 100% objective.
- **Semantic check (only when hard check passes)**: Independent LLM judge for "test passes but implementation is hack". Optional, not a gate.

```python
# Per round: hard_check() only — fast, free
# On hard_check passed: optional semantic_check() as final review
# If hard check passes but semantic fails → still succeed (MVP policy)
```

**Why**: For code, pytest pass/fail IS the ground truth. LLM judgment adds cost without proportional value. Semantic check is useful for coverage gaps, not as a mandatory gate.

### 3. Three-Tier Context Management

**Anti-pattern**: Dump full history every round → token explosion by round 10.

**Correct design**:
| Tier | Content | Retention | Generation |
|------|---------|-----------|------------|
| L0 Complete | Last N rounds full interaction | 3 rounds | Raw |
| L1 Summary | Earlier rounds: action + result + status | All rounds, ~200 chars each | Auto-generated (not LLM) |
| L2 Global | Goal, constraints, verified items, progress | Always | Structured |

```python
# Key: L1 summaries are NOT generated by LLM (that adds cost + latency)
# They're structured one-liners: "Round 3: [read_file, write_file, run_command] → 2 tests failed"
```

**Why**: The context the agent needs is "what did I try, what happened, what's left" — not the full transcript.

### 4. Error Classification → Differentiated Retry

Not all failures are equal. Classify and respond differently:

| Error Type | Signal | Strategy |
|-----------|--------|----------|
| Recoverable | Feedback says what's wrong | Next round with defect list |
| Directional | Same class of error 2+ rounds | Trigger reflection round |
| Capability | Tool/permission/knowledge gap | Circuit break, mark needs-human |
| Transient | Network timeout, rate limit | Exponential backoff, don't burn a round |

### 5. Reflection Mechanism

When `consecutive_stagnant_rounds >= threshold`:
- Skip execution, run reflection prompt instead
- Agent reviews what went wrong, why, and proposes new strategy
- Reflection output feeds into next execution round
- Reset stagnation counter after reflection

**Key**: Reflection should still allow tool calls — the agent might want to read files to diagnose, then immediately act on the insight.

### 6. Progressive Strictness (Thresholds)

Early rounds: be lenient (pass at confidence ≥ 0.6). Later rounds: tighten (≥ 0.9). This prevents getting stuck on minor details early while ensuring quality at the end.

**Note**: Only relevant when semantic verification is used. With hard-check-only (recommended for MVP), this is unnecessary.

## Common Anti-Patterns to Catch in Reviews

1. **JSON-from-text parsing** instead of native function calling — fragile, 2023-era
2. **Semantic verification as mandatory gate** — expensive, unreliable for code
3. **Single action per round** — wastes round budget on trivial operations
4. **Reflection rounds that don't execute tools** — reflection without action is wasted budget
5. **LLM-generated context summaries** — adds cost; structured one-liners are better
6. **`os.path.join` with string concatenation** for paths — use `os.path.normpath` + boundary check
7. **Shell command whitelist checking first word only** — `python -c "os.system(...)"` bypasses it
8. **No max tool calls per turn** — agent can loop forever calling tools

## Implementation Checklist

When building a loop system:

- [ ] LLM client abstraction with function calling support (not just text completion)
- [ ] Multi tool-call per turn in executor
- [ ] Tool schemas in OpenAI function calling format
- [ ] Hard verification (deterministic) every round
- [ ] Semantic verification (LLM) as optional final check only
- [ ] Three-tier context: L0 complete + L1 structured summary + L2 global state
- [ ] Stagnation detection + reflection trigger
- [ ] Circuit breakers: max rounds, token budget, timeout
- [ ] Tool sandbox: path boundary check, command whitelist with injection detection
- [ ] Structured logging of every round (for debugging and replay)
- [ ] Graceful interrupt handling (Ctrl+C)

## Pitfalls

- **OpenAI-compatible ≠ OpenAI**: Many providers (vLLM, Ollama, Anthropic proxy) claim compatibility but differ in tool calling behavior. Test with your actual provider.
- **pytest exit code 5** = "no tests collected", not "all passed". Don't treat it as success.
- **Tool result truncation**: Long tool outputs (e.g., reading a 10K line file) bloat context. Truncate at ~10K chars per tool result.
- **`shlex.split` for command parsing**: Don't split on spaces — `shlex` handles quotes and escapes correctly.
- **`os.makedirs(parent)` when parent is empty string**: `os.path.dirname("file.py")` returns `""`. Guard against it.
