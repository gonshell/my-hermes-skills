# 功能需求清单分析法

## Overview

对结构化功能需求清单（Excel/文档）进行分层分析：结构化梳理 → COSMIC拆解 → 可行性初筛 → 复杂度评级 → 工时估算。

**核心原则**：
- 不猜测：无法验证的结论只能标注"待验证"
- 不精确：需求阶段工时估算误差±50%，不是精确预测
- 分层处理：可行/中风险/高风险分开处理，不混在一起
- 依据公开：每个评级必须有明确标准，不是主观估计
- 通用方法论：报告模板参数化，数据从Excel读取，不绑定特定系统

## When to Use

- 收到功能需求清单（Excel/文档），需要做分析报告
- 用户要求评估可行性、复杂度、工时
- 需要把分析结果写回Excel（新增列）

**When NOT to use**：
- 已有明确设计文档和约束条件（预算/时间/技术栈）→ 直接做方案设计
- 非结构化需求（口头描述/对话）→ 先结构化再分析

## Core Pattern

### 整体流程

```
输入：功能需求清单（Excel/文档）
     ↓
Step 1：结构化梳理（理解需求是什么）
     ↓
Step 1.5：COSMIC颗粒度拆解（将功能点拆解为可计算FP的颗粒度）
     ↓
Step 2：可行性初筛（判断能不能做）
     ↓ Step 2.1-2.5（见Step 2详细设计）
     ↓
Step 3：复杂度评级（判断有多难）
     ↓
Step 4：工时估算（判断要多少时间）
     ↓
输出：分析报告（Word通用模板）+ 更新的Excel（含新增列）
```

## Step 1：结构化梳理

**目的**：把原始清单转化为可分析的结构，不遗漏信息。

### 新增列定义（4列）

| 新增列名 | 数据来源 | 说明 |
|----------|----------|------|
| 数据实体 | 功能详细描述 | 核心业务对象：人/任务/薪资/合同/项目 |
| 外部依赖 | 功能详细描述 | "对接"/"接入"/"集成"等关键词 |
| 非功能需求 | 功能详细描述 | "实时"/"不可篡改"/"高可用"等词 |
| 业务规则 | 功能详细描述 | 判断条件/计算公式/流转逻辑 |

### 识别规则

**处理合并单元格的模块名称继承**：
- Excel中一级模块（B列）、二级模块（C列）通常是合并单元格，只有第一行有值
- 读取时需维护 `current_m1` / `current_m2` 状态
- 当 row[B] 非空时更新 current_m1，为空时沿用上一行值
- 示例代码：
```python
current_m1 = ''
current_m2 = ''
for row in ws.iter_rows(min_row=3, values_only=True):
    if row[1]: current_m1 = row[1]   # 一级模块合并单元格继承
    if row[2]: current_m2 = row[2]   # 二级模块合并单元格继承
    data.append({'module1': current_m1, 'module2': current_m2, ...})
```

**外部依赖关键词识别**：
- "对接" → 标注外部系统名（人脸识别、银行API）
- "接入" → 标注硬件设备名（门禁、RTK）
- "集成" → 标注第三方服务名
- "上传"/"下载" → 标注文件存储服务
- "实时" → 性能约束
- "不可篡改" → 安全约束
- "五级穿透"/"全链路" → 可用性/一致性约束

## Step 1.5：COSMIC颗粒度拆解

**目的**：将"三级功能点"拆解为"COSMIC功能过程"，计算FP值，作为工时估算的基础。

### COSMIC数据移动定义

| 符号 | 名称 | 定义 | 示例 |
|------|------|------|------|
| **E** | Entry | 从边界进入系统的数据（用户输入） | 表单提交,文件上传 |
| **X** | Exit | 从系统离开到边界的数据（输出） | 列表展示,报表生成,文件下载 |
| **R** | Read | 从存储读取数据 | 查询数据库,读缓存 |
| **W** | Write | 向存储写入数据 | 新增记录,更新状态 |

### 功能过程识别规则

每个"三级功能点"需要拆解为多个"COSMIC功能过程"：

| 功能类型 | 典型功能过程数 | 说明 |
|----------|---------------|------|
| 简单CRUD | 3-5个 | 新建/编辑/删除/列表/详情 |
| 表单+流程 | 5-8个 | 含审批流,每节点一个过程 |
| 外部集成 | 6-10个 | 上传/调用API/获取结果/确认/绑定/归档 |
| 数据报表 | 4-6个 | 筛选/查询/统计/导出/打印 |
| 实时监控 | 5-8个 | 连接/接收/解析/展示/告警/记录 |

### FP值计算

```
单个功能过程 FP = E + X + R + W
总FP = Σ(所有功能过程的FP)
```

### 转换因子（按团队成熟度）

| 团队成熟度 | 1 FP = X 人天 | 说明 |
|------------|---------------|------|
| 初级团队 | 1.0人天 | 学习曲线陡,代码复用低 |
| 中级团队 | 0.5人天 | 有一定经验,复用一般 |
| 高级团队 | 0.25人天 | 熟练,高复用,自动化程度高 |

### COSMIC工时计算

```
COSMIC工时 = Σ(功能过程FP) × 转换因子
```

### 新增列（5列）

| 列名 | 数据类型 | 说明 |
|------|----------|------|
| COSMIC功能过程 | 文本 | 逗号分隔的多个过程名 |
| E/X/R/W数量 | 文本 | 如"10/5/3/2",对应各过程汇总 |
| FP值 | 数值 | 总FP |
| 转换因子 | 数值 | 1 FP = X人天 |
| COSMIC工时 | 数值 | FP × 转换因子 |

### 拆解示例

**原始数据**：序号15, 图纸导入与工程量解析

| COSMIC功能过程 | E | X | R | W | FP |
|----------------|---|---|---|---|----|
| 上传图纸文件 | 2 | 1 | 0 | 1 | 4 |
| 选择图纸格式 | 3 | 1 | 0 | 0 | 4 |
| 调用解析API | 2 | 3 | 0 | 0 | 5 |
| 获取工程量结果 | 0 | 5 | 1 | 0 | 6 |
| 人工确认工程量 | 8 | 2 | 1 | 1 | 12 |
| 绑定到施工区域 | 3 | 1 | 1 | 1 | 6 |
| **合计** | | | | | **37** |

COSMIC工时 = 37 FP × 0.5人天（中级团队）= 18.5人天

## Step 2：可行性初筛

**目的**：识别风险项，判断能否做。不编解决方案，只标注验证方式和缓解措施。

**核心逻辑**：已知依赖查表，未知依赖用矩阵动态评估。两者输出格式统一。

```
功能点涉及外部依赖？
     ↓ 否 → ✅可行
     ↓ 是
依赖类型在已知依赖库（模板表）中？
     ├── 命中 → 查表，取P/I默认值/风险等级/验证方式
     └── 未命中 → 进入P×I矩阵动态评估
```

### 2.1 已知依赖库（可扩展模板）

依赖库是**模板参考**，不是写死的17种。使用者可在Excel中维护自己的依赖库。

**模板格式**（CSV示例，使用者可扩充）：

**查表匹配规则**：
- 逐行匹配"依赖模式关键词"（支持正则或模糊包含）
- 匹配到则使用该行的P/I/风险等级/验证方式
- 未匹配到任何行 → 进入P×I矩阵评估

### 2.2 P×I矩阵（用于未知风险）

**可能性P（量化版）**：

| 等级 | 分值 | 标准 | 判断依据 |
|------|------|------|----------|
| 高 | 3 | 0-20%成功率 | 无先例，技术不成熟，供应商<2家 |
| 中 | 2 | 20-60%成功率 | 有类似案例，供应商2-3家，有定制需求 |
| 低 | 1 | 60-100%成功率 | 已有成熟方案，供应商>3家 |

**影响I（量化版）**：

| 等级 | 分值 | 标准 | 判断依据 |
|------|------|------|----------|
| 高 | 3 | 延期>2周 或 成本+30% | 影响核心业务/合规 |
| 中 | 2 | 延期1-2周 或 成本+10-30% | 影响部分功能 |
| 低 | 1 | 延期<1周 或 成本<10% | 影响边缘功能 |

**风险分值 = P × I**：

| 分值 | 风险等级 | 颜色 | 是否进工时池 |
|------|----------|------|-------------|
| 9 | 🔴极高风险 | 红色 | 否 |
| 6 | 🟠高风险 | 橙色 | 否 |
| 4 | 🟡中风险 | 黄色 | 是（×1.2） |
| 3 | 🟡中风险 | 黄色 | 是（×1.2） |
| 2 | 🟢低风险 | 浅绿 | 是（×1.0） |
| 1 | ✅可忽略 | 绿色 | 是（×1.0） |

### 2.3 风险调整系数

| 风险等级 | 调整系数 | 是否进入工时池 |
|----------|----------|---------------|
| 🔴极高 | 不进入，单独估算 | 否 |
| 🟠高 | 不进入，单独估算 | 否 |
| 🟡中 | ×1.2 | 是 |
| 🟢低 | ×1.0 | 是 |
| ✅可忽略 | ×1.0 | 是 |

### 2.4 新增列（8列，Step 2）

| 列 | 内容 | 数据类型 | 说明 |
|----|------|----------|------|
| P | 风险类别 | 枚举 | 技术/资源/外部依赖/合规/规模 |
| Q | 外部依赖类型 | 文本 | 具体依赖名称，匹配依赖库模板 |
| R | 依赖库命中 | 自动 | 命中/未命中 |
| S | 概率P值 | 自动/手动 | 查表默认值或矩阵评估 |
| T | 影响I值 | 自动/手动 | 查表默认值或矩阵评估 |
| U | 风险分值P×I | 自动 | =S×T |
| V | 风险等级 | 自动 | 查表或矩阵计算 |
| W | 验证方式 | 自动/手动 | 查表或手动填写 |

**P值/I值来源优先级**：
1. 依赖库命中 → 自动填入模板中的P/I默认值，可手动覆盖
2. 依赖库未命中 → 手动判断P值和I值，再计算分值

**绝对禁止**：不编造"应对方案"，只写"验证方式"和"缓解措施"（仅高风险项填缓解措施）。

### 2.5 执行流程（Step 2）

```
Step 2.1：识别外部依赖
     ↓ 从H列（外部依赖）中提取关键词
Step 2.2：匹配已知依赖库
     ├── 命中 → 填入P/I/风险等级/验证方式，R列标"命中"
     └── 未命中 → R列标"未命中"，进入Step 2.3
     ↓
Step 2.3：P×I矩阵评估（仅针对未命中依赖）
     → 手动判断P值和I值 → 计算分值 → 确定风险等级
     ↓
Step 2.4：统一输出
     → 高风险项（🔴极高/🟠高）单独记录，不进入工时池
     → 🟡中风险 ×1.2，🟢低/✅可忽略 ×1.0
     ↓
Step 2.5（可选）：使用者可扩充依赖库模板
     → 在Excel中维护自己的依赖库表（模板外置）
```

## Step 3：复杂度评级

**目的**：量化每个功能点的技术难度,用于工时分配。

### 三维度评估法

```
综合复杂度 = max(数据维度, 界面维度, 外部集成维度)
```

### 数据维度（看数据孤岛程度）

| 等级 | 标准 | 示例 |
|------|------|------|
| L1 | 单表CRUD,无关联 | 基础参数配置 |
| L2 | 2-4表关联,有简单业务规则 | 员工档案（含关联表） |
| L3 | 5-10表关联,跨模块数据汇聚 | 薪资核算（依赖考勤+任务+规则） |
| L4 | 10+表关联,或外部系统集成+事务要求 | 银行API对接,多方数据一致性 |

### 界面维度（看交互复杂度）

| 等级 | 标准 | 示例 |
|------|------|------|
| L1 | 列表页+详情页,纯展示 | 数据看板 |
| L2 | 表单+状态流转 | 合同备案 |
| L3 | 多步向导/可视化编辑/地图集成/实时刷新 | 图纸任务绑定 |
| L4 | 实时协同/硬件交互/大文件处理/离线支持 | 视频监控+人脸考勤 |

### 外部集成维度（看对外依赖）

| 等级 | 标准 | 示例 |
|------|------|------|
| L1 | 无外部依赖,纯内部逻辑 | 内部统计报表 |
| L2 | RESTful API对接（1-2个） | 地图服务 |
| L3 | 厂商SDK对接（人脸/推送/存储） | 华为云EFG SDK |
| L4 | 硬件协议对接 或 金融监管类集成 | 门禁一体机,银行卡API |

### 加分项（满足任一则复杂度+0.5级）

| 加分项 | 说明 |
|--------|------|
| 数据量>10万条 | 大数据量处理 |
| 需要离线支持 | 移动端离线功能 |
| 需要多语言/多时区 | 国际化支持 |

### 复杂度系数

| 等级 | 系数 |
|------|------|
| L1 | ×0.8 |
| L2 | ×1.0 |
| L3 | ×1.3 |
| L4 | ×1.6 |

### 新增列（4列）

| 列名 | 数据类型 | 说明 |
|------|----------|------|
| 数据维度 | 枚举：L1/L2/L3/L4 | 根据数据表数量和关联判断 |
| 界面维度 | 枚举：L1/L2/L3/L4 | 根据交互复杂度判断 |
| 外部集成维度 | 枚举：L1/L2/L3/L4 | 根据外部依赖类型判断 |
| 综合复杂度等级 | 自动计算 | =max(数据维度,界面维度,外部集成维度) |

## Step 4：工时估算

**目的**：给出可执行的工时预算,含误差范围和假设条件。

### 两阶段工时估算

**第一阶段：COSMIC工时**（基于COSMIC功能点拆解）

```
COSMIC工时 = Σ(功能过程FP) × 转换因子
```

**第二阶段：调整后工时**（考虑复杂度和风险）

```
调整后工时 = COSMIC工时 × 复杂度系数 × 可行性系数

可行性系数：
  ✅可忽略/🟢低风险：×1.0
  🟡中风险：×1.2
  🟠高风险/🔴极高风险：单独估算,不进入此公式
```

### 工时组成（按角色）

| 角色 | 估算方式 | 说明 |
|------|----------|------|
| 后端 | COSMIC工时×0.6 | 数据处理+业务逻辑 |
| 前端 | COSMIC工时×0.25 | 界面交互 |
| 测试 | COSMIC工时×0.15 | 功能测试+回归 |

### 误差控制

| 阶段 | 误差范围 | 说明 |
|------|----------|------|
| 需求阶段（当前） | ±50% | 信息不完整,只能做粗估 |
| 设计阶段 | ±30% | 详细设计后更清晰 |
| 开发阶段 | ±15% | 迭代调整 |

### 工时估算必须附带

1. 总工时（含🟢低风险和🟡中风险,不含🟠高风险和🔴极高风险）
2. 高风险项单独估算（不计入总工时）
3. 误差范围：±50%
4. 假设条件：团队成熟度级别、技术栈、部署方式

### 新增列（6列）

| 列名 | 数据类型 | 说明 |
|------|----------|------|
| 后端人天 | 数值 | COSMIC工时×0.6 |
| 前端人天 | 数值 | COSMIC工时×0.25 |
| 测试人天 | 数值 | COSMIC工时×0.15 |
| 总人天 | 数值 | 后端+前端+测试 |
| 工时误差范围 | 文本 | ±50% |
| 假设条件 | 文本 | 团队成熟度/技术栈/部署方式 |

## Common Mistakes

| 错误做法 | 正确做法 |
|----------|----------|
| 写"应对方案"列,编造未验证的解决方案 | 只写"验证方式"和"缓解措施" |
| 给精确工时数字,不附误差范围 | 工时必须注明"±50%需求阶段误差" |
| 把高风险项计入总工时 | 高风险项单独估算,不进入标准池 |
| 跳过Step 1直接分析 | 先结构化梳理,再分析 |
| 复杂度评级无标准依据 | 三个维度各有L1-L4明确定义和示例 |
| 直接用人天估算,跳过COSMIC拆解 | 先拆解到COSMIC功能过程,计算FP,再转人天 |
| 报告模板绑定特定系统 | 模板通用参数化,数据从Excel读取 |
| 用固定列索引读取合并单元格数据 | 合并单元格的模块名列（如B列/C列）只有第一行有值，后续行需继承上文，需在读取时维护current_m1/current_m2状态 |
| 依赖库关键词与实际描述不匹配 | "五级穿透""全链路"等建筑行业特有词是业务约束而非外部依赖，应归入"非功能需求"而非"外部依赖" |
