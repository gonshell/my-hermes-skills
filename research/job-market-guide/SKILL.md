# 岗位/职场全景 + 求职行动指南

## 适用场景

用户给出一个求职方向（如"Agent 工程师"、"Harness Engineer"、"推理优化"），要求产出该领域的求职全景——有哪些公司在招、什么 Title、薪资多少、怎么准备、怎么投递。

**读者核心问题**："我想做 X 方向，该投什么公司、什么 Title、薪资多少、怎么准备？"

**不适用**：
- 纯技术调研（"Harness Engineering 是什么原理" → `tech-research-doc`）
- 面经深度汇总（单个公司的面试题库 → 独立 task）
- 在职人员内部晋升规划

## 与 tech-research-doc 的边界

| | tech-research-doc | job-market-guide |
|---|---|---|
| 读者目的 | 做判断（能不能用/选哪个） | 做行动（投什么/怎么投） |
| 核心产出 | 分析报告 | 行动指南 |
| 数据来源 | Datasheet / 论文 / Benchmark | JD / 招聘平台 / 公司官网 / 面经 |
| 独特章节 | 术语表 / 架构图 / 性能对比 | 决策卡 / 投递策略 / 中文关键词表 |
| 交付物 | 飞书研究报告 | 飞书求职指南 |

## 触发条件

用户说以下任一时触发：

- "X 方向有哪些公司在招？"
- "X（Harness Engineer / Agent 工程师 / 推理优化）在国内怎么找工作？"
- "帮我做一份 X 领域的求职全景"
- "X 方向的薪资、岗位、投递路径"
- 英文 Title（Harness / Context / Loop / Agent Engineer）+ "国内" / "招聘" / "JD"
- "X 的简历该怎么写？搜什么关键词？"

## 流程导航

```
收到需求
  │
  ├→ 阶段 0：需求澄清（不动笔）
  │   明确求职方向 → 目标公司范围 → 时间窗口
  │   ⚠️ 不问过多问题，3 个以内
  │
  ├→ 阶段 1：多路径搜索（不动笔）
  │   ① 搜中文策展长文 → ② 公司官网 → ③ LinkedIn / Boss 直聘 → ④ 行业报告
  │   ⚠️ 搜不到换路径，不重试同一关键词 3 次以上
  │
  ├→ 阶段 2：结构设计 + 一次写入
  │   8 章骨架 → 一次性写入本地 markdown → 推送飞书
  │   ⚠️ 不分段交付，一次写完
  │
  └→ 阶段 3：自评 + 扩展（可选）
      写入自评局限 → 用户追加时增量扩展（JD 细节 / 面经 / 所有权转移）
```

## 阶段 0：需求澄清

最多问用户 3 个问题：

1. **求职方向**：用户想做什么？（如"Agent 工程"、"推理优化"、"Context Engineering"）
2. **目标公司范围**：大厂 / 创业公司 / 全部？（默认：头部 5-8 家）
3. **时间窗口**：当前市场 / 某个时间段？（默认：当前）

如果用户给的方向是英文 Title（如 Harness Engineer），先判断国内是否有同岗位中文 Title 存在——**不要假设用户知道中文对应 Title 是什么**，这本身就是文档要回答的问题。

---

## 阶段 1：多路径搜索

### 搜索铁律

| 规则 | 原因 |
|------|------|
| ❌ 不搜英文 Title | 搜索引擎会"翻译"走样，中文 JD 搜索极差 |
| ✅ 搜中文工作内容关键词 | "推理优化工程师" / "Agent 平台" / "RAG 工程师" |
| ✅ 优先找策展长文 | 掘金/知乎/CSDN 上有人已扒过数百份 JD |
| ✅ 补公司官网 + LinkedIn | 一手信息，但反爬严重时降级 |
| ✅ 搜不到就停 | 同一关键词搜 3 轮无结果 → 换路径 |

### 搜索路径优先级

```
① 中文关键词搜策展长文（掘金/知乎/CSDN）
   → 高赞长文通常已做了 JD 拆解 + 薪资统计
   → 命中后深挖全文（web_url_read）
   → 示例搜索词："大模型算法工程师 招聘 知乎 拆解 JD"

② 公司官网招聘页
   → jobs.bytedance.com / alibaba.com / deepseek.com / zhipuai.cn
   → 获取真正在招的 Title + 投递入口

③ LinkedIn / Boss 直聘
   → 补充薪资区间 + 岗位详情
   → Boss 直聘反爬严重 → 降级为参考

④ 行业报告 / 媒体
   → 薪资趋势、招聘趋势
   → 置信度较低，标注 [T] / [E]
```

### 搜索失败处理

| 失败场景 | 正确做法 |
|---------|---------|
| 英文关键词零结果 | 立即换中文关键词 |
| 中文关键词结果噪音多 | 直接搜"XX 招聘 知乎 拆解"找策展内容 |
| Boss 直聘反爬 | 降级，用策展长文中的薪资数据 |
| 公司官网招聘页找不到 | 用 LinkedIn + 脉脉补充 |
| 同一关键词 3 轮无结果 | 停止，换完全不同路径 |

## 阶段 2：文档结构 + 写入

### 章节优先级

| 章节 | 必选/可选 | 裁剪条件 |
|------|----------|---------|
| 第 1 章 术语演进 | **必选** | 不可裁剪——这是 skill 的核心价值（gap 分析） |
| 第 2 章 公司逐家拆解 | **必选** | 不可裁剪——求职者需要具体投递目标 |
| 第 3 章 岗位全景 | **必选** | 不可裁剪——回答"有哪些方向可以选" |
| 第 4 章 薪资全景 | 必选 | 数据缺失时降级为"薪资参考区间 [T/E]"，不删章 |
| 第 5 章 技能点 | 必选 | 数据缺失时降级为"JD 高频关键词" |
| 第 6 章 投递策略 | **必选** | 不可裁剪——行动指南的核心 |
| 第 7 章 来源链 | 必选 | 极简版可合并到附录 |
| 第 8 章 决策卡 | **必选** | 不可裁剪——整篇文档的收束 |
| 附录 自评 | 必选 | 不可裁剪——诚实性底线 |

**最小可行结构**（信息不足时）：第 1 章 + 第 2 章（至少 3 家公司）+ 第 8 章 + 附录。其他章可压缩但不删。

### 文档骨架（8 章 + 附录）

```markdown
# {领域} 求职全景指南

> 范围：{公司列表}
> 时点：{YYYY-MM}
> 读者：求职/转型决策参考
> 一句话结论：{核心判断，如"国内没有独立英文 Title JD，但同岗位以中文 Title 大量存在"}

---

## 第 1 章 · 术语演进：新 Title 为什么找不到 JD

| Title | 起源 | 命名者/定义 | 当前状态 | 是否有独立 JD |
|---|---|---|---|---|

一句话区分：
| 概念 | 管什么 | 一句话 |
|---|---|---|

> 替代假设：如果这些 Title 真的出现 JD，会是什么样？
> 倾向：{时间判断 + 理由}
```

**为什么必写"替代假设"**：读者最关心的不是"现在有没有"，而是"将来会不会有、什么时候有"——这直接影响求职策略选择。

```markdown
---

## 第 2 章 · 公司逐家拆解（投递清单）

### {公司名}

- **公司主页**：{url}
- **在招真 Title**：{中文 Title 列表，从官网/LinkedIn 核实}
- **做什么**：{业务方向，一句话}
- **薪资区间**：{月薪范围}·{薪数}（来源：{策展长文/官网/LinkedIn}）
- **投递路径**：{首选渠道 + 备选}
- **特点**：{一句话差异化，如"极小团队招极精的人"}

### 横向对比表

| 公司 | 团队规模 | 核心业务方向 | 投递难度 | 内推优先级 |
|---|---|---|---|---|
```

**为什么逐家拆解**：求职者需要的是"这家公司具体怎么投"，不是泛泛的行业趋势。每家公司的 Title、薪资、投递路径都不同。

```markdown
---

## 第 3 章 · 全景：N 大类热门岗位

> 数据源：{策展长文标题 + 日期 + URL}

| # | 类别 | 典型 Title | 占比趋势 | 头部公司 |
|---|---|---|---|---|

### N.1 {类别名}
- **技能**：{技能列表}
- **学历**：{要求}
- **薪资**：{区间}
- **代表 Title**：{具体公司 + Title}
```

```markdown
---

## 第 4 章 · 薪资全景

> 数据源：{策展长文 + 时间点}
> 下表为月薪区间（大部分岗位标 14-16 薪，实际年包 = 月薪 × 薪数 + 奖金/股票）

### 三档分布

| 档位 | 范围 | 典型 Title | 典型薪数 |
|---|---|---|---|

### 城市差异
### 影响薪资的关键变量
```

```markdown
---

## 第 5 章 · 技能点大揭秘

> JD 出现频率最高的技能关键词（按热度排序）

### 必会技能（100% JD 要求）
### 高频加分项（>50% JD）
### 编程语言优先级
### 领域知识
```

```markdown
---

## 第 6 章 · 投递策略

### 搜什么关键词能找到 JD

> 不要搜英文 Title。搜中文工作内容关键词。

| 你想做的方向 | 搜的关键词（中文） |
|---|---|

### 投递渠道优先级
### 简历优化建议
```

```markdown
---

## 第 7 章 · 来源链

### 主要来源
1. {来源名}（{平台}, {日期}）
   - URL：{url}
   - 置信度：{Tier / 说明}
   - 内容：{一句话概括}

### 二手来源
### 关于"标题滞后于现实"的补充来源
```

```markdown
---

## 第 8 章 · 一页纸决策卡

### 你想做什么 → 应该投什么

| 你的目标 | 中文 Title 关键词 | 目标公司 | 渠道 |
|---|---|---|---|

### 快速判断标准
### 你的下一步（5 步行动清单）
```

```markdown
---

## 附录 · 自评

- ✅ 覆盖 N 家公司 + M 类岗位
- ⚠️ 局限 1：{具体说明}
- ⚠️ 局限 2：{具体说明}
```

### 写入方式

1. 一次性写入本地 markdown（`/tmp/cchistory-media/{topic}.md`）
2. 推送飞书：`lark-cli docs +create --api-version v2 --markdown @{filepath}`
3. 报告 doc_id 和 URL 给用户

**不分段交付**：一次写完 8 章 + 附录，避免碎片感。

---

## 阶段 3：自评 + 扩展

### 自评（必写）

文档末尾的"附录·自评"必须包含：
- ✅ 覆盖范围（多少家公司、多少类岗位）
- ⚠️ 数据时效（薪资基于哪个时间点的抽样）
- ⚠️ 搜索盲区（哪些平台没拿到数据、为什么）
- ⚠️ 概念演进（术语仍在变化，3-6 个月内可能有新变化）

### 扩展（用户追加时执行）

当用户说"找全具体岗位信息 + 面经"时：
1. 读取已有文档（`lark-cli docs +fetch`）
2. 针对性搜索 JD 细节 + 面经
3. 增量追加到已有文档（不重写全文）
4. 可选：转移文档所有权（用 `lark-doc-transfer-ownership` skill）

## 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 搜索失败处理 | 换路径，不重试 3 次以上 | 搜索引擎中文 JD 搜索极差，重试浪费时间 |
| 文档结构 | 固定 8 章，可裁剪 | 复用已验证结构，不重新发明 |
| 交付方式 | 一次性写入 + 推飞书 | 避免分段交付的碎片感 |
| 第 1 章"替代假设" | 必写 | 读者最关心"这些 Title 会不会很快出现"——决策关键 |
| 来源链 | 必写，含置信度 | 求职指南的数据可信度直接影响决策 |
| 自评局限 | 必写 | 主动披露比被发现更好 |
| 扩展阶段 | 增量追加，不重写全文 | Day 2 追加 JD + 面经的模式已验证 |
| 不搜英文 Title | 铁律 | 搜索引擎"翻译"走样，零有效结果 |

## 踩过的坑

| 坑 | 后果 | 正确做法 |
|---|---|---|
| 搜英文 Title（如 "Harness Engineer"） | 搜索引擎走样，零有效结果 | 搜中文工作内容关键词 |
| 依赖 Boss 直聘一手数据 | 反爬封锁，拿不到 | 降级为参考，用策展长文替代 |
| 薪资不标时效 | 数据可能已过期 | 标注抽样时间点 + 说明可能变化 |
| 跳过"替代假设"章节 | 读者最关心的问题没回答 | 必写，给出时间判断 |
| 没有写自评局限 | 被读者发现数据盲区时被动 | 主动披露，写在附录 |
| 同一搜索词反复重试 3+ 轮 | 浪费工具调用，无新结果 | 3 轮无果立即换路径 |
| 分段交付文档 | 用户体验碎片化 | 一次性写完再推飞书 |

## 自检清单

- [ ] 每个薪资数字有来源标注
- [ ] 公司的"在招 Title"是真实在招（官网/LinkedIn 核实），不是推测
- [ ] 投递路径有可执行的 URL 或平台名
- [ ] 中文关键词表覆盖了所有提到的方向
- [ ] 决策卡能回答"我想做 X → 该怎么做"
- [ ] 自评局限写明了数据时效和搜索盲区
- [ ] 第 1 章的"替代假设"有时间判断和理由
- [ ] 来源链标注了置信度
- [ ] 文档一次性写入（不分段交付）
- [ ] 推送飞书后报告了 doc_id 和 URL
