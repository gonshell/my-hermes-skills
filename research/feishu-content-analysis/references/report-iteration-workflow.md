# 报告迭代精简工作流

> **来源**：2026-06-14 v6 → v9 迭代 session
> **对应 SKILL.md 教训**：12（工程脚手架排除）/ 13（融合≠删除）/ 14（冗余审查方法）
> **用途**：当用户在已有报告上说"反思冗余" / "X 融合到 Y" / "删 X" / "反思目录结构"时加载

---

## 迭代命令分类（用户实际用过的 4 类）

| 用户原话 | 类型 | 标准动作 |
|---|---|---|
| "反思一下全文有没有冗余，和废话？" | **冗余审查** | 教训 14：表格诊断 → 等确认 → 执行 |
| "新趋势章节融合到后面的章节吧" | **融合** | 教训 13：映射表先行 → 不直接删 |
| "删除 附录A.3 小节。" | **删除** | 直接删，不解释 |
| "反思一下目录结构，是否可以优化一下" | **结构重构** | 出 2-3 方向 + 利弊 → 等确认 |

**判别关键**：融合 = 内容保留 + 位置移动；删除 = 内容移除。两者不能混。

## 冗余审查的诊断表模板（教训 14 落地工具）

当用户说"反思一下全文有没有冗余"时，产出此表：

```
| § | 内容 | 评判 | 字数（估） | 真冗余 | 真冗余占比 |
|---|---|---|---|---|---|
| <章节> | <1 句话描述> | ✅紧凑 / ⚠️有冗余 | <N 字> | <M 字> | <%> |
```

然后给"砍掉 / 简化 / 保留"清单 + 预估长度变化。

**关键**：真冗余占比要诚实——不是所有内容都是冗余，紧凑的章节标 ✅ 不动。

## 高频冗余模式（v6→v7 实测 8 个点）

按出现频率排序：

1. **TL;DR 与末尾总结重叠** → 删末尾，TL;DR 作唯一入口
2. **同一事件在事件表 + 维度章节都讲** → 维度只保留独有内容
3. **边界声明分散两处**（§0 + §9.4）→ 合并全文唯一
4. **"风险"字段同义反复** → 删字段或合并到"产出"
5. **附录"观察"行**（装饰性解读）→ 删
6. **附录"可能根因"**（未验证猜测）→ 改 1 行事实
7. **方法论限制的工程细节**（Twitter 未做、arXiv 未做）→ 删，这是作业承认
8. **漏斗 ASCII 图 / 评估硬指标列表**（自然语言伪装方法论）→ 改 1 段

## 工程脚手架排除清单（教训 12 落地工具）

以下内容**不属于交付物**，写入报告前自检：

| 内容类型 | 例子 | 处理 |
|---|---|---|
| 版本对照 | "v1 → v2 关键修正" | ❌ 删，放 session log |
| 工具元数据 | revision_id / document_id / log_id | ❌ 删 |
| 系统级问题 | HERMES HOME 偏移 / cron 输出缺失 | ❌ 删，独立排查 |
| 作业承认 | "Twitter 采集未做" | ❌ 删 |
| 自创评级 | "高/中/低置信"（用户没要求时） | ❌ 删 |
| 数据来源限制 | "工具挂了 / 视频未直读" | ✅ 写，压成 1-2 段 |
| 补救工作流 | "怎么验证的" | ✅ 写，简短 |

**原则**：读者关心"你能信到什么程度 + 你分析了什么"，不关心"你怎么做的 + 你作业哪里没做完"。

## 覆盖写入命令（迭代用）

每次迭代用同一飞书 docx 覆盖：

```bash
# 本地写 markdown → 覆盖写入 → 验证 outline → 删本地文件
lark-cli docs +update --api-version v2 \
  --doc <token> --doc-format markdown \
  --command overwrite --content @<file>.md

lark-cli docs +fetch --api-version v2 \
  --doc <token> --doc-format markdown \
  --scope outline | python3 -c "..."  # 验证 revision + outline

rm <file>.md
```

**关键**：`--command overwrite` 是全文覆盖，不是 append。revision_id 每次自增。
