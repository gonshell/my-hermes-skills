# Cron 内容管道的结构性陷阱

## 适用场景

当数据源是 **cron 定时任务** 写入飞书的内容（早报/晚报/视频榜单/资讯聚合）时，分析者必须先识别"这批数据本身的质量边界"，否则会基于不完整数据做不完整分析。

---

## 陷阱 1：多档 cron 写同一份文档 = "赢者通吃"

**典型架构**：

```
4 个 cron 任务（早上 06:00 / 晚上 20:00 / B 站 21:00 / 全站 22:00）
        ↓ 都写
同一份飞书 docx（doc_token = X）
```

**结果**：

- 最后写入的档**完全覆盖**前几档（如果用 `+update` 而不是 `+append`）
- `~/.hermes/cron/output/` 下的本地 XML 文件名带时间戳，但**不是每档都有保留**——5-30 之前的本地文件被清理
- 飞书文档本身**不存历史 revision 的内容摘要**，只能看当前 revision

**典型发现**（本 session 2026-06-14 验证）：

```bash
lark-cli docs +fetch --api-version v2 --doc "HhyMdusqdoVcW9xLyd2c2Yc2nnf" --format json | \
  python3 -c "import json,sys; d=json.load(sys.stdin)['data']['document']; print('revision:', d['revision_id']); print('blocks:', len(d['content']))"
# 输出：revision: 25, blocks: 5067
# 看到的内容：仅 2026-06-13 晚间档
```

**对策**：

1. 报告开头**强制段**：明说"当前文档只承载晚间档输出"
2. 给出本地 XML 路径 + 找到的最早一档文件日期，让用户知道**数据范围真实边界**
3. 不假装"看到了 3 档内容"——只分析**实际可见**的那一档

---

## 陷阱 2：URL 是 Bing 视频搜索页，不是 YouTube 直链

**典型数据**（本 session 验证）：

```html
<a href="https://www.bing.com/videos/search?q=Top 3 AI platform updates from Google I/O 2026">Top 3 AI platform updates from Google I/O 2026</a>
```

**问题**：

- 这是 **Bing 视频搜索结果页**，不是 YouTube 视频页
- 播放量/频道/上传时间**来自 Bing 摘要**，**无法用 YouTube Data API 独立核验**
- 如果 Bing 摘要过时（缓存），数据会与 YouTube 实际不符
- 标题可能与 YouTube 实际标题有出入（搜索 query ≠ 视频 title）

**对策**：

1. 在报告中明说"URL 是 Bing 搜索聚合，播放量未独立核验"
2. 跨源验证时**用 Bing 搜索的 query 关键词去 YouTube 站内搜**，找到真实视频后再用 YouTube Data API 验证播放量
3. 永远不要把 Bing 摘要的播放量直接当事实引用

---

## 陷阱 3：本地 cron 输出文件清理策略不一致

**典型发现**：

```
~/.hermes/cron/output/
├── bilibili-ai_2026-05-30_113354.xml
├── youtube-ai_2026-05-30_200221.xml
└── youtube-ai_2026-05-30_210047.xml
```

**问题**：

- 5-30 之前的文件被清理
- 5-30 当天有 3 档
- 5-31 之后没有任何本地文件
- 6-13 跑过的 cron 任务没存本地文件

**可能原因**：

- cron 任务的 prompt 里有写文件步骤，但执行时跳过了（顺序问题、路径问题）
- 文件被自动清理（TTL / 大小限制）
- 输出被 direct pipe 到 `+update` 飞书 API，没有落盘

**对策**：

1. **不要假设本地有原始数据**——读飞书文档是唯一可靠路径
2. 如果要做"多日趋势分析"，要先**确认 cron 任务的写入策略**（是 append 还是 overwrite？本地是否落盘？）
3. 必要时**改造 cron 任务**（不在分析任务范围内，留作"建议"）

---

## 陷阱 4：YouTube 端 vs B 站端选题重合度极低

**典型发现**（2026-06-13 晚间档 + 当日新发）：

| 平台 | 主要话题 |
|------|---------|
| YouTube 长视频 | Anthropic 估值、Prometheus 融资、Google I/O、AI 暗黑面、Siri 兼容 |
| B 站 AI 早报矩阵 | Kimi K2.7、MiniMax M3、小米 MiMo、网信办监管、阿里云 Token 优惠 |

**重合点**：几乎只有"Google I/O 2026 平台更新"出现在 YouTube + 间接影响 B 站（"Gemini Omni"被 B 站早报转述）。

**这是"两个独立传播圈层"**，不是"信息不全"。

**对策**：

1. 报告里点出"中国视角 vs 国际视角"的分裂
2. 跨平台主题（如"AI 监管"）用并置对比展开
3. 不要用"信息不全"作为分析局限——这是结构性的视角分布

---

## 陷阱 5：时间锚是"X 天前"不是 ISO 时间戳

**典型数据**：

```html
<a href="...">Top 3 AI platform updates from Google I/O 2026</a> ｜上传：4 天前
```

**问题**：

- 没有 ISO 时间戳，无法做精确时间序列分析
- "4 天前"取决于**文档生成时刻**，会随读文档时间漂移
- 不同档对同一视频的时间描述可能不一致（早上看"2 天前"，晚上看"1 天前"）

**对策**：

1. 报告里给出"读文档时间"作为唯一时间锚
2. 不要做"过去 24 小时"这种精确时间窗分析
3. 趋势分析时用"近期 / 上一档 / 上上档"的相对时序而非绝对时间

---

## 陷阱 6：cron 任务本身的"覆盖盲区"必须在报告中指出

**这是用户偏好**："三个 cron 任务的覆盖盲区"这种"对数据源本身的元评估"用户会**主动追问**。

**标准元评估清单**：

- [ ] 视频 URL 是否直链（不是搜索页）？
- [ ] 播放量/数据是否可独立核验？
- [ ] 飞书文档是否保留历史档位（还是"赢者通吃"）？
- [ ] 本地是否落盘原始 XML？
- [ ] 跨平台话题重合度如何？
- [ ] 时间锚精度（"X 天前" vs ISO 时间戳）？
- [ ] cron 任务的 prompt 是否明确（避免 prompt 漂移）？

→ 把这个清单**单独一节**写在报告末尾，比塞在分析正文里更清晰。

---

## 改造建议（不在本任务范围内）

如果用户后续要求**改造 cron 任务**本身（而不是分析其输出），建议的最小改动：

1. **每档用单独的飞书文档**（不要再共用）
2. **本地落盘 + 日期**：`~/.hermes/cron/output/<job-name>_<ISO-date>_<time>.xml`
3. **URL 直链化**：Bing 搜索结果需要二次抓取 YouTube 直链
4. **时间戳用 ISO 8601**（替换"X 天前"）
5. **数据范围元信息**：每个文档头部加 `<meta source="..." range="..." />`

但这些是"数据质量改进"，与"读+分析"任务是两件事——**不要把改造建议强加到分析报告里**，可作为末尾"数据源改进建议"短段落给出。
