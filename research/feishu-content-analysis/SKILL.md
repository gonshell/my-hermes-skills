# 飞书内容综合分析

## 适用场景

用户给一个飞书文档（docx）token 或链接，要求"分析""综合解读""对里面提到的内容做交叉验证"。典型来源：

- 早报/晚报文档（cron 推送的 AI/科技/财经早报）
- 选题清单文档（upstream 收集的素材库）
- 调研摘要文档（同事/agent 整理的事实清单）

**核心工作流**：读飞书 → 识别关键主题 → 跨源验证 → 五维分析 → 输出报告。

**不适用**：纯创作（无输入文档）、纯事实查询（直接 web_search 即可）、要求"按文档原样展示"（那是读+展示，不是分析）。

---

## 核心设计原则

### 原则 1：选对结构 — 五维框架 vs 事件流+判断流

飞书内容分析有**两种合法的报告结构**，**用户偏好 / 任务复杂度决定选哪个**：

#### 结构 A：五维框架（适合"快速日报 / 概览"）

- 维度：趋势 / 技术 / 热点 / 商业 / 产品
- 优点：模板化、上手快、覆盖全
- 缺点：**5 个维度不正交**（趋势↔热点、技术↔产品 重叠），同一事实在不同维度被重复描述 = 信息密度低；强断言容易藏身于维度叙述中不易被审视

#### 结构 B：事件流 + 判断流（适合"深度分析 / 审查 / 评估"）

- 结构：① 数据范围声明 → ② 已验证事件清单（按主体分组）→ ③ 跨事件的可成立判断（每条带 推演链 + 反方证据 + 概率评级）→ ④ 自批判（未证实 / 已知未知 / 方法论限制）→ ⑤ 下一步动作（按 ROI 排序）
- 优点：信息流单一不重复、强断言必带推演链、必有反方观点
- 缺点：需要主动做事件去重 + 跨事件关联

#### 结构 C：三层金字塔（适合"决策导向读者 / 混合受众 / 用户要 TL;DR"）

- 结构：① **TL;DR（5 秒）**——3 句话浓缩全文核心 → ② **核心事件表（30 秒）**——5-7 行表，每行 1 事件 + 一句话 + 视频ID → ③ **§0 边界声明（全文唯一）** → ④ **§1 调研建议**（决策导向读者先看） → ⑤ **§2-§7 维度展开**（含新趋势内联，不单列） → ⑥ **§8 视频 ID 索引** → ⑦ **附录 A 元数据**（视频完整索引 + 延迟矩阵）
- 优点：**读者自选阅读深度**（5 秒 / 30 秒 / 5 分钟三档）；事件不重复（在维度下展开而非在维度间切片）；新趋势内联到维度章节避免独立成节导致重复
- 缺点：维度感比 A 弱（维度降级为章节标题不再是思维框架）；附录可能膨胀

**触发条件**（用户实际信号 → 推荐结构）：

| 用户原话 / 信号 | 推荐结构 | 原因 |
|---|---|---|
| "做简要分析报告" / "综合分析" | **A** | 用户要"快" |
| "深度思考" / "审查" / "评估" / "信息目录结构是否可以" | **B** | 用户要"严" |
| 报告面向运营者、消费者 | **A** | 受众需要模板化输出 |
| 报告面向政策制定者、投资者、决策层 | **B** | 受众需要可证伪的判断 + 边界声明 |
| 文档条目 ≤ 15 条 | **A** | 五维足以覆盖 |
| 文档条目 > 15 条 | **B** | 五维下每个主题会被稀释，事件流更清晰 |
| "方案 C" / "三层金字塔" / "TL;DR" / "便于阅读和思考" | **C** | 用户要分层入口 |
| 报告面向混合受众（既要快速结论又要深度分析） | **C** | 单一结构满足不了矛盾需求 |
| 报告已存在 + 用户要"融合 X 到 Y" / "删附录 X" | **C**（迭代） | 在现有结构上做外科手术，不重写 |

**如果走 A，参考 `templates/five-dimension-analysis.md`**。
**如果走 B，参考 `templates/event-flow-judgment.md`**（v2 新增）。

#### 不论 A/B 都要遵守的硬约束

- **每个主题 / 事件在全文只能出现 1-2 次**，不能五维各写一遍
- 强断言必须配"反方证据"或"未证实"标注
- 概率评级用"高 / 中 / 低置信"代替"最大风险 / 最被低估"等情绪化措辞

### 原则 2：每个高权重主题必须有"一手信源验证"

文档里出现的"声明/数字/事件"必须能在**文档之外**找到原始信源（官方博客、原始视频链接、新闻报道、官方账号推文）。**没有验证的声明必须标注"未独立核验"**，不能让文档转述变成"事实链"。

验证优先级：

```
① 官方一手信源    → 公司博客、官方公告、官方账号推文
② 主流财经/科技媒体 → Bloomberg / TechCrunch / The Verge / 36Kr / 财新
③ 学术/行业分析   → arXiv / IDC / 行业研报
④ 个人博客/论坛    → 仅作补充，不作主结论
```

### 原则 3：明确"数据范围限制"，不假装完整

飞书文档是滚动追加的，**多档 cron 写同一份文档 = 赢者通吃，最后写入的覆盖之前的**。本地 `~/.hermes/cron/output/` 也不一定存有原始 XML。

**动笔前必须先验证**：
- 文档当前承载的是哪一档（早上/晚上/B 站）？
- 早间档和晚间档是否都被保留？
- URL 是 YouTube 直链，还是 Bing 搜索页/聚合链接？

把限制写进报告开头，**不要假装你看到了完整数据**。

### 原则 4：识别"市场共识盲点"比罗列共识更值钱

五维分析结束后，单独留一节给"三个反差"或"被低估的信号"——**指出现有叙事里被忽略的关联**。这是用户偏好（明确要求"识别市场共识盲点"）。

## 飞书 docx 读取的两条路径

⚠️ 这是高频陷阱，详见 `references/lark-docx-read.md`。简要：

| 路径 | 适用场景 | 失败模式 |
|------|---------|---------|
| `feishu_doc_read` 工具 | 仅在飞书**评论**上下文中可用 | 普通对话上下文 → "not in a Feishu comment context" 错误 |
| `lark-cli docs +fetch --api-version v2 --doc <token>` | **通用路径**，所有上下文都可用 | `--api-version v1` 已弃用；v2 单次一屏 |

`--format pretty` 直接出可读 HTML（带 `<title>` `<h1>` `<li>`），是首选格式；`--format json` 适合程序化处理（`--jq` 过滤）。

## 教训（已固化的方法论）

**教训 0（最高优先级，2026-06-14 新增）：用户说"反思后整理方案让我确认"=必须先反思出方案，不直接重写**

用户反馈"反思后整理方案让我确认"代表的是**完全不同的工**作模式——不是"重写报告"，是"先复盘 → 出修正方案 → 等待确认 → 才执行"。

**触发信号（任一即触发）**：
- "反思一下" / "请反思" / "认真反思"
- "整理出方案" / "出方案让我确认"
- "总结什么、趋势预测、热门热点是什么"
- "别把乱七八糟的东西写进来"（代表前一次输出已经过度塞内容）

**标准动作**：
1. 列出"我之前做错了什么"（对前一次输出的具体批评）
2. 列出"根因"（为什么错——结构问题？信息密度？方向？）
3. 列出"重新出方案"（新的结构 / 砍掉什么 / 保留什么 / 目标长度）
4. 列出"需要用户确认的决策点"（如有结构歧义）
5. **停下等用户确认**——不要直接动手重写

**反例（2026-06-14 session v1→v2 错误路径）**：用户说"反思 → 整理方案 → 让我确认"→ 我直接写 v2 报告并写入了飞书。**正确做法**：先出方案，等用户确认方案，再执行。

---

**教训 1：先验证文档承载范围，再开始分析**

飞书 cron 推送文档"赢者通吃"是常见架构。`+update` 写入会覆盖前一档。如果你看到的文档只承载 1 档，**必须明说**——不要假装看到了 3 档。

验证命令：
```bash
# 看文档当前 revision 和 block 数
lark-cli docs +fetch --api-version v2 --doc <token> --format json | \
  python3 -c "import json,sys; d=json.load(sys.stdin)['data']['document']; print('revision:', d['revision_id']); print('blocks:', len(d['content']))"
```

**教训 2：URL 是搜索页不是直链时，反查 YouTube 真实 ID**

cron 任务用 Bing 视频搜索抓数据时，所有 URL 都是 `bing.com/videos/search?q=...`。**反查手段**（10/12 命中率）：
```
site:youtube.com "<Bing 摘要标题>" <频道名>
```
详见 `references/source-verification.md` §1。反查到 ID 后用 YouTube Data API 查播放量、评论数、真实标题。**反查本身能修正很多"以为是 X 实际是 Y"的错误**（本 session 中 Fable 案例就是典型：以为是某独立模型，实际是 Anthropic Fable 5）。

**教训 3：跨平台重合度低是覆盖盲区，不是数据问题**

B 站 AI 早报和 YouTube AI 长视频的选题重合度极低（中国 vs 国际视角分裂）。**这不是"信息不全面"，是"两个独立传播圈层"**。报告中必须点出。

**教训 4：结构 A 还是结构 B？用户偏好决定**

5 维框架的硬伤：**不正交 + 重复描述**。用户说"深度思考"时**默认走结构 B（事件流 + 判断流）**，模板见 `templates/event-flow-judgment.md`。详细触发条件见原则 1。

**教训 5：识别"反方/盲点"是用户强偏好，但必须真反方，不是自我延伸**

用户明确要求"末尾必给反方观点/盲区/无法证伪的假设"。**反方观点的本质是"找你结论的反例"**——不是"找你结论的延伸"。

**反方观点的真假辨别**：
- ✅ 真反方："Anthropic 估值反超 ≠ 收入反超"（v1 用此；v2 一手数据 $47B ARR 证伪）
- ❌ 假反方："Anthropic 965B 估值不等于 965B 收入"（这是修辞包装的批评，不是反方证据）

**教训 6：cron 静默失败不要被 last_status=ok 误导**

`last_status=ok` + `last_delivery_error=null` **不代表本地文件存在**。本 session 发现 3 个任务 6-14 全部 ok，但 `cron/output/` 最新文件 5-30。**任务诊断不要相信 last_status**——读文件验证。

**教训 7：HERMES HOME 路径可能双层嵌套**

`~/.hermes/cron/output/` 实际可能是 `~/.hermes/home/.hermes/cron/output/`。**用前先 `ls` 验证**。详细见 `references/source-verification.md` §4。

**教训 8（2026-06-14 修订）："深度思考"≠默认走 v2 重写**

**修订原因**：原教训"用户说深度思考 = 默认走结构 B（事件流+判断流）+ v2 重写协议"被 2026-06-14 session 错误贯彻——用户实际偏好是"先反思 → 出方案 → 让我确认"，不是"自动重写"。

**新规则**：
- 用户说"深度思考" / "深度评估" / "审查" → **先做自我批判 + 列出 3-5 个具体问题 + 提修正方案 + 等用户确认**（不要自动重写）
- 用户说"反思后整理方案让我确认" / "整理出方案" → **必须先出方案，绝不直接动手**（教训 0 触发）
- 用户说"按做好的方案修改调整" / "直接写" / "可以，请继续" → 才进入执行

**v2 重写协议的合法触发**（仅以下情况自动执行）：
- 用户明确说"重写" / "出 v2" / "请重做"
- 用户在反思环节明确批准了 v2 方案
- 用户在对话历史中已经隐含授权（如"按你建议的改"）

**结构 A 还是 B 的最终选择权在用户**（不是 agent 擅自决定）。详见教训 0。

**教训 9（2026-06-14 新增）："反思 → 出方案 → 让我确认"被误解为自动重写**（教训 0 的强化）

**真实失败路径（2026-06-14 v1→v2 session）**：
- 用户第一轮反馈："信息目录结构是否可以，信息组织方式是否可以"——看似要重构
- agent 把"批评"误读为"重做指令"，写了 v2（16.4 KB 6 节结构 + 5 一手公告 + 9 YouTube ID + 自批判 + 元数据发现）
- 用户第二轮反弹："写的是什么？分析什么了，请回顾我让你分析什么...别把乱七八糟的东西写进来，你反思后整理出方案让我确认"
- 关键信号："**别把乱七八糟的东西写进来**"——明确表达前一次输出过度塞内容
- 关键信号："**反思后整理方案让我确认**"——明确要求"先出方案，等批准，再执行"

**正确动作链**：
1. 用户反馈"目录结构是否可以"→ 不是"重写"指令，是"评估你交付的质量"
2. 第一反应应该是"列出我交付了什么 + 它哪里不对 + 应该怎么改"，**等用户批准改法**
3. v2 的"5 个一手公告 + 9 个 YouTube ID 修正"作为**纠错**价值高，但包装在 6 节 + 4 子节自批判 + 元数据发现 = **结构膨胀**，违背"简要分析报告"原需求
4. 用户原话"简要"是硬约束，不是软建议

**教训 15（2026-06-14 新增）：多源 cron 写多份 doc 时——必须真读所有 doc，不读就别下"综合分析"结论**

**真实失败路径（v1-v9 session）**：
- 4 个 cron 任务（早间档 / 晚间档 / B 站 AI / B 站全站）**各写各的 doc**（不是赢者通吃的同一份）
- v1-v9 "综合分析"只读了 1 个 doc（晚间档 `HhyMdu...`），**早间档 / B 站 AI / B 站全站 3 个 doc 都没读**
- 报告却叫"综合分析 · 2026-06-13"——**名实不符**
- 用户明确反馈："三个定时任务的飞书文档是否全部都被阅读并分析过了？"
- 真实情况：**只读过 1/4**

**判别规则（按 cron 任务数 → 数据源数）**：
1. 拿到飞书 doc token 时，**第一步**确认这个 doc 是 1 个 cron 任务写的，还是 N 个 cron 任务写的
   - 1 个 doc 1 个任务（如 4 个任务各写 1 份）→ **要分析"综合"，必须读全部 N 个 doc**
   - N 个任务 1 个 doc（赢者通吃）→ 读 1 个就够，但要标注"只看到最后写入的 N 档"
   - 不知道？**去查每个 cron 任务的 prompt 看 doc token**（见 `hermes-cron-job` skill 的"查找 doc token 标准动作"）
2. 标题用"综合" / "全局" / "全景" 时，**数据源覆盖率必须 ≥ 80%**，否则降级标题为"X 档分析"
3. 数据源覆盖率 < 100% 时，**报告开头§0 显式列**：哪些源已读 / 哪些源未读 / 未读的影响（如"B 站视角完全缺失，结论只覆盖 YouTube 国际视角"）
4. **不读就别下"综合"结论**——用户最反感"名实不符"

**`hermes-cron-job` 引用**：本任务（分析 cron 推送的飞书 doc）必须先知道 N 个 cron 任务写 N 个 doc token，参见 `hermes-cron-job` 的"查找 doc token 标准动作"和 4 个视频任务的真实 doc token 表。

**教训 17（2026-06-14 新增）："反思缺洞见"的 4 类根因 + 5 条洞见模板**

**触发信号**：用户说"反思一下" + "**缺少深入的、本质的洞见**" 或类似短语。在已有报告上做"质量提升"型反思时使用。

**v9 报告缺洞见的 4 类根因（实测）**：
1. **缺数据**：基于 1/3 数据源做"综合分析"= 名实不符（v9 漏读 2/3 doc）
2. **缺反方**：强判断（"商业基本面反超"）没有反方证据
3. **缺量化**：用"强""重大""首次"等情绪化词，无数据支撑（72 条视频没统计）
4. **评论当事件**：把 Fireship 5:09 短片（评论）跟 Bloomberg 47 分钟（事件）并列分析 = 错位

**5 条洞见模板（按类型分）**：
1. **结构性差异**——3 doc 主题互补/分裂的分析（"B 站偏教学 / 早间档偏实测 / 晚间档偏事件"）
2. **重复信号识别**——跨 doc 同一主题出现频次（"橘鸦Juya 4 次 = 矩阵化"）
3. **传播链 / 生产链**——二阶传播（事件 → 评论）、单信源多次转述
4. **流量结构性差异**——播放量加权（"B 站 1.07 亿是晚间档 899 万的 11.9 倍"）
5. **跨语种叙事差异**——同一主题两套叙事（"AI 替代：中文=焦虑+反向解读，英文=必然+资本集中"）

**反方观点的"真假辨别"**（与教训 5 强化版）：
- ✅ 真反方：基于独立证据的反驳（"ARR $47B 是自报数据，无 SEC 审计"）
- ❌ 假反方：修辞包装的批评（"估值反超 ≠ 收入反超"——这是修辞不是证据）
- ❌ 假反方：自我延伸（"未来可能逆转"——这是推演不是反方）
- **真反方必须有"独立来源 / 可量化指标 / 公开数据"**

**判别规则**：
- 用户说"反思 / 缺洞见" → 列根因 + 给洞见模板 + 等用户选哪些进报告
- 每条强判断必须有 ≥1 条真反方
- 强判断的反方不一定要进报告正文，但**至少在脑子里过一遍**再写

**v9→v10 实测**：补 5 条洞见（每节开篇 1 段）+ 4 条反方（穿插在强判断处），长度 14.7 → 17.6 KB，**用户反馈"判断标准：§2-§6 每节开篇【洞见】和结尾反方是不是你说的深入洞见"**——证明这是有效补强方向。

**教训 11（2026-06-14 新增）：所有外部抓取工具同时失败时——记录失败模式 + 退回已有上下文，**不要假装抓到了**

**教训 12（2026-06-14 新增）：工程脚手架不属于交付物——revision_id / v1→v2 对照 / 方法论限制段都是 agent 自跟踪，不是读者内容**

- ❌ 版本对照（v1→v2 修正了什么）→ 这是 agent 自跟踪，放 session log
- ❌ 工具调用的元数据（revision_id / document_id / log_id）→ 这是工程脚手架
- ❌ "已知未知"段（系统级问题如 HERMES HOME 偏移）→ 与报告主题无关，独立排查
- ❌ 方法论限制的工程细节（"Twitter 采集未做"、"arXiv 未做"）→ 这是作业承认，不是分析

- ✅ 补救工作流（怎么验证的）→ 写，但简短

**判别原则**：读者关心"**你能信到什么程度 + 你分析了什么**"，**不关心你怎么做的 + 你作业哪里没做完**。

**教训 13（2026-06-14 新增）："融合" ≠ "删除"——用户说"X 章节融合到 Y" 时先做映射表**

**真实操作（v8 session）**：用户说"新趋势章节融合到后面的章节吧，简化，去冗余"。

**正确做法**：
1. **先做映射表**（不动手）：原章节每个子节 → 融合到哪个维度章节
2. **识别冲突点**：原 §2.2 的"阿里云 Token 营销"已在 §7.3 出现 → 这种直接删，不重复带过去

**教训 14（2026-06-14 新增）：全文冗余审查的具体方法——表格 × 字数估算 × 原文证据**

**教训 19（2026-06-14 新增）：洞见的"标签"≠ 洞见——"现象 + 反常识 + 落地" 3-part 写作模板 + "人"层要求**

**触发信号**：用户在"已加洞见"的报告上反馈：
- "洞见是有了" → 接受了**结构**，但接下来 1 句 "**每条洞见想说什么？**" → 揭示**洞见的标签是空壳**
- "如何写好一条洞见" → 用户在要**方法论**
- "你现在的结构 AI 味太足了" → 用户在要**人的声音**
- "是 B 站是补课中心，还是程序员在对抗被替代的焦虑，还是其他的什么？" → 用户在要**"人"层**（谁在做、为什么做、这说明了什么关于他们的事）

**v10 / v11 失败模式（实测）**：
- v10 的"洞见 1" = "**Fable 5 = 二阶传播链，不是产品发布**" → 这是**标签**，不是洞见。读者不知道"二阶传播链"意味着什么
- v11 改用 3 段：现象 / 可能的"人"解释(3 个) / 未验证 → 看起来"全面"，但**用户说"AI 味太足"**——3 段平衡 = 套路
- **根因**：把"洞见"当成**信息分类器**（结构性差异 / 重复信号 / 传播链 / 流量差异 / 跨语种），没当成**判断 + 解释 + 影响**的话

**正确的洞见写作模板：3 段合一**

一条洞见必须**同时**回答 3 个问题（不是 3 段，是 3 件事塞进 1 段或 1 句话）：

1. **现象（看到了什么）** — 1 个数据点 + 1 个对比
2. **反常识（这意味着什么）** — 跟主流叙事的偏差在哪
3. **落地（这对谁意味着什么不同）** — 投资人 / 从业者 / 政策制定者 / 研究者各看到什么

**示例**（"现象 + 反常识 + 落地" 3-in-1）：
> Fable 5 的 47 分钟 Bloomberg 深剖和 5 分钟 Fireship 短片播放量是 92.8 万 vs 56.9 万，但**短片靠长篇喂养**——这说明"主动受限"在 KOL 圈层是被 Anthropic 主动制造的话题，不是中立的产品发布。**对投资人**：这波传播是公关不是产品；**对从业者**：KOL 评 = 事件重要性是错的逻辑，二阶传播层要分开看；**对治理者**：技术侧"主动受限" = 双向操作（技术 + KOL 配合），不能只看技术。

**v10 对比（标签）**：
> 洞见 1：Fable 5 = 二阶传播链，不是产品发布。
> 现象：Bloomberg 47 分钟 92.8 万 + Fireship 5 分钟 56.9 万 = 二阶传播。

**差在哪**：
- 标签式洞见 = **只给"是什么"**（"这是二阶传播"），不说"为什么是"和"所以呢"
- 完整洞见 = **给"是什么 + 为什么 + 所以呢"**（"是 X，因为 Y，所以 Z"）

**"人"层要求（用户隐藏的强偏好）**

每条洞见必须有**"人"层**——不是"系统在做 X"，而是"**人**在做 X / 这关于**人**意味着什么"。

**触发信号**：用户原话 "B 站是补课中心，还是程序员在对抗被替代的焦虑，还是其他的什么？" —— 用户在要 agent 替**人**找动机，**不是**替数据找模式。

**5 个"人"层视角**（写洞见时自检）：
1. **谁在做** — 是 UP 主 / 厂商 / KOL / 工程师 / 学生 / 投资人？
2. **为什么做** — 流量 / 教育责任 / 商业利益 / 个人焦虑 / 品牌建设？
3. **这关于他们说明什么** — 行业趋势 / 个人处境 / 集体心理 / 商业模式？
4. **谁是目标受众** — 学生 / 老板 / 同行 / 政府 / 媒体？
5. **受众看完会做什么** — 订阅 / 投资 / 焦虑 / 模仿 / 监管？

**v11 的"可能的'人'解释"3 段式 = 错**——不该用 3 段，应该用 1 段把"人"的视角**嵌入**到洞见里：

**v11 错**（3 段，AI 味）：
> 现象：B 站 22 条总播放 1.07 亿
> 可能的"人"解释（3 个）：技术：LLM 把"Prompt+Agent"变成新基础能力 / 产品：UP 主搬运英文课程做本地化 / 商业：教学流量是 B 站护城河
> 未验证：我没看 B 站弹幕/评论区

**正确**（1 段嵌入"人"）：
> B 站 22 条总播放 1.07 亿，是晚间档英文 YouTube 的 11.9 倍。**这是中文互联网在用 B 站当"再教育基础设施"**——LLM 把"Prompt + Agent"变成新基础能力，吴恩达 / PyTorch / 秋葉aaaki 这些 UP 主在做的事情，本质上是把"中文版的英文课程"变成"中文工作流"，因为国内没法直接读英文教材。**对从业者**：B 站是新的"再教育中心"，不是娱乐平台；**对投资人**：教学流量是中文 AI 应用的护城河；**未验证**：UP 主商业化路径没数据。

**反 AI 味的 4 条具体规则**（洞见写作时自检）：
1. **不要 3 段式结构**（现象 / 解释 3 个 / 未验证）——这是 LLM 模板
2. **不要 5 个"人"解释并列**——选 1-2 个最关键的，**嵌入**到主句里
3. **不要"这意味着 / 标志着 / 由此可见"**——直接说事
4. **不要平衡的"挑战 + 机遇"二段式**——选一边说，说清"谁要做什么"

**判别规则（用户的实际信号 → 选什么写作模板）**：

| 用户原话 | 模板 | 长度 |
|---|---|---|
| "如何写好一条洞见" | 现象 + 反常识 + 落地 3-in-1 | 1 段 3-5 句 |
| "每条洞见想说什么" | 先回答 1 句（标签 + 解释），再展开 | 1 句 + 1 段 |
| "AI 味太足了" | 反 3 段式 + 嵌"人"层 | 1 段到底，不分小节 |
| "是 X 还是 Y 还是 Z" | 选 1 个最可能的 + 1 个反方 + 标注未验证 | 1 段 3 句 |
| "B 站是补课中心还是焦虑" | 直接给 1 个最可能的 + 嵌入"人"动机 | 1 段到底 |

**v11 → v12 实测迭代方向**：
- v10：5 条洞见（标签式）→ 用户问"洞见 1 想说什么"
- v11：5 条洞见（3 段式）→ 用户说"AI 味太足"
- **v12 应该**：5 条洞见，每条**1 段 3-5 句**，**嵌入"人"动机**到主句里，**不分子节**

**教训 21（2026-06-14 新增）：洞见模板 v2——从 3 段式到"反常+涌现+倾向+替代假设"**

**触发信号**：用户在已改好的 3 段式洞见上追问"洞见小节的信息描述方式有什么好建议"——不是要重写，是要**迭代写作模板本身**。

**v1（教训 17）→ v2（教训 19）→ v3（本教训）的演进**：
- v1：标签式（"洞见 1：Fable 5 = 二阶传播链"）→ 被否："每条洞见想说什么？"
- v2：3 段式（现象 / 技术解释 / 产品解释 / 商业解释 / 未验证）→ 被否："AI 味太足"
- **v3**：反常断言 + 涌现式维度 + 倾向 + 替代假设 → **用户批准批量执行**

**v3 模板（洞见模板 v2，用户已确认）**：
```
现象：[1-2 句反常描述 + 判断钩子，不用数据罗列]
可能的解释（维度名从现象涌现，不预设 技术/产品/商业）：
- [维度名即问题]：[解释]
- [维度名即问题]：[解释]
倾向：[最相信哪个解释 + 一句理由]
未验证 / 替代假设：[如果洞见是错的，最可能的替代解释 + 怎么区分]
```

**v2 → v3 的 4 个改进点**：

| 改进 | v2（3 段式） | v3（v2 之后的改进） |
|---|---|---|
| **现象段** | 数据罗列（1.07 亿、11.9 倍） | 反常断言 + 判断钩子（"高播放 ≠ 热点，高播放 = 补课"） |
| **维度名** | 固定三维（技术/产品/商业） | 涌现式命名（"为什么是搬运+中文字幕而不是原创"） |
| **倾向** | 无（并列 = 放弃判断） | 主动标立场（"更相信结构性依赖，因为…"） |
| **未验证** | 缺数据（"需 UP 主历史数据"） | 替代假设（"如果错，替代解释是算法推课；区分方法：完播率"） |

**与教训 19 的关系**：教训 19 的"反 AI 味 4 规则"（不 3 段式 / 不并列 / 不套话 / 不平衡二段式）在 v3 里被部分修正——v3 **保留**了分段结构（现象/解释/倾向/未验证），但把"技术/产品/商业"固定标签改为涌现式，加了"倾向"行打破并列。教训 19 的核心精神（嵌入"人"动机、不要空壳标签）不变。

**完整 v3 示例**（洞见 2：B 站再教育基础设施）：
```
现象：B 站排名第一的不是 AI 新闻，是搬运 MIT 课程的账号。22 条里 11 条
是教学，但播放量最高的那条跟 AI 无关。高播放 ≠ 热点，高播放 = 补课。

可能的解释：
- 为什么是"搬运 + 中文字幕"而不是原创：中文技术圈对英文教学资源的依赖
  是结构性的——不是知识缺口，是语言缺口。
- 为什么集中在这个时间点：2026 H1 是 LLM 能力跨过"可用"阈值，补课需求
  集中爆发。
- B 站的生态选择：把"难学"的长视频教学当护城河。

倾向：更相信"结构性依赖"——如果只是补课需求，不会集中出现"搬运 + 重组"
这种特定形态。

未验证 / 替代假设：如果"再教育基础设施"是错的，替代解释是"算法对长视频
教学内容的系统性倾斜"。区分方法：看完播率（主动补课 → 高完播；算法推流
→ 低完播）。
```

**用户确认路径（2026-06-14 实测）**：
1. 出 4 个改进点 + 1 个样板（洞见 2）→ 用户判断方向
2. 用户："采用改进后模板。倾向加上。维度名改用涌现式命名。"
3. 批量改 5 条洞见 → 用户无异议

**操作技巧——批量重构多节洞见**：
```python
# 用 execute_code 批量 block_replace + block_delete
# 每条洞见：replace 标题block（含完整新内容）→ delete 旧子blocks
for xml_file, replace_id, delete_ids in insights:
    terminal(f"lark-cli docs +update ... --command block_replace --block-id {replace_id} --content @./{xml_file}")
    terminal(f"lark-cli docs +update ... --command block_delete --block-id {delete_ids}")
```

**教训 22（2026-06-14 新增）：调研建议从"话题清单"升级为"可执行调研简报"**

**旧模板问题**：
1. 是"话题标题"不是"可执行调研指令"——读完不知道第一步做什么

**v2 模板（用户已确认）**：
```
### 建议 N：[一句核心问题，不是话题标题]

类型：技术 / 市场 / 商业 / 政策 / 战略分析
为什么现在做：[时间窗口 + 不做会错过什么]
切入路径：
  1. [第一步：具体动作 + 数据源]
  2. [第二步]
  3. [第三步]
预期产出：[查完手里有什么]
来源锚点：洞见 N 的未验证假设 [一句话复述]
```

**调研建议来源不限于洞见**：可以来自事实层直接学习需求（如"1M token 上下文的技术实现路径"是技术学习，不经过洞见环节）。

**教训 20（2026-06-14 新增）：重构文档时不要添加"速览/摘要"层——主体已可扫描时，预告层是冗余**

**触发信号**：用户说"速览章节没有用。删除吧。" 或"目录结构可以"后看到 agent 自作主张加了"速览/摘要/一句话版"章节。

**真实失败路径（v12→v13 session）**：
- 用户要求"深度思考如何构建目录结构、信息架构，让文档清晰简单"
- agent 诊断出 4 个结构问题（主轴选错、事件跨章节重复、洞见不可寻址、元信息混排）
- 重构方案核心：每章拆为「事实/反方/洞见」H2 子标题 + 全局洞见编号 + 速览索引
- 用户批准了"目录结构"
- agent 用 `overwrite` 覆写文档——**加了一个"速览"H1 章节**（5 条洞见一句话版 + 4 条反方速览）
- 用户立刻否决："速览章节没有用。删除吧。"

**根因分析**：
- agent 把"提高可扫描性"等同于"加摘要层"——**错误**
- 主体结构本身已通过 H2 子标题（事实/反方/洞见）+ 全局洞见编号（洞见 1-5）实现了可扫描性
- 额外的"速览"层 = **把正文内容浓缩成一句话列表再放一遍** = 冗余噪音
- 本质是"禁止自我预告"在文档结构上的体现（与 MEMORY 中用户编辑标准一致：禁止自我预告、禁止解释自身结构安排）

**判别规则——什么时候"摘要层"是噪音，什么时候是导航**：

| 条件 | 判断 | 理由 |
|---|---|---|
| 主体已有 H2/H3 子标题 + 全局编号 | ❌ **不加摘要层** | 目录树本身就是导航，摘要层 = 内容重复 |
| 主体章节用粗体段落（无子标题） | ✅ 可以加 TL;DR | 目录树不可展开，读者无法跳读，需要入口 |
| 文档 > 5000 字 + 纯叙述无结构 | ✅ 可以加 TL;DR | 读者需要快速判断"值不值得读" |
| 文档已有"事实/反方/洞见"分层 | ❌ **不加摘要层** | 信息密度分层本身就是可扫描性，再加摘要 = 三重重复 |
| 用户明确说"TL;DR" / "便于阅读和思考" | ✅ 按结构 C（三层金字塔）做 | 用户主动要的，不是 agent 自加的 |

**与教训 12/14 的区别**：
- 教训 12：工程脚手架（revision_id / v1→v2 对照）不属于交付物
- 教训 14：识别并删除**已有**冗余（TL;DR 与末尾总结重叠）
- **教训 20：重构时不要**引入**新的**冗余层——预防 > 侦探

**删除冗余章节的操作技巧**：
```bash
# 1. 用 range + with-ids 一次性拿到要删的所有 block id（含子 block）
lark-cli docs +fetch --api-version v2 --doc <token> \
  --scope range --start-block-id <章节H1-id> \
  --end-block-id <下一章节H1-id> --detail with-ids --doc-format xml

# 2. block_delete 批量删除（逗号分隔所有 block id，含 H1/H2/列表项/hr）
lark-cli docs +update --api-version v2 --doc <token> \
  --command block_delete \
  --block-id "id1,id2,id3,...,idN"
```

**全文结构重组的 overwrite 工作流**（2026-06-14 实测，0 错误）：
1. `outline --max-depth 4` 探目录结构
2. `range`/`section` 逐段拉内容（分 2-3 次拉完）
3. 本地写完整 XML 文件（用 `write_file`）
4. `docs +update --command overwrite --content @./file.xml`（注意 `@./` 必须相对路径，见 `lark-doc` skill）
5. `outline` 验证新结构
6. 比逐 block 操作（insert/replace/delete 10+ 次）更可靠、更不容易遗漏
