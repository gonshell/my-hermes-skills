# Reflect-then-Confirm

> **Core principle:** 在任何大改动 / 设计 / 决策点之前，
> 先反思 + 给 2-3 个方案 + 标倾向 + 等用户确认。
> 永远不要"直接动笔"或"列 5+ 个 A/B/C/D/E 选项让用户挑"。

---

## Why this skill exists

我有一个**反复出现的失败模式**（在 2026-06-14 飞书报告 v1-v12 迭代中验证过）：

1. 反思时**堆叠方案**（A/B/C/D/E 5 个选项）→ 用户说"你把我弄懵了"
2. **直接动笔**写报告 / 改结构 / 改 skill → 用户说"反思后整理方案让我确认"
3. **自以为** 已想清楚 → 实际方案里漏了关键维度 → 用户说"那个方案不对"
4. 反思只讲**正面方案**，不讲**反方 / 盲区 / 不可证伪假设** → 用户说"深度不够"
5. 用户说"完事了么" / "是否需要确认" → 我**继续解释**而不是**检查 + 汇报**
6. **新增（2026-06-21）**：用户问"对话 vs 文档是否一致、是否需要融合"→ 我**直接重写**而不是**先核对一致性**（用 ✅/⚠️/❌/➕ 分类逐条核对）→ 丢失了"哪些一致、哪些缺失、哪些有冲突"的可追溯信息
7. **新增（2026-06-21）**：用户问"客观么、是否实事、评分标准是什么"→ 我用"主观评分（9/10、8/10）" + "无锚定" + "接受营销话术"凑出"伪科学分析"→ 用户两次追问后被迫自我承认。这是"没把'方法论适用性'做成'显性判断'"的根因。
8. **新增（2026-06-21）**：估值/分析报告用"目标价 40 元"等**单点估值** → 用户问"反思是否科学" → 自我承认这是用精确形式包装主观判断的伪科学。**估值/结论必给区间、不给点估值**。

这些都来自同一个根因：**没把"反思"做成"有结构的判断"**。

本 skill 沉淀 5 个**反复验证**的工作流元素：
- 反思粒度
- 方案堆叠上限
- 倾向标注
- 反方必填
- "完事了么"检查清单

---

## The 5-element template

每次大改动 / 设计 / 决策前，**必须**按这 5 个元素组织反思：

### 1. 反思粒度 = 2-3 个方案，不超过 3

- **不是** "A/B/C/D/E" 5 个方案
- **不是** "方案 A + 方案 A 的变体" 2 个微差异
- **是** 2-3 个**互相差异大**的方案（如"保留 v11 骨架"vs"减为 4 主题"），不是 1+2 个变体

```markdown
# ❌ 错（堆叠 5 个变体）
方向 1: 6 主题不变
方向 2: 6 主题不变 + 改章节顺序
方向 3: 6 主题不变 + 改标题格式
方向 4: 减为 4 主题
方向 5: 减为 4 主题 + 改标题格式

# ✅ 对（2 个差异大）
方向 1: 保留 v11 骨架 6 主题不变
方向 2: 减为 4 主题（合并 热点+趋势 / 商业+产品）
```

### 2. 每个方案 = 1 段假设 + 1 段依据 + 1 段盲区

不要 5 段铺开。**简短**是好的——用户是看思路对不对，不是看论文。

```markdown
方向 1：保留 v11 骨架 6 主题不变
  - 假设：6 维度结构稳定，用户已认可
  - 依据：v11 验证过可读性，每章 H3 子节点可展开
  - 盲区：6 主题对 200+ 条数据可能过细

方向 2：减为 4 主题
  - 假设：4 主题信息密度更高
  - 依据：72 条数据时 6 主题显得稀薄
  - 盲区：合并会丢失维度切片
```

### 3. 主动标"我倾向 X"——把判断做掉

```markdown
我倾向方向 1（保留 6 主题），因为：
  - 6 维度结构用户已确认 OK（"目录结构可以了"）
  - 合并 2 主题 = 丢维度，但当前数据没密到需要合并
  - 用户偏好反对过度设计
```

**让用户做"确认 vs 改方向"两步决策**，不是"挑 5 个选项"。

### 4. 反方 / 盲区 / 不可证伪假设 = 必填

反思末尾**必加**一段"反方 / 盲区"：

```markdown
反方 / 盲区：
- 我倾向方向 1 是因为数据密度不够——但如果下次数据变成 300+ 条，6 主题会变稀薄
- 合并方案虽然丢维度，但减少读者的"扫读疲劳"也是真实收益
- 我的判断基于"用户偏好反对过度设计"——但这偏好可能不适用所有报告类型
```

不要只给正面方案。**自我批判**是用户判断"思路对不对"的关键依据。

### 5. 不明列选项标签

不要写 A/B/C/D，直接"方向 1 / 方向 2 / 方向 3"——标签化 = 选择题化。

## When to use this skill

**触发条件（满足任一）：**
- 用户问"接下来怎么走" / "有几种方案" / "你倾向哪个"
- 用户说"反思一下" / "请分析下应该怎么改"
- 任何"接下来要做大改动"前的判断点
- 多版本迭代中（每次大改前先反思）
- **新增**：用户说"与对话核对一致性" / "内容是否一致" / "是否需要融合"——先做结构化核对（✅/⚠️/❌/➕ 分类），再给融合 vs 重写 vs 仅追加 vs 不动 的方向选择

**不触发：**
- 用户明确说"按你的方案做" / "直接做" / "动手"——直接做，不重复确认
- 单步骤机械任务（"把 X 复制到 Y"）
- 纯实现无设计取舍（"把这个函数写完"）
- 紧急 bug 修复（"X 报错了赶紧修"）

## The "完事了么" check protocol

用户说"完事了么" / "是否需要确认" / "继续做"——**立即执行 4 步检查**，不解释：

### Step 1：查 todo 状态（todo tool）
列出当前 todo，识别"已完成"和"未完成"

### Step 2：查磁盘 / 工具状态（terminal / search_files）
对每个"已完成"项，**真的查**文件 / doc / DB 存在性：
- 文件：`ls -la` 确认存在 + 大小合理
- 飞书 doc：`lark-cli docs +fetch` 确认 revision_id 递增
- DB / 状态：实查，不靠记忆

### Step 3：交叉对比预期 vs 实际
- 预期完成项 = todo 里"completed"状态
- 实际完成项 = Step 2 查到的
- **两者一致** → 完工
- **不一致** → 列出剩余项（不解释为什么之前没做完）

### Step 4：极简汇报
```
完工：
- [✅ 已完成项 1]
- [✅ 已完成项 2]
- [❌ 未完成项 1 + 简短原因]
- [❌ 未完成项 2 + 简短原因]
```

**不写** "我去检查一下"、"让我看看"这种过渡语。
**不写** 长篇总结"v1-v9 历程"。

## "对话 vs 文档" 一致性核对模板（新增 2026-06-21）

用户问"对话 / 之前分析与文档是否一致，是否需要融合"时，按以下 4 步做：

### Step 1：加载对话上下文
回顾本会话最近的 N 轮对话，识别对话中的关键论断（数据点、判断、结论）。

### Step 2：加载文档内容
读取目标文件 / 飞书 doc，对每个论断做四档分类：

| 标记 | 含义 | 处理 |
|------|------|------|
| ✅ 一致 | 文档有相同数据 / 论断 | 保留，不动 |
| ⚠️ 未覆盖 | 对话有新内容，文档没有 | 列入"必须新增"清单 |
| ❌ 冲突 | 文档与对话矛盾 | 列入"必须修订"清单，必须高亮 |
| ➕ 增量 | 文档已有，对话有补充 / 深化 | 列入"可补充"清单 |

### Step 3：列出必须新增 + 必须修订 + 可补充
**不要在核对阶段做任何修改动作**——只是分类 + 列表。

### Step 4：给 2-3 个融合方向 + 标倾向
- 方向 A：完整重写（保留原数据 + 新增 / 修订 / 补充）
- 方向 B：只新增章节（patch 插入新 H2，其他不动）
- 方向 C：重写关键章节 + 保留其余
- 方向 D：追加版（保留原文 + 下半部分新增）

**让用户选**——不擅自重写或追加。

## "不要看记忆" 强制实时查证

用户原话 "先不要看记忆" = 强制走实时查证 = **信任磁盘 / CLI > 信任记忆**。

**判别规则**：
- 记忆里的"4 个视频任务都是 docx 类型"是历史观察
- **不能**代替今天的 doc token 拉取
- **必须**实时跑 `cronjob list` / `cat ~/.hermes/cron/jobs.json` / `lark-cli drive files list`
- 拿到的实时数据 > 记忆里的"我之前以为"

`session-state-verification` skill 已覆盖此 pattern 的查询技术细节，本 skill 关注**反思粒度**和**用户沟通协议**。

## Output structure (执行时的输出格式)

```markdown
# 关于 [决策点] 的反思

## 我倾向的方向
[方向 N：[1 句话命名] + 1 句"为什么"]

## 方向 1：[命名]
- 假设：[1 句]
- 依据：[1-2 句]
- 盲区：[1 句]

## 方向 2：[命名]
- 假设：[1 句]
- 依据：[1-2 句]
- 盲区：[1 句]

## (可选) 方向 3：[命名]
- ...

## 反方 / 盲区（对自己倾向的）
- [1-2 句质疑 + 1 句"该判断需补 XX 才能下结论"]

## 需要你拍板
[1 句] 选方向 1 / 方向 2？或你说一句"按你的方案"我直接做。
```

**总长度 ≤ 200 字**（5 元素 + 1 决策点）。**不要 1000 字反思**——用户没时间看。

## Pitfalls

### P0：堆叠 5+ 个方案让用户懵
- **症状**：列出 A/B/C/D/E 5 个方案 + 各自利弊
- **解法**：只列 2-3 个**互相差异大**的方案
- **正例**：5 洞见 → A/B/C 三选项 + "我倾向 A"+ 一句理由
- **反例**：5 维变体 × 5 内容形式 × 3 元信息位置 = 75 个组合

### P0：反思只讲正面方案
- **症状**：反思 = 列 3 个方案 + 利弊 + "我倾向 X"
- **根因**：未要求"对自己倾向也要反方"
- **解法**：反思末尾**必加**"反方 / 盲区"段——对自己的倾向质疑
- **用户反馈原话**："方案选择看'思路对不对'不看'具体形式'——选完方案会给完全不同的实现。给方案时重点让他看思路对不对。"

### P0：删除/重写时保留原版作为对比基准（新增 2026-06-21）
- **症状**：发现分析有重大问题（伪科学、客观性不足）时，直接删除原版重写——用户失去"伪科学 vs 诚实版"的对比锚点
- **解法**：
  - **保留原版**作为"伪科学反面教材"
  - **重写新版本**（标 v2 / 诚实版 / 反思版）
  - 让用户能看到"为什么改、改了什么"
- **正例**：本会话宇通估值 → 发现"目标价 40 元"是伪科学后：
  - 保留原版（伪科学）
  - 新建"估值分析_诚实版.md"（区间 25-40 元 + 不确定性 ±30%）
  - 在整合报告里同时引用两个版本，让对比可见
- **反例**：直接 `rm` 原版 + `write_file` 新版 → 用户看不到反思过程

### P0：用户说"完事了么"= 立即执行检查，不解释
- **症状**：回答"让我先检查一下" / "我先看看磁盘"
- **解法**：直接跑 4 步检查 + 极简汇报
- **配套 skill**：`session-state-verification`

### P0：用户说"不要看记忆"= 强制实时查证
- **症状**：套用 MEMORY.md 里的"4 个 doc token"等历史结论
- **解法**：实时跑 cronjob list / jobs.json / drive list
- **配套 skill**：`session-state-verification`

### P0：用户问"对话 vs 文档一致性"→ 直接重写（新增 2026-06-21）
- **症状**：不做核对直接重写文档，丢失可追溯信息
- **解法**：先做 ✅/⚠️/❌/➕ 四档分类核对，列出"必须新增 / 必须修订 / 可补充"三档清单，再给 2-3 个融合方向让用户选
- **用户反馈原话**："请深入思考：xxx 和刚才的分析是否一致，内容是否需要融合"

### P0：评分/估值无锚定 = 伪科学（新增 2026-06-21，本会话核心学习）
- **症状**：用"9/10、8/10"等主观分数评估护城河强度，或用"目标价 40 元"等单点估值代替区间
- **根因**：评分标准未定义 / 估值方法适用性未评估 / 营销话术当事实用
- **解法**：
  - **删除主观分数**——改用"相对位置"判断（领先/落后/持平）+ 横向对比表（vs 同行/全球巨头）
  - **估值给区间不给点**——25-40 元（±30% 不确定性），不说"目标价 40 元"
  - **方法适用性必须先评估**——DDM 在 g<r 约束下不适用就不写，PB 在轻资产股不适用就不写
  - **拒绝"伪科学话术"**——"行业首个""国内独一档"等营销话术必须标"公司宣称，待第三方验证"
- **用户反馈原话**："是否客观、是否实事、评分标准是什么"——本会话两次追问
- **重写示例**：从"目标价 40 元"改为"估值参考区间 25-40 元，不确定性 ±30%"

### P1：接受营销话术作为客观事实（新增 2026-06-21）
- **症状**：报告中直接使用"行业首个""国内独一档""最先进"等公司宣传语，作为客观事实陈述
- **根因**：未标注"信息来源 + 可靠度"
- **解法**：
  - 营销话术必须标"**公司宣称，待第三方验证**"
  - 数据来源 + 可靠度（高/中/低）显式标注
  - 区别"事实层"（年报数据）和"判断层"（我的解读）
- **正例**：本会话宇通"睿控 E+ 行业首个" → 改为"**公司宣称行业首个，需第三方验证**"
- **反例**：直接写"宇通的睿控 E+ 平台是行业首个软硬件一体化电动专属平台，国内独一档" → 没有事实依据的客观化表述

### P1：方案堆叠 + 每个方案 5 段 = 反思 = 论文
- **症状**：反思 1000+ 字
- **解法**：反思 ≤ 200 字；方案 = 1+1+1 段；倾向 1 句；反方 1-2 句

### P1：用户没明确说"按你的方案" = 仍需等确认
- **症状**：反思完直接动笔，**不**等用户回复
- **解法**：反思末尾**明确**问"需要你拍板"——给用户"确认 vs 改方向"两步决策空间
- **例外**：用户说"按你的方案做" / "直接做"——直接做

### P1：方法适用性评估缺失（新增 2026-06-21）
- **症状**：套用标准估值方法（PE/DDM/DCF）不评估是否适用公司特性
- **根因**：教科书罗列方法，未做"方法-公司特性"匹配
- **解法**：套用前先问 3 个问题：
  - **DDM 适用吗？**——是否 g<r（永续增长 < 必要回报率）+ 分红率是否长期可持续
  - **PE 适用吗？**——是否业绩稳定 + 行业可比公司多
  - **DCF 适用吗？**——是否增长可预测 + 永续价值可估计
- **反例**：宇通 DDM 估值（g=2.5%, r=8%），但行业是结构性下滑（公交客流 -55%），g 应该 < 0，分母 (r-g) 接近 0.10 极度敏感——5 个百分点 g 变化 = 100% 估值变化
- **正例**：宇通 → 明确说"DDM 不适用"，只给 PE（有限适用）+ 情景分析（适用）作为主方法

### P1：每章"事实/反方/洞见"模板不分场景硬套
- **症状**：所有章节都套 3 段 H3（包括 §6 安全治理）
- **解法**：某些章节（如治理）可标"本章为事实层，暂无洞见"——不强造

### P1：方案 = "判断方向" vs "内容变体集合"
- **症状**：用户问"内容形式有几种"，我又列 5 种
- **解法**：方案 = "判断方向"，不是"内容变体集合"——内容形式属于"实现细节"，不是"反思维度"

### P1：用 5 段铺开描述一个方案（vs 1+1+1）
- **症状**：方案描述 200+ 字
- **解法**：方案 = 假设(1 句) + 依据(1-2 句) + 盲区(1 句) = 4 句话内

## Examples

### ✅ 正例 1：保留 v11 骨架还是减为 4 主题

```markdown
# 关于报告主题数量的反思

## 我倾向方向 1（保留 6 主题）
v11 骨架已被用户认可"目录可以了"——不应推翻。72 条数据看起来稀薄，
但报告靠"洞见 + 反方"密度补偿，不是靠"主题数"。

## 方向 1：保留 v11 骨架 6 主题
- 假设：6 维度结构稳定
- 依据：用户已确认"目录可以了，整体内容框架也可以了"
- 盲区：数据量小（72 条）时 6 主题显得稀薄

## 方向 2：减为 4 主题
- 假设：4 主题信息密度更高
- 依据：72 条 / 6 章 = 12 条/章，6 主题单章信息量低
- 盲区：合并 = 丢维度（趋势/热点 合并 = 难区分"事件"和"信号"）

## 反方
- 6 主题不一定真稳定——用户认可"目录"，但不一定认可"6 个"——可能希望 7 或 5
- 72 条数据下 6 主题虽然稀薄，但 4 主题也未必能填满

## 拍板
选方向 1 还是方向 2？
```

### ✅ 正例 3（新增 2026-06-21）：对话 vs 文档一致性核对

```text
# 一致性核对：[报告] vs [刚才对话]

## 四档分类
| 标记 | 项 | 处理 |
|------|-----|------|
| ✅ | 股价节点 / 财务数据 / 销量序列 / 海外数据 | 一致，保留 |
| ⚠️ | 上涨动力三层叠加、估值迁移路径 | 文档没有，必须新增 |
| ⚠️ | 下跌动力三层反向兑现 | 文档没有，必须新增 |
| ➕ | "高增长阶段结束"反常断言 | 文档 7.2 已弱化，可强化 |

## 修复方案
- 方向 A：完整重写（融合所有维度）
- 方向 B：只新增章节（在 7.1/7.2 插入新 H2 + 2.5 估值迁移）
- 方向 C：重写关键章节

## 反方
- 方向 A 改动大但一致性最强；方向 B 改动小但章节编号会乱
- 用户可能只是想要"补全核心缺口"，不是重写

## 拍板
选 A / B / C？默认我倾向 B（只新增 3 个 H2，最小改动）。
```

### ✅ 正例 4（新增 2026-06-21）：评分无锚定的反思

```text
# 关于"护城河强度"评分的反思（用户问"评分标准是什么"）

## 我倾向的方向
用"相对位置"代替"绝对分数"。理由：9/10 这种数字没有客观锚定，
宇通的护城河对比金龙/比亚迪/Daimler 才有意义。

## 方向 1：保留 9/10 主观分数
- 假设：评分能传达护城河强度
- 依据：传统卖方报告常用这种形式
- 盲区：评分无锚定 = 伪科学；和"金龙的 7/10"对比没有标尺

## 方向 2：删除评分，改用横向对比
- 假设：相对位置比绝对分数更有信息量
- 依据：宇通 17.88 亿研发 vs 金龙 8.23 亿 = 2.17 倍，比说"9/10"更具体
- 盲区：损失了"综合强度"的直观表达

## 反方
- 完全删除分数会让卖方分析师不习惯
- 横向对比需要多列数据，对话框里占空间

## 拍板
选方向 2（删除评分），同时加"反方/盲区"段
```
