# 写后反思自检清单(post-write reflection checklist)

## 适用场景

技术博客在阶段 7 优化修改 + 阶段 7.5 读者视角剪裁之后、阶段 8 飞书发布**之前**的最后一轮自检。这是一份**只给作者本人**的元检查——不修改文章本身,只诊断"这份文章能不能正式发出去"。

**不要跳过**。阶段 4-7 的所有审查(一致性 / 双轨 / 5 角色 / 优化)都是**加法思维**——找出问题、补上内容。但**没有一关负责从"这份文章对最终读者意味着什么"做整体诊断**。这一环节填补这个空白。

---

## 五维度自检模板

按以下 5 维度逐项审视。**每维度都给出明确的"判断标准"和"如果不通过应该怎么办"**。

### 维度 1:内容完整性

**判断问题:**
- 概念定义清晰吗?(一个不懂行的读者读完能讲出这是什么)
- 工程落地路径具体吗?(一个工程师读完能列出 6 步)
- 反方观点覆盖了主要风险吗?(至少 3-5 个独立维度)
- 配套的工程产物(state schema / verifier prompt / 示例配置)齐备吗?

**常见缺口信号:**
- 概念有定义但没有"与相似概念的边界"(如 Loop Engineering 没讲清 vs "agent while loop")
- 落地步骤是"步骤 1:..."但每步没说"具体配什么"
- 反方只有"哲学层面"(认知债),没有"工程层面"(失败案例 / 预警信号 / 回滚流程)

**如果不通过:** 补一节最小可执行的工程内容(如 state schema),或在反方章节加 1-2 个失败模式。

### 维度 2:叙述口吻一致性

**判断问题:**
- 全文是否使用统一的人称?(你 / 我 / 读者 / 隐含)
- 章节间语气没有突然跳变?(从"工程克制"突然到"激情宣告"是 P1 警告)
- 称谓/指代是否一致?(同一概念在不同章节叫不同名字)

**常见割裂信号:**
- 突然出现"我认为这一节比前面更重要"——从工程解读跳到个人偏好宣告
- 第八节反方观点开始"作为读者我也..."
- 标题党式断言(用感叹号、夸张副词)

**如果不通过:** 删除主观判断,改用结构说话("反方观点的工程层面补充如下")。

### 维度 3:内容真实性与权威性

**判断问题(关键!)——逐项扫描:**

| 检查项 | 不通过示例 | 严重性 |
|---|---|---|
| 引用是否有可点击的源头? | "某专家在某场合说..." 没链接 | 🔴 P0 |
| 数字是否可核实? | "800 万人围观"、"月活 10 亿" | 🔴 P0 |
| 断言是否可证伪? | "中文市场空白" | 🔴 P0 |
| 价格 / 单价是否标注时点? | "Sonnet 4 单价 $3 / 1M tokens" | 🟡 P1 |
| 产品功能是否标注发布时间? | "Codex 支持 .codex/agents/ TOML" | 🟡 P1 |

**如果不通过:** 删除或软化(改"被广泛转发"代替"800 万人围观";改"截至本文写作,搜索结果显示..."代替"中文市场空白")。

### 维度 4:逻辑一致性

**判断问题:**
- 论证链是否闭环?(提出问题 → 给出方案 → 验证 → 收尾)
- 章节间是否有过渡?(避免读者突然失去方向)
- 关键洞见是否前置?(不要让读者自己推断)
- 因果关系是否清晰?(不要把相关当因果)

**常见跳跃信号:**
- 第二节讲完范式,第三节突然讲定义——缺"为什么需要这个范式"的桥梁
- 表格中出现了未解释的术语(如 `/loop`、`/goal` 第一次出现)
- 反方观点放在文章最末尾,读者耐心已耗尽

**如果不通过:** 把关键洞见上移到章节开头,或加一句过渡句。

### 维度 5:可作为落地方案吗

**这个维度要诚实回答——大多数范式解读类博客的答案是"不能"。**

**判断问题:**
- 读者读完能直接 clone 一个仓库跑起来吗?
- 有真实生产案例支撑吗?(还是只是概念演示)
- verifier / 评测 / 成本控制等关键设计点都讲了吗?
- 团队实施需要投入多少人 / 月?(没有量化就不可作为方案)

**如果答案是"不能":** 在文末加"时效声明"和"参考边界"——明确告诉读者**这是一份"概念解读",不是"实施方案"**。这是诚实而非失败。

## 输出格式:三档优先级清单

反思输出应该按以下三档分类:

### 🔴 必修(P0,不修会损害可信度)

通常是:
- 数字 / 断言不可核实
- 引用源缺失
- 主观语气跳变
- 时效声明缺失

**用户通常希望全部完成。**

### 🟡 强烈建议(P1,显著提升价值)

通常是:
- 关键洞见位置优化
- 工程失败案例补充
- 章节间逻辑补桥
- 认知债 / 认知投降类专有名词开门定义

**用户通常选做。**

### 🟢 锦上添花(P2)

通常是:
- 加更多示意图
- 加非主流场景案例
- 行业历史背景
- 海外市场对照

**用户通常不做。**

**实操提示(本会话实证):** 用户被问到"选 P0 / P0+P1 / 全部"时,几乎都给出明确偏好。**不要让用户自己组合**——提供 3 个预定义套餐:

| 套餐 | 时间 | 适合场景 |
|---|---|---|
| 只 P0 | 5-10 分钟 | 快速发版,文章够用即可 |
| P0 + P1 | 15-25 分钟 | 想发到严肃平台(团队 wiki / 公开博客) |
| 全做(P0+P1+P2) | 30+ 分钟 | 想做"中文社区第一篇标杆解读" |

## 与现有阶段的关系

| 阶段 | 关心什么 | 不关心什么 |
|---|---|---|
| 4-7 流水线 | 内容质量、逻辑、读者反馈 | "这份文章对最终读者意味着什么" |
| 7.5 读者视角剪裁 | 删除过程性内容 | 数字 / 断言真实性 |
| **本反思环节** | 数字 / 断言真实性 + 整体判断 | 段落级质量(已被 7.5 覆盖) |

**不重复**——本环节只补 7.5 没覆盖的"作者整体姿态"。

## 2026-06-13 Loop Engineering 博客实证

本次反思发现的 4 项 🔴 必修,经验证全部是**5 角色反馈没抓到**的:
1. "800 万人围观"——未核实数字,5 角色无一提及
2. "中文市场空白"——不可证伪断言,5 角色无一提及
3. token 单价没标时点——5 角色提了"量化数据缺失"但没追到"时点声明"
4. 全文缺时效声明——5 角色没人要求"明确本文保质期"

**教训:5 角色反馈擅长抓"段落级问题",但**对"文章整体姿态"(未核实数字 / 不可证伪断言 / 时效声明)覆盖不够**。本环节必须由作者本人执行,不能外包。**

---

## 反思后:优化修改的执行规则

按用户"逐个设计、逐个确认、逐个修改"偏好:

1. **不要一次性贴出"完成清单"**——用户不要"汇报完成项",要"你做了什么"
2. **每个 patch 只改一个具体问题**——不要为了"效率"合并多个修改
3. **修改顺序:必修 → 强烈建议**——严格按优先级
4. **改完直接给结果,不解释**——简短回报"改了几处、涉及哪些章节",不重复清单
5. **不要重新发布**——除非用户明确说"重新发布",否则改完 .md 即可

**反例:** 修改完成后写一段总结性叙述"我完成了以下 8 项修改..."——这种叙述是给作者看的自评,不是给用户的产出。
