# Reader Persona Feedback — 5角色并行独立反馈框架

## 概述

本 Skill 实现一个 **5角色并行独立反馈** 框架，用于对技术博客文章进行多维度质量验证。核心设计原则：

> **各角色独立运行，互不讨论，防止观点平均化。**

每个角色从自身视角出发，独立阅读文章并给出反馈。最终汇总时才进行交叉对比，确保每个角色的真实意见不被"和稀泥"。

---

## 5个读者角色定义

### 1. 资深工程师（Senior Engineer）
- **画像**：10年以上工程经验，关注架构设计、性能、可维护性
- **关注点**：技术准确性、深度是否足够、代码示例是否实用、方案是否经得起生产环境考验
- **阅读习惯**：快速扫读，重点看架构图和代码段，跳过基础概念解释
- **典型口头禅**：「这个方案在生产环境下扛得住吗？」

### 2. 开源社区成员（Open Source Community Member）
- **画像**：活跃于 GitHub/技术论坛，热衷分享和协作，关注生态兼容性
- **关注点**：开源协议合规性、社区友好度、贡献门槛、与其他开源工具的集成可能
- **阅读习惯**：先看 README 式的摘要，再深入细节；关注有没有 reproduction 步骤
- **典型口头禅**：「能直接 fork 来玩吗？文档够不够友好？」

### 3. 普通读者（General Reader）
- **画像**：对技术有兴趣但非专业人士，通过搜索引擎或社交媒体进入文章
- **关注点**：是否通俗易懂、有没有不需要的前置知识、排版是否舒适、是否读完有收获
- **阅读习惯**：从头到尾顺序阅读，遇到不懂的地方容易放弃
- **典型口头禅**：「这段我看不太懂，能不能举个例子？」

### 4. 技术写作者（Technical Writer）
- **画像**：专注于技术文档和内容创作的专业人士，关注写作质量
- **关注点**：文章结构、段落衔接、用词一致性、图文配合、排版规范、标题层级
- **阅读习惯**：先扫结构（标题层级），再看内容流畅度，最后检查细节
- **典型口头禅**：「这段逻辑跳跃了，读者跟不上。」

### 5. 商业决策者（Business Decision Maker）
- **画像**：技术管理者或产品负责人，关注投入产出比和战略价值
- **关注点**：技术选型的商业价值、落地成本、团队学习曲线、竞品对比、ROI
- **阅读习惯**：只看摘要和结论，关注图表和关键数字
- **典型口头禅**：「这套方案落地要花多少人和时间？」

## 执行流程

### 阶段一：输入准备

```
输入：
  - 文章全文（Markdown 格式）
  - 文章元信息（标题、目标受众、发布平台）
  - （可选）特定关注点或已知问题
```

### 阶段二：并行独立反馈（核心）

**严格规则**：每个角色独立运行，生成反馈时**不得参考其他角色的意见**。

对每个角色执行以下步骤：

1. **角色代入**：以该角色的视角和经验水平阅读全文
2. **独立评分**：按照该角色关注的维度进行 1-5 分评分
3. **具体反馈**：指出具体段落/句子的问题，给出改进建议
4. **致命问题**：标注该角色视角下的 "阻断性问题"（即如果未修复，该角色群体会直接关闭文章）

### 阶段三：汇总与冲突分析

仅在所有 5 个角色反馈完成后才进行汇总：

1. **评分矩阵**：汇总各角色在各维度的评分
2. **共识区**：找出所有角色一致认可的优点和问题
3. **冲突区**：标注不同角色间的分歧（如"资深工程师觉得太浅显，普通读者觉得太难"）
4. **优先级排序**：基于影响范围和严重程度，给出改进建议的优先级列表

### 关键设计说明

1. **并行无依赖**：5个 task 同时启动，互不等待，互不参考对方结果。这保证了每个角色的独立性和真实性。
2. **context 包含三要素**：每个 task 的 context 都包含完整的（角色定义 + 反馈模板 + 文章全文），确保子任务无需额外查询即可独立完成。
3. **汇总在并行之后**：所有5个子任务返回结果后，由主 Agent 执行「阶段三：汇总与冲突分析」，生成评分矩阵、共识区、冲突区和优先级排序。
4. **变量替换**：`{文章全文内容}`、`{title}`、`{audience}`、`{platform}` 为调用时需替换的变量，从阶段一的输入准备中获取。

## 反馈输出格式

每个角色的反馈使用 `references/feedback-template.md` 模板，包含：

- **角色名称与画像摘要**
- **总体印象**（2-3 句话）
- **评分表**（各维度 1-5 分 + 简短理由）
- **具体问题清单**（定位到段落/行号）
- **改进建议**（按优先级排序）
- **阻断性问题**（如有）

## 注意事项

1. **防止角色漂移**：每个角色必须严格保持自己的视角，不要"替其他角色考虑"
2. **反馈要具体**：避免「文章不错」这类泛泛评价，必须定位到具体段落
3. **区分偏好与问题**：区分"我不喜欢这种风格"（偏好）和"这段有事实错误"（问题）
4. **文化敏感性**：如果文章面向中文读者，注意中文表达的地道性
5. **阻断性问题优先**：汇总时，先处理所有角色的阻断性问题

## 角色维度对照表

| 维度 | 资深工程师 | 开源社区 | 普通读者 | 技术写作者 | 商业决策者 |
|------|:---:|:---:|:---:|:---:|:---:|
| 技术准确性 | ★★★ | ★★ | - | ★ | ★ |
| 内容深度 | ★★★ | ★★ | ★ | - | ★ |
| 可读性 | ★ | ★★ | ★★★ | ★★★ | ★ |
| 结构逻辑 | ★ | ★ | ★★ | ★★★ | ★★ |
| 实用性 | ★★★ | ★★★ | ★★ | - | ★★★ |
| 商业价值 | - | - | - | - | ★★★ |
| 社区友好度 | ★ | ★★★ | - | ★ | - |
| 排版规范 | - | ★ | ★★ | ★★★ | ★ |

> ★ 表示关注，★★ 表示重要，★★★ 表示核心维度，- 表示不关注
