# 审查清单生成器（Review Checklist Generator）

采用"单Agent深度生成 + 多Agent交叉验证"两阶段模式，为各类交付物生成高质量、结构化的审查清单。支持技术审查清单和编辑审查清单两大类，确保审查覆盖全面、无遗漏。

## 核心原则

1. **深度优先**：阶段1由单一Agent按模板深度生成，保证连贯性和完整性
2. **交叉验证**：阶段2由3个不同角色Agent独立验证，发现盲区和遗漏
3. **人机协同**：阶段3聚合结果并提交人工确认，确保最终清单可执行

## When to Use

- 需要为技术文档、设计方案、代码PR、架构评审生成审查清单
- 团队需要标准化审查流程，避免审查遗漏
- 项目交付前需要全面的质量把关
- 需要生成可复用的审查模板

**Don't use for:** 简单的单文件检查、已有成熟checklist的场景、一次性快速review

## 工作流程

### 阶段1：单Agent深度生成

由主Agent根据待审查物和标准模板，深度生成审查清单初稿。生成过程按**双轨分离**执行：

- **技术审查轨道（熔炉型）**：目标是收敛到唯一结论（已验证✅ / 待验证⚠️ / 存疑❓）。每个检查项必须有明确的事实依据和收敛判定标准，不允许模糊地带。
- **编辑审查轨道（放大器型）**：目标是穷举候选、保留分歧。允许多种审查视角和风格偏好并存，最终由人工选择。

#### 输入要求

向用户提供以下信息（缺失则主动询问）：

| 参数 | 说明 | 示例 |
|------|------|------|
| 审查对象 | 待审查的交付物类型 | 技术方案文档、代码PR、设计稿、API文档 |
| 审查领域 | 所属技术领域 | 后端服务、前端应用、数据Pipeline、基础设施 |
| 审查重点 | 特别关注的方面 | 安全性、性能、可维护性 |
| 受众 | 清单使用者 | 技术负责人、QA、产品经理 |

#### 生成步骤

1. **分析审查对象**：理解交付物的类型、规模、复杂度和关键风险点
2. **加载对应模板**：从 `references/checklist-templates.md` 中选择匹配的模板
3. **定制化填充**：根据审查对象的实际情况，调整检查项的优先级和适用性
4. **生成两类清单（双轨分离）**：
   - **技术审查清单（熔炉型轨道）**：代码质量、架构合理性、安全合规、性能指标、可维护性。每个检查项附带收敛判定标准（已验证✅/待验证⚠️/存疑❓），目标是将所有项收敛到确定状态。
   - **编辑审查清单（放大器型轨道）**：文档结构、表述准确性、术语一致性、排版格式、可读性。允许多种审查视角并存，保留不同意见作为候选，不强制收敛。
5. **优先级标注**：每个检查项标记 P0（必须通过）/ P1（应当通过）/ P2（建议改进）
6. **输出初稿**：结构化Markdown格式的审查清单

#### 初稿输出格式

```markdown
# [审查对象名称] — 审查清单

- 审查对象：[名称/描述]
- 审查类型：[技术审查 / 编辑审查]
- 生成时间：[日期]
- 审查重点：[关键词]

## 技术审查清单

### 1. 架构与设计 (P0)
- [ ] [检查项描述] — 验证方法：[具体验证方式]
- [ ] ...

### 2. 代码质量 (P0/P1)
- [ ] [检查项描述] — 验证方法：[具体验证方式]

### 3. 安全与合规 (P0)
- [ ] [检查项描述] — 验证方法：[具体验证方式]

### 4. 性能与可靠性 (P1)
- [ ] [检查项描述] — 验证方法：[具体验证方式]

### 5. 可维护性与可扩展性 (P1/P2)
- [ ] [检查项描述] — 验证方法：[具体验证方式]

- [ ] [检查项描述]
```

### 阶段2：双轨交叉验证

阶段1初稿按双轨分别进行交叉验证，两条轨道使用不同的角色组合和验证策略。同一轨道内的角色**并行执行、互不共享中间结果**。

#### 技术审查轨道（熔炉型验证）

**策略**：3个角色对技术审查清单进行验证，目标是收敛到一致结论。每个角色独立验证后，对比结论：一致则标记为"已验证✅"，不一致则标记为"待验证⚠️"并触发复验。

**角色1：原始材料回溯Agent（Source Tracer）**

**关注点**：检查项的原始依据是否充分

验证任务：
- 追溯每个检查项的原始来源（规范文档、代码仓库、设计决策记录）
- 验证检查项是否必要且有据可依
- 标记无明确来源或来源过时的检查项
- 对每个检查项给出收敛判定：已验证✅/待验证⚠️/存疑❓

```python
delegate_task(
    goal="""你是一位原始材料回溯专家。审查以下技术审查清单，追溯每个检查项的原始依据。

你的角色：原始材料回溯Agent（Source Tracer）
关注维度：
1. 来源追溯 — 每个检查项是否有明确的规范/文档/决策依据？
2. 来源时效 — 引用的规范是否为最新版本？
3. 必要性验证 — 检查项是否真正必要？是否存在冗余？
4. 收敛判定 — 对每个检查项给出：已验证✅/待验证⚠️/存疑❓

输出格式（严格JSON）：
{
```

- 优先级标注是否合理（P0/P1/P2）
- 是否遗漏了领域特有的合规要求

**角色3：代码验证Agent（Code Verifier）**

**关注点**：可自动化验证的检查项的实际执行

验证任务：
- 识别哪些检查项可通过代码/脚本自动验证
- 对可自动化的检查项编写并执行验证脚本
- 输出通过/失败结果及具体证据
- 对无法自动化的检查项标记为"需人工验证"

```python
delegate_task(
```

#### 编辑审查轨道（放大器型验证）

**策略**：3个角色对编辑审查清单进行验证，目标是穷举审查视角、保留分歧。每个角色独立提出审查意见，所有意见均保留为候选，不强制收敛。最终由人工从中选择。

**角色1：首席编辑（Chief Editor）**

**关注点**：文档整体结构和逻辑连贯性

验证任务：
- 评估文档结构是否完整、层次是否清晰
- 检查各章节之间的逻辑过渡是否自然
- 评估信息的组织方式是否便于读者理解
- 提出结构优化建议（保留多个候选方案）

```python
delegate_task(

3. 信息架构 — 信息的组织方式是否便于读者理解？
```

验证任务：
- 非专业审查者能否理解和执行每个检查项
- 验证方法是否需要特殊工具或权限

- 预估审查耗时是否可接受

### 阶段3：聚合 + 人工确认

#### 聚合规则

| 验证角色 | 发现类型 | 处理方式 |
|----------|----------|----------|
| 领域专家 | 遗漏项 | 直接纳入初稿 |
| 领域专家 | 优先级调整 | 标记争议，等人工确认 |
| 使用者代理 | 不可执行项 | 修改验证方法 |
| 使用者代理 | 顺序建议 | 按工作流重新排列 |
| 魔鬼代言人 | 边缘情况 | 纳入，标记为"进阶检查" |
| 魔鬼代言人 | 安全缺口 | 强制纳入，优先级提升至P0 |

#### 聚合步骤

1. **去重**：3个角色可能提出相似的建议，合并为一条
2. **分类整合**：将所有补充项归入对应类别
3. **冲突解决**：当角色意见冲突时，保留两个版本并标记 `[争议]`
4. **生成终稿**：整合后的完整审查清单
5. **变更摘要**：列出相对于初稿的所有变更

## 常见陷阱

1. **生成过多检查项**：初稿应控制在 20-40 项，超过50项的清单难以执行。区分"必须检查"和"建议检查"。

2. **验证方法过于模糊**：避免"检查是否合理"这类描述，应写明"检查XX字段是否符合YY规范第ZZ条"。

3. **忽略审查对象规模**：小型改动用精简清单（10-15项），大型交付用完整清单。强行套用全套模板会浪费审查时间。

4. **3个角色串行执行**：应并行调用3个角色Agent，节省时间。它们之间不应共享中间结果。

5. **直接采用终稿不确认**：聚合后必须经人工确认，否则可能引入不符合团队实际的检查项。

6. **模板不裁剪直接使用**：模板是起点，必须根据具体审查对象裁剪。不是每个检查项都适用每个场景。

## 验证清单

- [ ] 初稿包含技术审查清单和编辑审查清单两部分
- [ ] 每个检查项都有明确的优先级标注（P0/P1/P2）
- [ ] 每个检查项都有具体的验证方法
- [ ] 3个角色Agent已并行执行验证
- [ ] 聚合结果已去重和分类
- [ ] 争议项已标记并等待人工确认
- [ ] 最终清单项数在合理范围（15-50项）
- [ ] 变更摘要已生成并呈现给用户
