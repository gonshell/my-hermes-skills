# 审查清单模板库

> 本文件包含4种常用审查清单模板。Agent根据审查对象类型选择匹配模板。

---

## 1. 技术文档审查模板

**适用场景**：技术博客、API文档、架构设计文档、技术白皮书

| ID | 检查项 | 验证方法 | 严重性 |
|----|--------|---------|--------|
| T-001 | 技术事实准确性 | 逐一核实关键技术声明，对比权威源（官方文档、RFC、论文） | P0 |
| T-002 | 技术深度充分性 | 检查核心概念是否有定义、解释或外部链接；标记"点到为止"未展开的术语 | P1 |
| T-003 | 边界条件覆盖 | 搜索文章是否提及局限性、前提条件、不适用场景 | P1 |
| T-004 | 数据/引用验证 | 核实每个数据点（性能数字、市场份额、版本号）的来源是否可追溯 | P0 |
| T-005 | 代码示例正确性 | 逐行检查代码片段的语法和逻辑；标注不可运行的示例 | P1 |
| T-006 | 技术时效性 | 检查引用的技术/工具/框架版本是否为当前有效版本 | P2 |

---

## 2. 代码PR审查模板

**适用场景**：代码变更审查、合并请求、补丁评审

| ID | 检查项 | 验证方法 | 严重性 |
|----|--------|---------|--------|
| C-001 | 逻辑正确性 | 追踪代码路径，检查边界条件和异常处理 | P0 |
| C-002 | 安全漏洞 | 检查输入验证、权限控制、敏感数据处理 | P0 |
| C-003 | 性能影响 | 评估时间/空间复杂度变化，检查N+1查询、内存泄漏 | P1 |
| C-004 | 可维护性 | 检查命名规范、代码结构、注释充分性 | P2 |
| C-005 | 测试覆盖 | 确认变更是否有对应测试；标记缺失的测试场景 | P1 |

---

## 3. 设计方案审查模板

**适用场景**：技术方案、架构设计、RFC文档、系统设计评审

| ID | 检查项 | 验证方法 | 严重性 |
|----|--------|---------|--------|
| D-001 | 需求覆盖 | 逐条对照需求文档，标记未覆盖的需求点 | P0 |
| D-002 | 架构合理性 | 检查模块划分、职责边界、依赖方向是否合理 | P1 |
| D-003 | 扩展性 | 评估方案对未来需求变化的适应能力 | P2 |
| D-004 | 风险评估 | 检查是否识别了技术风险、业务风险并有缓解措施 | P1 |
| D-005 | 替代方案对比 | 确认是否考虑了至少2个备选方案并说明选择理由 | P2 |

---

## 4. 编辑审查模板

**适用场景**：文章发布前的编辑审查、内容质量把关

| ID | 检查项 | 验证方法 | 严重性 |
|----|--------|---------|--------|
| E-001 | 结构清晰度 | 检查标题层级是否合理（H1→H2→H3递进）、章节是否有明确主题 | P1 |
| E-002 | 逻辑连贯性 | 检查段落间过渡是否自然、论证链是否完整 | P1 |
| E-003 | 节奏控制 | 检查技术密度是否均匀、是否有"信息过载"或"空洞"段落 | P2 |
| E-004 | 措辞精准度 | 标记模糊表述（"大概""可能""某种程度"）、冗余修辞 | P2 |
| E-005 | 目标读者匹配度 | 根据目标受众画像，评估文章难度、语气、篇幅是否合适 | P1 |

---

## 严重性等级定义

| 等级 | 含义 | 处理要求 |
|------|------|---------|
| P0 | 阻断性/事实错误 | 必须修复，阻塞发布 |
| P1 | 重要问题 | 强烈建议修复 |
| P2 | 改进建议 | 可选修复 |
| P3 | 锦上添花 | 记录即可 |
