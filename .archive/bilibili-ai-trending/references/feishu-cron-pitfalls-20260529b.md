# 飞书文档写入缺陷日志（2026-05-29 补充）

来源：调试 YouTube AI热门 cron job（早/晚）会话。

---

## P3: `--new-title` 参数完全无效（v1/v2 均不生效）

**问题现象**：cron prompt 的步骤5中传入 `--new-title "每日AI热门视频推送"`，期望修改文档标题，但文档标题始终不变。

**测试结论**（2026-05-29）：
- v2 API：`+update` 只有 `--command` 参数，没有 `--new-title` 参数
- v1 API：有 `--new-title` 参数，但传入后 `+fetch` 的 `title` 始终为 `null`，XML 内容中 `<title>` 标签被作为普通文本写入
- 文档 UI 标题由飞书系统管理，`lark-cli` 的 `--new-title` 参数**既不报错也不生效**

**实际影响**：prompt 中的 `--new-title` 只是冗余参数，不报错，不影响执行，但也不能改变标题。文档标题只能通过飞书界面手动修改。

**对 cron job 的影响**：无需移除 `--new-title`（不报错，不影响执行），但也不要期望它能改标题。如果需要改标题，只能手动在飞书界面操作。

## P4：追加写入 + 7天过滤的实现逻辑缺陷

**设计意图**：
```
步骤2：读取文档现有内容（fetch）
步骤3：解析内容，只保留7天内的视频记录
步骤4：追加今日内容到过滤后内容头部
步骤5：整体 overwrite 写入
```

**实现缺陷**：

1. **XML 日期解析依赖 LLM 生成质量，无结构化校验**
   - 过滤逻辑在 prompt 里用自然语言描述，完全依赖 LLM 正确生成 XML 中的日期标记
   - 如果日期格式不统一（如 `05-29` vs `2026-05-29`）或漏写日期，过滤会失效
   - 没有代码级校验机制，全部靠 prompt 引导

2. **时间戳注释只能防 degrade_code=1011，不能保证过滤正确**
   - 时间戳注释 `<!-- 每日AI热门视频推送 | 2026-05-29 20:00:00 -->` 确保每次内容不同
   - 但过滤7天逻辑是否正确执行，无法自动验证
   - 如果 LLM 在生成新 XML 时漏掉日期标记，已积累的视频记录会全部失去7天保护

3. **整体重写（overwrite）而非追加（append）**
   - 每次写入都是「过滤后内容 + 新内容」的全量重写
   - 如果步骤3的过滤逻辑失效，会不断累积过期内容（超过7天但未被过滤）
   - append 模式更适合追加，但需要 LLM 正确找到文档末尾位置

**改进方向（已知可行但本次未实现）**：
- 将视频记录以结构化格式存储（如每条记录一行，带固定日期字段）
- 用代码级解析器提取日期，而非依赖 LLM XML 解析
- 考虑 append 模式替代整体 overwrite
